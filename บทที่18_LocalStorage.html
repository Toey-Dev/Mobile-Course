1. 🖼️ ภาพเล่าเรื่อง: กระดานไวท์บอร์ด กับสมุดบัญชีในตู้เซฟ

ร้านอาหารของเรา (ที่มี "ห้องประชาสัมพันธ์" จากบทที่แล้ว) มีความลับที่น่าตกใจ: ทุกครั้งที่ปิดร้าน กระดานทุกแผ่นถูกลบเกลี้ยง

🌆 ระหว่างเปิดร้าน (แอปกำลังรัน) 🌃 ปิดร้าน (ผู้ใช้ปิดแอป / OS ฆ่าแอป) 📋 กระดานไวท์บอร์ด (state ใน RAM) ไฟดับทั้งอาคาร... · เขียนเร็ว อ่านเร็ว กระดานถูกลบเกลี้ยง 😱 · แต่เป็น "หมึกลบได้" _tasks ทั้งหมดหายไปตลอดกาล ทางแก้ของร้านมืออาชีพ: 📔 สมุดบัญชีในตู้เซฟ (ไฟล์ในเครื่อง) · เขียนช้ากว่ากระดาน แต่ "หมึกถาวร" — ไฟดับกี่รอบก็ไม่หาย (persistence) · กติกา 2 ข้อ: ① กระดานเปลี่ยนเมื่อไหร่ → จดลงสมุดทันที (save ทุกการแก้) ② เปิดร้านเช้ามา → เปิดสมุด คัดลอกขึ้นกระดาน (load ตอนเปิดแอป)

แต่มีปัญหาหนึ่ง — สมุดเล่มนี้เขียนได้แค่ "ข้อความ"

Task object ในหน่วยความจำ ข้อความในสมุด (JSON) ┌──────────────────┐ │ 📦 Task │ แบนให้เป็น [{"title":"อ่านหนังสือ","isDone":false}, │ title: "อ่าน..." │ ──────────► {"title":"ซื้อนม","isDone":true}] │ isDone: false │ toJson + │ toggle() │ jsonEncode ◄────────── └──────────────────┘ พองกลับเป็น object: jsonDecode + fromJson เหมือนถ่ายรูปของจริงเก็บในสมุด (แบน 2 มิติ) แล้ววันหลังปั้นกลับจากรูปได้เป๊ะ กระบวนการนี้ชื่อ serialization — และ "ภาษากลาง" ที่ทั้งโลกใช้เขียนรูปนี้คือ JSON
🤔 ทำไมบทนี้เจอ JSON ก่อนบทที่ 19

วันนี้คุณจะใช้ JSON กับข้อมูล "ของตัวเอง" ในสเกลเล็กก่อน (แบนเอง พองเอง เห็นครบวงจร) — พอถึงบทหน้าที่ JSON ถูกส่งมาจากเซิร์ฟเวอร์จริงทีละเป็นพันบรรทัด คุณจะอ่านมันออกเหมือนเจอเพื่อนเก่า

2. 💻 ลงมือจริง: ต่อท่อความจำถาวร (4 ขั้น)

ขั้นที่ 1 — ติดตั้งสมุดบัญชี

flutter pub add shared_preferences

shared_preferences คือที่เก็บแบบ key-value — จำตู้ล็อกเกอร์ของ Map (บทที่ 7) ได้ไหมครับ นี่คือตู้ล็อกเกอร์เวอร์ชันหมึกถาวร: 'tasks' → ข้อความ JSON ของเรา

ขั้นที่ 2 — สอน Task ให้ "ถ่ายรูป" และ "คืนชีพจากรูป" (lib/models/task.dart)

class Task {
  Task({required this.title, this.isDone = false});

  final String title;
  bool isDone;

  void toggle() => isDone = !isDone;

  /// ถ่ายรูป: Task → Map (ขั้นแรกของการแบนเป็นข้อความ)
  Map<String, dynamic> toJson() => {'title': title, 'isDone': isDone};

  /// คืนชีพจากรูป: Map → Task (factory = constructor พิเศษที่สร้างจากข้อมูลดิบ)
  factory Task.fromJson(Map<String, dynamic> json) {
    return Task(
      title: json['title'] as String,
      // ?? = เกราะจากบทที่ 4: ถ้าข้อมูลเก่าไม่มีฟิลด์นี้ ใช้ค่าตั้งต้นแทน ไม่พัง
      isDone: json['isDone'] as bool? ?? false,
    );
  }
}
🧠 หลักการเดิมจากบทที่ 8

ความรู้เรื่องงาน อยู่กับตัวงาน — วิธีแปลงร่างของ Task ก็อยู่ในคลาส Task ไม่กระจายไปที่อื่น

ขั้นที่ 3 — ห้องประชาสัมพันธ์รับหน้าที่จดสมุด (lib/providers/task_provider.dart)

import 'dart:convert';                                    // jsonEncode / jsonDecode
import 'package:flutter/foundation.dart';
import 'package:shared_preferences/shared_preferences.dart';
import '../models/task.dart';

class TaskProvider extends ChangeNotifier {
  static const _storageKey = 'tasks';   // ป้ายล็อกเกอร์เป็น const — กันพิมพ์ผิด
  final List<Task> _tasks = [];

  List<Task> get tasks => List.unmodifiable(_tasks);
  int get doneCount => _tasks.where((t) => t.isDone).length;

  /// 📔 จดกระดานลงสมุด — private เพราะเป็นเรื่องภายในห้อง
  Future<void> _saveTasks() async {
    final prefs = await SharedPreferences.getInstance();  // ขอกุญแจตู้เซฟ (ต้องรอ = async บท 9)
    final jsonString = jsonEncode(_tasks.map((t) => t.toJson()).toList());
    await prefs.setString(_storageKey, jsonString);       // เขียนหมึกถาวร
  }

  /// 🌅 เปิดร้าน: อ่านสมุดขึ้นกระดาน — เรียกครั้งเดียวตอนแอปเริ่ม
  Future<void> loadTasks() async {
    final prefs = await SharedPreferences.getInstance();
    final jsonString = prefs.getString(_storageKey);
    if (jsonString == null) return;       // เปิดร้านครั้งแรก สมุดยังว่าง (guard clause บท 5)

    final list = jsonDecode(jsonString) as List<dynamic>;
    _tasks
      ..clear()
      ..addAll(list.map((e) => Task.fromJson(e as Map<String, dynamic>)));
    // (.. คือ cascade: "ทำกับ object เดิมต่อเนื่อง" — ท่าย่อ อ่านเจอจะได้ไม่งง)
    notifyListeners();                    // 📢 ประกาศ: กระดานพร้อมแล้ว ทุกจอวาดเลย
  }

  void addTask(String title) {
    _tasks.add(Task(title: title));
    notifyListeners();
    _saveTasks();   // กติกาข้อ ①: กระดานเปลี่ยน → จดสมุดทันที
  }                 // (ตั้งใจไม่ await — จอไม่ต้องยืนรอคนจดสมุด)

  void toggleTask(Task task) {
    task.toggle();
    notifyListeners();
    _saveTasks();
  }

  void removeTask(Task task) {
    _tasks.remove(task);
    notifyListeners();
    _saveTasks();
  }
}

4. 🔧 เทคนิคการประยุกต์ใช้

เทคนิคที่ 1 — Save ทุกการแก้ ไม่มีปุ่ม "บันทึก"

แอปมือถือไม่เหมือนโปรแกรมคอมพิวเตอร์ยุคเก่า: OS ฆ่าแอปได้ทุกวินาที (สายเรียกเข้า, แรมเต็ม, ผู้ใช้ปัดทิ้ง) — จึงไม่มีจังหวะ "ค่อยกดเซฟ" การจดสมุดต้องเกิดอัตโนมัติทุกการแก้ และเพราะเราแก้ข้อมูลผ่าน method ของ provider เท่านั้น (บทที่ 17) จุดที่ต้องเติม _saveTasks() จึงมีแค่ไม่กี่บรรทัด — วินัยเก่ากำลังจ่ายดอกเบี้ย

เทคนิคที่ 2 — ?? ใน fromJson คือเกราะข้ามเวอร์ชัน

วันหนึ่งแอปเวอร์ชันใหม่เพิ่มฟิลด์ dueDate — แต่ผู้ใช้เก่ามีข้อมูลที่ "ไม่มีฟิลด์นี้" อยู่ในเครื่อง! json['dueDate'] ?? ค่าตั้งต้น ทำให้ข้อมูลเก่าเปิดได้ไม่พัง — ปัญหานี้ชื่อ data migration และเป็นเรื่องที่แอปจริงต้องคิดทุกครั้งที่แก้ model

เทคนิคที่ 3 — key เป็น static const เสมอ

'tasks' กับ 'task' คือคนละล็อกเกอร์ — พิมพ์ผิดหนึ่งตัวอักษร ข้อมูล "หายทั้งก้อน" แบบไม่มี error ใดๆ (บั๊กเงียบตระกูลเดียวกับ Map บทที่ 8) รวม key ไว้เป็นค่าคงที่ที่เดียว ปิดประตูบั๊กตระกูลนี้ถาวร

เทคนิคที่ 4 — อย่าเอาของผิดประเภทใส่สมุดเล่มนี้

รูปภาพ/ไฟล์ใหญ่ → เก็บเป็นไฟล์จริง · token/รหัสผ่าน → secure storage เท่านั้น (shared_preferences ไม่เข้ารหัส!) · ข้อมูลหลักหมื่นรายการ → ฐานข้อมูล — สมุดบัญชีมีไว้จดตัวเลข ไม่ใช่แปะรูปติดทั้งอัลบั้ม

5. 🏭 ตัวอย่างการเขียนจริงในงานจริง

🏭 สถานการณ์จริง: "แบบร่างที่รอดจากสายเรียกเข้า"

ผู้ใช้คนหนึ่งพิมพ์โพสต์ยาว 20 นาทีในแอปโซเชียล — จู่ๆ สายเรียกเข้า คุยไป 10 นาที ระหว่างนั้น OS ฆ่าแอปเงียบๆ เพราะแรมเต็ม... เปิดแอปกลับมา:

แอปที่ไม่ได้ทำการบ้าน: โพสต์หายเกลี้ยง → ผู้ใช้โกรธ → รีวิว 1 ดาว → ลบแอป แอปมืออาชีพ: "มีแบบร่างที่ยังไม่ได้โพสต์" — ข้อความอยู่ครบทุกตัวอักษร ❤️ เบื้องหลังของแอปมืออาชีพ คือ pattern เดียวกับที่คุณเพิ่งเขียนเป๊ะ: ทุกครั้งที่ผู้ใช้พิมพ์ (หยุดพิมพ์ ~1 วิ) → จดร่างลงเครื่อง (save) เปิดหน้าเขียนโพสต์ → เช็คว่ามีร่างค้างไหม (load) → ถามผู้ใช้ว่าเอาต่อไหม

ลองสังเกตแอปที่คุณใช้ทุกวัน: Line จำห้องแชทที่เลื่อนค้าง, YouTube จำวินาทีที่ดูค้าง, เกมจำด่านที่เล่นถึง — ทั้งหมดคือ save/load ที่ทำงานเงียบๆ ตลอดเวลา ความรู้สึก "แอปนี้เข้าใจฉัน" ของผู้ใช้ แท้จริงคือ persistence ที่ออกแบบมาดี

7. ⚠️ ข้อผิดพลาดที่เจอบ่อย

① "type 'List<dynamic>' is not a subtype of type 'List<Task>'"

สาเหตุ: jsonDecode ไม่รู้จักคลาสของเรา — มันคืน List<dynamic>/Map<String, dynamic> ดิบๆ เสมอ เอาไปใช้เป็น List<Task> ตรงๆ เลยชนกำแพงชนิดข้อมูล (บทที่ 4)
วิธีแก้: แปลงทีละชิ้นผ่าน fromJson เสมอ: list.map((e) => Task.fromJson(e as Map<String, dynamic>)).toList() — jsonDecode ให้ "วัตถุดิบ" เรามีหน้าที่ปั้นเป็น object เอง

② เปิดแอปมารายการว่างเปล่า ทั้งที่เคยบันทึกไว้

สาเหตุ: ไล่ 3 จุดตามลำดับ: ① ลืมเรียก loadTasks ตอนเปิดแอป (ขั้นที่ 4) ② loadTasks ทำงานแต่ลืม notifyListeners ท้ายฟังก์ชัน (โหลดขึ้นกระดานแล้วแต่ไม่ประกาศ — จอเลยไม่วาด) ③ key ไม่ตรงกันระหว่าง save กับ load
วิธีแก้: นักสืบบทที่ 14 ทำงาน: breakpoint ใน loadTasks ดูว่า jsonString เป็น null หรือมีของ → รู้ทันทีว่าปัญหาอยู่ฝั่ง save หรือฝั่ง load

③ "FormatException: Unexpected character" ตอน jsonDecode

สาเหตุ: ข้อความในล็อกเกอร์ไม่ใช่ JSON ที่ถูกต้อง — มักเกิดตอน dev: เคยเก็บรูปแบบเก่าไว้ แล้วเราแก้โครงสร้างข้อมูล ของเก่าค้างในเครื่องเลยอ่านไม่ออก
วิธีแก้: ระหว่าง dev: ลบของเก่าทิ้ง (prefs.remove('tasks') ครั้งเดียว หรือถอนแอปแล้วลงใหม่) · ในแอปจริง: ครอบ load ด้วย try-catch กันแอปเปิดไม่ขึ้นเพราะข้อมูลเสีย — บทที่ 20 จะสอนจัดการ error แบบเต็มระบบพอดี

8. 🏋️ แบบฝึกหัดท้ายบท (ลงมือทำจริง)

🥉 ระดับ 1 — ความจำถาวรครั้งแรก

ทำครบ 4 ขั้น แล้วพิสูจน์: เพิ่มงาน + ติ๊ก → ปิดแอปจริง (ปัดทิ้ง ไม่ใช่ hot restart) → เปิดใหม่ ทุกอย่างอยู่ครบ — โบนัสให้เข้าใจลึก: debugPrint(jsonString) ใน _saveTasks แล้วอ่าน JSON ของตัวเองให้ออกทุกตัวอักษร (นี่คือหน้าตาข้อมูลที่บทหน้าเซิร์ฟเวอร์จะส่งมา)

🥈 ระดับ 2 — แอปจำธีมได้

สร้าง SettingsProvider ใหม่ (แยกโดเมนตามเทคนิคบทที่ 17 — ใช้ MultiProvider ครอบสองตัว) เก็บ isDarkMode ลง shared_preferences + ปุ่มสลับธีมบน AppBar + MaterialApp เลือก ThemeData ตามค่า — ปิดเปิดแอปแล้วธีมต้องจำได้ commit ด้วย feat: persistent dark mode setting

🥇 ระดับ 3 — ซ้อมรับมือข้อมูลต่างเวอร์ชัน (migration drill)

เพิ่มฟิลด์ createdAt (วันที่สร้างงาน) ใน Task: (1) toJson เก็บเป็น createdAt.toIso8601String() (2) fromJson ต้องเปิดข้อมูลเก่าที่ไม่มีฟิลด์นี้ได้โดยไม่พัง (ใช้ ?? + DateTime.tryParse) (3) แสดงวันที่ในหน้ารายละเอียด — ห้ามลบข้อมูลเก่าตอนทดสอบ: ต้องเห็นงานเก่า (ไม่มีวันที่) อยู่ร่วมกับงานใหม่ (มีวันที่) ได้จริง — commit ด้วย feat: task creation date with backward compatibility

9. ✅ เช็กลิสต์และสรุปบทเรียน

  • อธิบายภาพ "กระดานไวท์บอร์ด vs สมุดในตู้เซฟ" และกติกา 2 ข้อ (save ทุกการแก้ / load ตอนเปิด) ได้
  • เขียน toJson / fromJson ให้ model ได้เอง และอ่าน JSON ที่ตัวเองเก็บออก
  • อธิบายได้ว่าทำไม ?? ใน fromJson คือเกราะข้ามเวอร์ชัน
  • ตอบได้ว่างานแบบไหนใช้ shared_preferences / ฐานข้อมูล / secure storage
  • หน้าจอทุกหน้าไม่มี import shared_preferences — ทุกอย่างจบใน provider
  • ปิดแอปจริงแล้วเปิดใหม่ ข้อมูลอยู่ครบ (ทดสอบแล้วกับตา)

📖 สรุปศัพท์ใหม่บทนี้

ศัพท์ความหมายสั้นๆภาพจำ
persistenceข้อมูลอยู่รอดข้ามการปิด-เปิดแอปหมึกถาวรในตู้เซฟ
shared_preferencesที่เก็บ key-value ขนาดเล็กในเครื่องสมุดบัญชีของแอป
key-value storageเก็บแบบ ป้าย→ค่าตู้ล็อกเกอร์หมึกถาวร (Map บทที่ 7)
toJson / fromJsonobject → Map / Map → objectถ่ายรูปเก็บ / คืนชีพจากรูป
jsonEncode / jsonDecodeMap/List ↔ ข้อความ JSONเครื่องแบน / เครื่องพองกลับ
getInstanceขอตัวเชื่อมต่อที่เก็บ (ต้อง await)ขอกุญแจตู้เซฟ

state ใน RAM คือหมึกลบได้ — ปิดแอปแล้วหาย · กติกา 2 ข้อ: แก้แล้วจดทันที, เปิดแล้วอ่านกลับ · serialization = ถ่ายรูปเก็บ (toJson→jsonEncode) แล้วคืนชีพ (jsonDecode→fromJson) · การเก็บข้อมูลจบใน provider ที่เดียว — หน้าจอไม่ต้องรู้เลยว่าสมุดอยู่ไหน

1. 🖼️ ภาพเล่าเรื่อง: เคาน์เตอร์ธนาคาร

การรับข้อมูลจากผู้ใช้ เหมือนเคาน์เตอร์ธนาคารรับแบบฟอร์มเปิดบัญชี — มีตัวละคร 3 ตัวทำงานร่วมกัน:

ลูกค้า (ผู้ใช้) พนักงานจดบันทึก พนักงานตรวจเอกสาร (TextEditingController) (validator) เขียนลงแบบฟอร์ม ──► จดทุกตัวอักษรที่ลูกค้า ──► ก่อนรับเรื่อง ตรวจทีละช่อง: ทีละตัวอักษร เขียนไว้ตลอดเวลา · ช่องชื่อว่าง? ❌ วงแดง ถามเมื่อไหร่ก็ตอบได้ว่า ส่งกลับไปแก้ พร้อมโน้ต "ตอนนี้เขียนว่าอะไร" "กรุณากรอกชื่อ" (.text) · ครบถูกต้อง? ✓ รับเรื่อง ┌─────────────────────────────────────────────────┐ │ 📋 แบบฟอร์มเพิ่มงาน (Form) │ │ ┌───────────────────────────────┐ │ │ │ ชื่องาน: [ อ่านหนังสือ......... ] │ ← TextFormField (ช่องกรอก 1 ช่อง) │ └───────────────────────────────┘ │ │ ⚠ กรุณากรอกชื่องาน ← ข้อความจาก validator │ │ [ บันทึก ] │ └─────────────────────────────────────────────────┘ Form = ทั้งแผ่นแบบฟอร์ม · TextFormField = ช่องกรอกแต่ละช่อง · validator = ผู้ตรวจประจำช่อง
🤔 ทำไมต้องตรวจก่อนรับ

ข้อมูลจากมนุษย์คือแหล่งความวุ่นวายอันดับหนึ่งของทุกระบบ — ช่องว่าง, เว้นวรรคล้วนๆ, ยาวเป็นกิโล, อีโมจิในช่องเบอร์โทร ล้วนเกิดจริงทุกวัน กฎเหล็กของ dev: อย่าเชื่อ input จากผู้ใช้ จนกว่าจะตรวจแล้ว (ธีมเดียวกับ normalize บทที่ 5 และ ?? บทที่ 4 — ระวังข้อมูลจากภายนอกเสมอ)

2. 💻 ลงมือจริง: หน้า "เพิ่มงานใหม่"

ขั้นที่ 1 — เริ่มจากตัวเล็กสุด: TextField + Controller

// TextField เปล่าๆ พิมพ์ได้ แต่เราต้องมี "พนักงานจด" ถึงจะรู้ว่าพิมพ์อะไร
final _titleController = TextEditingController();

TextField(
  controller: _titleController,            // ผูกพนักงานจดเข้ากับช่อง
  decoration: const InputDecoration(
    labelText: 'ชื่องาน',                   // ป้ายลอยเหนือช่องตอนพิมพ์
    hintText: 'เช่น อ่านหนังสือ 30 นาที',    // ตัวอย่างจางๆ ในช่อง
    border: OutlineInputBorder(),
  ),
)

// อยากรู้ว่าผู้ใช้พิมพ์อะไร ถามพนักงานจดได้ทุกเมื่อ:
print(_titleController.text);

ขั้นที่ 2 — หน้าเต็ม: Form + validator + ส่งค่ากลับด้วย pop

สร้าง lib/screens/add_task_screen.dart — สังเกตว่าบทนี้ใช้ความรู้ทุกบทของเฟส 3 พร้อมกัน:

import 'package:flutter/material.dart';

/// หน้าเพิ่มงาน — Stateful เพราะถือ controller และสถานะฟอร์ม
class AddTaskScreen extends StatefulWidget {
  const AddTaskScreen({super.key});

  @override
  State<AddTaskScreen> createState() => _AddTaskScreenState();
}

class _AddTaskScreenState extends State<AddTaskScreen> {
  // GlobalKey = "บัตรพนักงาน" ที่ทำให้เราสั่งงานแบบฟอร์มได้จากข้างนอก
  final _formKey = GlobalKey<FormState>();
  final _titleController = TextEditingController();

  @override
  void dispose() {
    // พนักงานจดต้อง "เลิกงาน" เมื่อหน้าถูกทำลาย — ไม่งั้นกินแรมค้างไว้ (memory leak)
    _titleController.dispose();
    super.dispose();
  }

  void _submit() {
    // สั่งผู้ตรวจทุกช่องทำงาน — ผ่านครบทุกช่องถึงได้ true
    if (_formKey.currentState!.validate()) {
      // trim() ก่อนใช้เสมอ — normalize จากบทที่ 5 อยู่ทุกที่จริงๆ
      Navigator.pop(context, _titleController.text.trim());  // pop พร้อมพัสดุ (บทที่ 15)
    }
  }

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      appBar: AppBar(title: const Text('เพิ่มงานใหม่')),
      body: Padding(
        padding: const EdgeInsets.all(16),
        child: Form(
          key: _formKey,                    // ผูกบัตรพนักงานเข้ากับแบบฟอร์ม
          child: Column(
            children: [
              TextFormField(
                controller: _titleController,
                autofocus: true,            // เปิดหน้ามา คีย์บอร์ดเด้งพร้อมพิมพ์เลย
                decoration: const InputDecoration(
                  labelText: 'ชื่องาน',
                  border: OutlineInputBorder(),
                ),
                // ผู้ตรวจประจำช่อง: คืน "ข้อความเตือน" = ไม่ผ่าน, คืน null = ผ่าน
                validator: (value) {
                  if (value == null || value.trim().isEmpty) {
                    return 'กรุณากรอกชื่องาน';     // โผล่ใต้ช่องเป็นสีแดงอัตโนมัติ
                  }
                  if (value.trim().length > 50) {
                    return 'ชื่องานยาวเกิน 50 ตัวอักษร';
                  }
                  return null;                    // ✓ ผ่าน (guard clause บทที่ 5!)
                },
                onFieldSubmitted: (_) => _submit(), // กด Enter บนคีย์บอร์ด = ส่งเลย
              ),
              const SizedBox(height: 16),
              SizedBox(
                width: double.infinity,           // ปุ่มกว้างเต็ม (บทที่ 11)
                child: FilledButton(
                  onPressed: _submit,
                  child: const Text('บันทึก'),
                ),
              ),
            ],
          ),
        ),
      ),
    );
  }
}

3. 🔧 เทคนิคการประยุกต์ใช้

เทคนิคที่ 1 — ทุก Controller ต้องมี dispose คู่กันเสมอ

กฎท่องจำ: "จ้างพนักงานจด (สร้าง controller ใน State) = ต้องให้เลิกงาน (dispose)" — ลืมแล้วแอปไม่พังทันที แต่แรมรั่วสะสมทีละนิด นี่คือบั๊กแบบที่ตรวจเจอยากที่สุด สร้างนิสัยตั้งแต่ตัวแรก: พิมพ์ controller เสร็จ เขียน dispose ทันทีก่อนลืม

เทคนิคที่ 2 — validator อ่านเป็น guard clause

โครงเดียวกับบทที่ 5 เป๊ะ: เช็คเงื่อนไขไม่ผ่าน → return ข้อความ (= ส่งกลับไปแก้), ผ่านทุกด่าน → return null — เขียน validator กี่ช่องก็ใช้โครงนี้ และแยกเป็นฟังก์ชันตั้งชื่อได้เมื่อยาว: validator: _validateTitle

เทคนิคที่ 3 — keyboardType + textInputAction = ความใส่ใจที่ผู้ใช้รู้สึกได้

ช่องเบอร์โทร → keyboardType: TextInputType.phone (คีย์บอร์ดตัวเลขเด้งแทน) · ช่องสุดท้ายของฟอร์ม → textInputAction: TextInputAction.done — รายละเอียดเล็กๆ ที่แยกแอปมือสมัครเล่นกับมืออาชีพออกจากกัน

เทคนิคที่ 4 — อย่าเก็บค่าที่พิมพ์ลง state ซ้ำ

controller คือแหล่งความจริงของข้อความในช่องอยู่แล้ว (single source of truth บทที่ 12!) — ไม่ต้อง onChanged: (v) => setState(...) ให้ rebuild ทุกตัวอักษร อ่านจาก .text ตอน submit ทีเดียวพอ (ยกเว้นต้องการ UI สดตามการพิมพ์ เช่น ตัวนับอักษร)

4. 🏭 ตัวอย่างการเขียนจริงในงานจริง

🏭 สถานการณ์จริง: "ฟอร์มสมัครสมาชิก — สนามรบ UX ที่ทีมจริงเถียงกันมากที่สุด"

ฟอร์มคือจุดที่ผู้ใช้ "ทิ้งแอป" มากที่สุด ทีมจริงจึงมีกติกาที่ตกผลึกจากข้อมูลผู้ใช้นับล้าน:

กติกาฟอร์มของทีมมืออาชีพ เหตุผล ───────────────────────────────────────────────────────────── ① error โผล่ "ใต้ช่อง" ไม่ใช่ popup popup บังฟอร์ม ต้องกดปิด น่ารำคาญ ② ข้อความ error บอก "วิธีแก้" ไม่ใช่ดุ "กรุณากรอกอีเมล" ✓ "ข้อมูลไม่ถูกต้อง!" ✗ (แก้ยังไงล่ะ?) ③ ตรวจตอนกดส่ง ไม่ใช่ด่าตั้งแต่ยังพิมพ์ไม่เสร็จ พิมพ์อีเมลได้ 3 ตัวอักษรแล้วขึ้นแดง = ประสบการณ์แย่มาก ④ ปุ่มส่งกว้างเต็มจอ อยู่ท้ายฟอร์ม นิ้วโป้งเอื้อมถึงง่ายสุด

สังเกตว่า TextFormField + validator ที่คุณเพิ่งเขียน ให้ข้อ ①③ มาแล้วอัตโนมัติ — ทีม Flutter ฝังบทเรียน UX พวกนี้ไว้ในเครื่องมือ ส่วนข้อ ② คือหน้าที่ของเราตอนเขียนข้อความ validator ทุกครั้ง

5. 🧹 ตัวอย่าง Clean Code: ฟอร์มที่โตได้

// ❌ ตรวจข้อมูลด้วย if หลังกดปุ่ม + แจ้ง error ด้วย SnackBar
onPressed: () {
  // ตรวจเองนอกระบบฟอร์ม — ต้องจัดการเองทุกอย่าง
  if (_titleController.text.isEmpty) {
    ScaffoldMessenger.of(context).showSnackBar(
      const SnackBar(content: Text('กรอกชื่องานด้วย!')),   // ผิดกติกาข้อ ①: error ลอยไปมา
    );
    return;
  }
  if (_titleController.text.length > 50) { /* SnackBar อีกอัน... */ }
  // เพิ่มช่องที่ 2, 3 เมื่อไหร่ = if กองพะเนิน + SnackBar รัวๆ
},
// ✅ ใช้ระบบ Form — ผู้ตรวจประจำช่อง ขยายกี่ช่องก็โครงเดิม
// แต่ละช่องถือ validator ของตัวเอง — เพิ่มช่องใหม่ = เพิ่ม TextFormField ใหม่ จบ
TextFormField(validator: _validateTitle, ...),
TextFormField(validator: _validateNote, ...),      // ช่องใหม่ ไม่กระทบช่องเดิม

// จุดส่งมีบรรทัดเดียวตลอดกาล ไม่ว่าฟอร์มจะมีกี่ช่อง:
if (_formKey.currentState!.validate()) { ... }

// validator แยกเป็นฟังก์ชันชื่อชัด (บทที่ 6) — ทดสอบแยกได้ ใช้ซ้ำข้ามฟอร์มได้
String? _validateTitle(String? value) {
  if (value == null || value.trim().isEmpty) return 'กรุณากรอกชื่องาน';
  if (value.trim().length > 50) return 'ชื่องานยาวเกิน 50 ตัวอักษร';
  return null;
}

ระบบ Form คือ "แต่ละช่องรับผิดชอบตัวเอง" — Single Responsibility (บทที่ 6) ในร่างของฟอร์ม เพิ่มช่องที่ 10 ก็ยังอ่านง่ายเท่าช่องแรก

6. ⚠️ ข้อผิดพลาดที่เจอบ่อย

① "A TextEditingController was used after being disposed"

สาเหตุ: ใช้ controller หลังหน้าถูกทำลาย (มักคู่กับงาน async) หรือกลับกัน — ลืม dispose แล้ว Flutter เตือนตอน hot reload/ปิดหน้า
วิธีแก้: ทุก controller ที่สร้างใน State ต้องมีคู่ dispose ใน dispose() เสมอ และงาน async ที่จะแตะ controller ให้เช็ค mounted ก่อน (บทที่ 15)

② กดปุ่มแล้ว validator ไม่ทำงาน — ฟอร์มผ่านทั้งที่ช่องว่าง

สาเหตุ: มี 3 จุดที่ลืมได้: ① ลืมผูก key: _formKey ที่ตัว Form ② ใช้ TextField ธรรมดา (ไม่มีระบบ validator) แทน TextFormField ③ ลืมเรียก validate() ตอน submit
วิธีแก้: เช็คสามจุดนี้ตามลำดับ — จำเป็นชุด: Form(key:) + TextFormField + validate() ขาดตัวไหน ระบบตรวจก็เงียบทั้งระบบ

③ คีย์บอร์ดเด้งขึ้นมาแล้วจอ overflow (แถบเหลือง-ดำโผล่)

สาเหตุ: คีย์บอร์ดกินพื้นที่ครึ่งจอ — ฟอร์มที่เคยพอดีเลยล้น (เพื่อนเก่าจากบทที่ 11 ในเงื่อนไขใหม่: พื้นที่จอ "หดได้" เพราะคีย์บอร์ด)
วิธีแก้: ครอบเนื้อหาฟอร์มด้วย SingleChildScrollView — ฟอร์มเลื่อนได้เมื่อที่ไม่พอ · ท่ามาตรฐาน: body: SingleChildScrollView(child: Form(...))

7. 🏋️ แบบฝึกหัดท้ายบท (ลงมือทำจริง)

8. ✅ เช็กลิสต์และสรุปบทเรียน

  • อธิบายหน้าที่ของ 3 ตัวละคร: TextField (ช่อง), Controller (พนักงานจด), validator (ผู้ตรวจ)
  • เขียนฟอร์มครบชุด Form + TextFormField + validate() ได้เอง
  • ทุก controller มี dispose คู่กัน และอธิบายได้ว่าทำไม
  • validator เขียนแบบ guard clause และข้อความ error บอกวิธีแก้
  • แก้ปัญหาคีย์บอร์ดดันจอด้วย SingleChildScrollView ได้

📖 สรุปศัพท์ใหม่บทนี้

ศัพท์ความหมายสั้นๆภาพจำ
TextFieldช่องพิมพ์ข้อความพื้นฐานช่องกรอกในแบบฟอร์ม
TextEditingControllerตัวจดและอ่านข้อความในช่องพนักงานจดบันทึก
Formกรอบรวมช่องกรอกทั้งชุดแผ่นแบบฟอร์มทั้งใบ
TextFormFieldช่องกรอกที่มีระบบตรวจในตัวช่องกรอกที่มีผู้ตรวจประจำ
validatorฟังก์ชันตรวจค่า: ข้อความ = ไม่ผ่าน, null = ผ่านพนักงานตรวจเอกสาร
GlobalKey<FormState>กุญแจสั่งงานฟอร์มจากข้างนอกบัตรพนักงานประจำฟอร์ม
disposeคืนทรัพยากรเมื่อหน้าถูกทำลายให้พนักงานเลิกงาน

1. 🖼️ ภาพเล่าเรื่อง: ครัวมั่ว กับครัวที่มีสถานี

ร้านอาหารของเราขายดีขึ้นเรื่อยๆ ต้องจ้างเชฟเพิ่ม — และนี่คือจุดที่ครัวสองแบบให้ผลต่างกันคนละโลก:

🌪️ ครัวมั่ว (โค้ดกองใน main.dart) 👨‍🍳 ครัวมืออาชีพ (มีสถานีชัดเจน) มีด กระทะ ผัก เนื้อ ซอส จานชาม ┌─────────────────────────────┐ กองรวมกันบนโต๊ะเดียว... │ 🥬 สถานีวัตถุดิบ (models/) │ เชฟเก่าคนเดียวรู้ว่าอะไรอยู่ไหน │ 🔥 สถานีปรุง (services/) │ (เพราะจำเอา) │ 📢 ผู้จัดการครัว (providers/) │ │ 🍽️ เคาน์เตอร์เสิร์ฟ (screens/) │ เชฟใหม่เข้ามาวันแรก: │ 🧂 อุปกรณ์กลาง (widgets/) │ "มีดอยู่ไหนครับ?" "ใต้กองผักมั้ง..." └─────────────────────────────┘ สองสัปดาห์ผ่านไป ยังไม่กล้าหยิบอะไร เชฟใหม่เข้ามาวันแรก: มองป้ายสถานี 30 วินาที → เริ่มงานได้เลย โครงสร้างโฟลเดอร์ = ป้ายสถานีของครัว: มันไม่ได้ทำให้อาหารอร่อยขึ้น แต่ทำให้ "หลายคนทำอาหารพร้อมกันได้ โดยไม่ชนกัน" — นั่นคือหัวใจของแอปที่โต

ฉากที่ 2 — กฎที่สำคัญกว่าชื่อโฟลเดอร์: ทิศทางการไหลของจาน (dependency direction)

🍽️ screens/widgets ──► 📢 providers ──► 🔥 services ──► 🥬 models (เสิร์ฟ) (ผู้จัดการ) (ปรุง) (วัตถุดิบ) จานไหลจากซ้ายไปขวาเท่านั้น: ชั้นซ้าย "รู้จัก" ชั้นขวาได้ แต่ขวาห้ามรู้จักซ้าย! · สถานีวัตถุดิบ (models) ไม่รู้จักโต๊ะลูกค้า — Task ไม่ import อะไรของ Flutter เลย · สถานีปรุง (services) ไม่รู้ว่าจานจะไปเสิร์ฟหน้าไหน — ใช้ซ้ำได้ทุกหน้า · เพราะงี้ตอนบท 18 เราเปลี่ยนวิธีเก็บข้อมูลได้โดยหน้าจอไม่สะเทือนแม้บรรทัดเดียว!
💡 ความจริงที่ควรรู้

คุณจัดครัวแบบนี้มาตั้งนานแล้ว — models/ เกิดบทที่ 12, services/ บทที่ 19, providers/ บทที่ 17, screens/ บทที่ 15, widgets/ บทที่ 12 — บทนี้ไม่ได้สอนของใหม่ แต่ตั้งชื่อให้สิ่งที่คุณทำมาตลอด แล้วขัดให้เป็นระบบเดียวกันทั้งแอป (หลักการนี้มีชื่อว่า separation of concerns: หนึ่งสถานี หนึ่งหน้าที่ — ญาติผู้ใหญ่ของ "หนึ่งฟังก์ชันหนึ่งงาน" บทที่ 6)

ขั้นที่ 2 — ผังเป้าหมาย (มาตรฐาน layer-first ที่ทีมส่วนใหญ่ใช้เริ่มต้น)

ก่อน (บท 19–20): หลัง (บทนี้): lib/ lib/ ├── main.dart ← หน้าจอ + วิว ├── main.dart ← เหลือจุดสตาร์ท ~20 บรรทัด │ ทั้งหมดยัดรวมในนี้ ├── constants.dart ← ค่ากลางของทั้งแอป (ใหม่!) ├── models/ ├── models/ │ └── weather.dart │ ├── weather.dart └── services/ │ └── city.dart └── weather_service.dart ├── services/ │ └── weather_service.dart ├── screens/ │ └── weather_screen.dart ← ย้ายหน้าจอมาบ้านตัวเอง └── widgets/ ├── weather_view.dart ← _WeatherView เดิม └── error_view.dart ← _ErrorView เดิม (ตัด _ ออก เพราะจะใช้ข้ามไฟล์แล้ว)
🤔 วิธีย้ายอย่างปลอดภัย

ลากไฟล์ใน VS Code (Explorer ฝั่งซ้าย) → VS Code ถามว่า "อัปเดต import ให้ไหม" → Yes เสมอ — ย้ายทีละไฟล์ รันเช็ค แล้วค่อยย้ายตัวถัดไป (จังหวะ Dev Loop บทที่ 3 ใช้กับ refactor ด้วย)

ขั้นที่ 3 — เกิดไฟล์ใหม่: lib/constants.dart — บอร์ดค่ากลางของร้าน

ค่าที่กระจายอยู่ทั่วแอป (URL, timeout, ระยะห่าง) ย้ายมารวมที่เดียว:

/// ค่ากลางของทั้งแอป — แก้ที่นี่ที่เดียว มีผลทุกจุด (DRY บทที่ 6 ระดับทั้งแอป)
class AppConstants {
  AppConstants._();   // constructor ปิดตาย — คลาสนี้มีไว้เก็บค่า ไม่ให้ใครสร้าง object

  static const apiBaseUrl = 'https://api.open-meteo.com/v1';
  static const requestTimeout = Duration(seconds: 10);
}

class AppSpacing {
  AppSpacing._();
  static const s = 8.0;    // ชุดระยะห่าง 8/16/24 จากบทที่ 11
  static const m = 16.0;   // — วันนี้ได้บ้านถาวรสักที
  static const l = 24.0;
}

ฝั่ง service เปลี่ยนเป็น:

final url = Uri.parse('${AppConstants.apiBaseUrl}/forecast?latitude=$lat&...');
final response = await http.get(url).timeout(AppConstants.requestTimeout);

ขั้นที่ 4 — ตรวจรับงานด้วย "การทดสอบคนใหม่"

จัดเสร็จแล้ว วัดผลด้วยคำถามเดียว: ถ้าเพื่อนที่เขียน Flutter เป็น เปิดโปรเจกต์คุณ 30 วินาที เขาตอบได้ไหมว่า — แอปนี้มีกี่หน้า? ข้อมูลหน้าตาเป็นยังไง? คุยกับ API ที่ไหน? — ถ้าคำตอบคือ "เปิดโฟลเดอร์ screens/models/services ก็เห็นเลย" = ผ่าน

3. 🔧 เทคนิคการประยุกต์ใช้

เทคนิคที่ 1 — บรรทัด import คือใบตรวจสุขภาพของโครงสร้าง

เปิดไฟล์ไหนก็ได้ ดูหัวไฟล์: models/ ที่มี import 'package:flutter/...' = วัตถุดิบแอบรู้จักโต๊ะลูกค้า ผิดสถานี! · services/ ที่ import ไฟล์ใน screens/ = จานไหลย้อนทาง — นิสัยสแกน import 5 วินาทีก่อน commit จับโครงสร้างเสื่อมได้ตั้งแต่เนิ่นๆ

เทคนิคที่ 2 — layer-first ตอนเล็ก, feature-first ตอนโต

ผังของเรา (จัดตาม "ประเภทสถานี") เหมาะกับแอปเล็ก-กลาง — แอปใหญ่ระดับ 20+ หน้าจอนิยม feature-first: จัดตาม "เมนู" แทน (lib/features/weather/ ที่มี models/services/screens ของตัวเองครบในนั้น) — หลักการเดียวกัน แค่หั่นคนละแกน รู้ไว้จะได้ไม่งงตอนเปิดโปรเจกต์คนอื่น

เทคนิคที่ 3 — ตั้งชื่อไฟล์ให้ตะโกนหน้าที่ตัวเอง

weather_service.dart, error_view.dart, task_provider.dart — ลงท้ายด้วยประเภทเสมอ (_service, _view, _provider, _screen) เห็นชื่อในผล search ก็รู้สถานีทันทีโดยไม่ต้องดู path (ต่อยอดกฎ snake_case บทที่ 2)

เทคนิคที่ 4 — README ต้องมีผังครัวเสมอ

เพิ่มหัวข้อ "โครงสร้างโปรเจกต์" ใน README: ผัง tree + หนึ่งบรรทัดต่อโฟลเดอร์ว่าเก็บอะไร (แบบที่ my_first_app ทำตั้งแต่บทที่ 1!) — นี่คือสิ่งแรกที่ dev ใหม่และตัวคุณเองในอีก 6 เดือนจะอ่าน

4. 🏭 ตัวอย่างการเขียนจริงในงานจริง

🏭 สถานการณ์จริง: "PR แรกของ dev ใหม่ — ตัววัดสุขภาพของโครงสร้าง"

ทีมจริงมีตัวชี้วัดที่พูดถึงกันบ่อย: dev ใหม่ใช้เวลากี่วันกว่าจะส่งงานชิ้นแรกได้ — และโครงสร้างโปรเจกต์คือตัวแปรใหญ่ที่สุด:

ทีม A (โครงสร้างมั่ว): ทีม B (โครงสร้างดี): วันที่ 1-3: ไล่อ่าน main.dart 4,000 บรรทัด วันที่ 1 เช้า: อ่าน README + ผังโฟลเดอร์ วันที่ 4-7: ถามรุ่นพี่วันละสิบรอบ วันที่ 1 บ่าย: ได้งาน "เพิ่ม field ใน วันที่ 8: แก้จุดเดียว แต่พังอีกสามจุด โปรไฟล์" → รู้ทันทีว่าแตะ 3 ไฟล์: (เพราะทุกอย่างพันกันหมด) models/user.dart, services/user_ วันที่ 14: ส่ง PR แรก... ที่แก้ผิดที่ service.dart, screens/profile_screen วันที่ 2: PR แรกผ่านรีวิว ✓ และปัญหาแบบทีม A ไม่ได้เจ็บแค่คนใหม่ — ทุกคนในทีมจ่ายภาษีนี้ทุกวัน: งานสองคนชนกันใน main.dart ตลอด (merge conflict), แก้บั๊กหนึ่งตัวต้องอ่านทั้งไฟล์, ประเมินเวลางานไม่ได้เพราะไม่รู้ว่าจะไปพันกับอะไร

โครงสร้างที่ดีจึงไม่ใช่เรื่อง "ความสวยงาม" — มันคือต้นทุนต่อวันของทั้งทีม และเป็นเหตุผลว่าทำไมคำถามสัมภาษณ์ยอดฮิตคือ "โปรเจกต์ของคุณจัดโครงสร้างยังไง เพราะอะไร" — ตอนนี้คุณมีคำตอบพร้อมเหตุผลครบทุกชั้นแล้ว

5. 🧹 และแล้ว Clean Architecture ล่ะ?

คุณอาจได้ยินคำว่า Clean Architecture จากประกาศรับสมัครงานหรือบทความ — มันคืออะไร แล้วเราต้องใช้ไหม?

สิ่งที่เราทำวันนี้ (พอดีตัวสำหรับแอปเล็ก-กลาง): screens → providers → services → models (4 ชั้น กฎไหลทางเดียว) Clean Architecture เต็มรูปแบบ (ครัวโรงแรมห้าดาว — สำหรับแอปองค์กรขนาดใหญ่): UI → Controllers → Use Cases → Repositories → Entities (ชั้นละเอียดกว่า + กฎเหล็กเพิ่ม: ชั้นในสุดห้ามรู้จักโลกภายนอกสิ้นเชิง + สลับ "ครัวทั้งครัว" ได้ผ่านสัญญากลาง)

หัวใจของทั้งสองแบบคือประโยคเดียวกันเป๊ะ: แยกหน้าที่เป็นชั้น + การพึ่งพาไหลทางเดียว — ที่คุณเข้าใจแล้ววันนี้คือ แก่นแท้ ส่วน Clean Architecture เต็มรูปแบบคือแก่นเดิมที่เพิ่มพิธีการสำหรับทีมใหญ่/แอปใหญ่

📎 กล่อง "เมื่อพร้อม"

อย่าเพิ่งเอา Clean Architecture เต็มรูปแบบมาใส่แอป 5 หน้า — พิธีการที่เกินขนาดงานคือความซับซ้อนฟรี (over-engineering) เหมือนสร้างครัวโรงแรมห้าดาวเพื่อขายข้าวมันไก่หน้าปากซอย เรียนรู้เพิ่มเมื่อ (1) เข้าทีมที่ใช้อยู่ หรือ (2) แอปโตจนชั้น service เริ่มพันกันเอง — ถึงวันนั้น พื้นฐานวันนี้จะทำให้เข้าใจมันในไม่กี่ชั่วโมง (อยู่ในบทเสริมท้ายหลักสูตร)

6. ⚠️ ข้อผิดพลาดที่เจอบ่อย (ของการจัดบ้านครั้งใหญ่)

① ย้ายไฟล์รวดเดียวทั้งโปรเจกต์ แล้ว import แดงทั้งจอ

สาเหตุ: ลากย้ายทุกไฟล์พร้อมกัน / ย้ายนอก VS Code (ผ่าน File Explorer) — ไม่มีใครอัปเดต import ให้ → error หลายสิบจุดพร้อมกัน มือใหม่ตกใจ undo รัวจนโปรเจกต์พังหนักกว่าเดิม
วิธีแก้: ย้ายทีละไฟล์ผ่าน VS Code (ตอบ Yes ให้อัปเดต import) → รันเช็ค → ย้ายต่อ · ถ้าพลาดไปแล้ว: อ่าน error ทีละบรรทัด — ทุกตัวคือ "หา path ไม่เจอ" แก้ตรงไปตรงมา (และถ้าทำขั้นที่ 1 ไว้ git checkout กลับได้เสมอ)

② จัดบ้านไป เพิ่มฟีเจอร์ไป — จบวันไม่รู้ว่าพังเพราะอะไร

สาเหตุ: refactor (จัดของเดิม) ปนกับ feature (เพิ่มของใหม่) ใน commit เดียว — พอพังไม่รู้ว่าพังจากการย้าย หรือจากของใหม่ (ผิดกฎ "เปลี่ยนทีละอย่าง" บท 3)
วิธีแก้: กฎเหล็กของมืออาชีพ: refactor คือ commit ของมันเอง — "จัดบ้านเสร็จ รันผ่าน commit ก่อน แล้วค่อยเริ่มของใหม่" ประวัติ git จะเล่าเรื่องชัดเจนด้วย

③ จัดเกินขนาดงาน (over-engineering)

สาเหตุ: อ่านบทความมาเยอะ เลยสร้าง 12 โฟลเดอร์ให้แอป 3 หน้า — ทุกฟีเจอร์ใหม่ต้องแตะ 6 ไฟล์ งานช้าลงเปล่าๆ
วิธีแก้: หลักจากบทที่ 18 ใช้ได้เสมอ: "เริ่มจากเล็กสุดที่พอ อัปเกรดเมื่อเจ็บจริง" — 5 โฟลเดอร์ของวันนี้รองรับแอปได้ถึงหลักสิบหน้า สัญญาณว่าถึงเวลาขยับ: โฟลเดอร์เดียวมีไฟล์เกิน ~10 ตัว หรือหาไฟล์ไม่เจอบ่อยขึ้น

7. 🏋️ แบบฝึกหัดท้ายบท (ลงมือทำจริง)

🥉 ระดับ 1 — จัดบ้าน

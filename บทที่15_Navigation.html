1. 🖼️ ภาพเล่าเรื่อง: กองจานของหน้าจอ

แอปมือถือจัดการหลายหน้าด้วยวิธีเดียวกับ การซ้อนจาน: หน้าใหม่ถูก "วางทับ" หน้าเดิม — หน้าเดิมไม่หายไปไหน แค่ถูกบัง รอวันที่จานข้างบนถูกยกออก

ผู้ใช้ทำอะไร กองจาน (Navigation Stack) เปิดแอป ┌──────────────┐ │ หน้าหลัก │ ← จานใบแรก └──────────────┘ แตะงาน "ซื้อนม" ┌──────────────┐ → push หน้ารายละเอียด │ รายละเอียด │ ← จานใหม่วางทับ ├──────────────┤ │ หน้าหลัก │ ← ยังอยู่! แค่ถูกบัง └──────────────┘ (state ไม่หาย) กดปุ่ม ← ย้อนกลับ ┌──────────────┐ → pop จานบนออก │ หน้าหลัก │ ← โผล่กลับมา └──────────────┘ เหมือนเดิมทุกอย่าง push = วางจานทับ · pop = ยกจานบนออก · ปุ่ม back ของมือถือ = pop นั่นเอง
🤔 ทำไมต้องเป็น "กอง" (stack)

เพราะพฤติกรรมผู้ใช้คือ "เข้าไปลึกแล้วย้อนกลับทางเดิม" — เปิดแชท → เข้าห้อง → ดูรูป → กดกลับ → ห้องแชท → กดกลับ → รายชื่อ ลำดับย้อนกลับต้องตรงกับขาเข้าเป๊ะ โครงสร้างกองจานให้พฤติกรรมนี้ฟรีๆ และ Flutter จัดการให้หมดผ่าน Navigator

2. 💻 ลงมือจริง: หน้ารายละเอียดงาน

ขั้นที่ 1 — สร้างหน้าใหม่ที่ "รับข้อมูลผ่าน constructor"

สร้างไฟล์ lib/screens/task_detail_screen.dart (โครงสร้างโฟลเดอร์จากบทที่ 2):

import 'package:flutter/material.dart';
import '../models/task.dart';

/// หน้ารายละเอียดงาน — Stateless เพราะรับข้อมูลมาแสดงเฉยๆ (ตารางบทที่ 10)
/// สังเกต: หน้าใหม่ก็คือ Widget ธรรมดา + รับข้อมูลผ่าน constructor เหมือนที่เคยทำ
class TaskDetailScreen extends StatelessWidget {
  const TaskDetailScreen({super.key, required this.task});

  final Task task;   // ← ข้อมูลเดินทางข้ามหน้า ด้วยท่าเดิมจากบทที่ 12!

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      // AppBar ของหน้าที่ถูก push จะมีปุ่ม ← ย้อนกลับให้อัตโนมัติ (ไม่ต้องเขียนเอง!)
      appBar: AppBar(title: const Text('รายละเอียดงาน')),
      body: Padding(
        padding: const EdgeInsets.all(16),
        child: Column(
          crossAxisAlignment: CrossAxisAlignment.start,
          children: [
            Text(task.title,
                style: const TextStyle(fontSize: 24, fontWeight: FontWeight.bold)),
            const SizedBox(height: 8),
            Text(task.isDone ? '✅ เสร็จแล้ว' : '⏳ ยังไม่เสร็จ',
                style: const TextStyle(fontSize: 16, color: Colors.grey)),
          ],
        ),
      ),
    );
  }
}

ขั้นที่ 2 — แตะแถวงานแล้ว push

ในหน้าหลัก เพิ่ม onTap ให้แถวงาน (ผ่าน callback ของ TaskTile — ท่าเดิมบทที่ 12):

onTap: () {
  Navigator.push(
    context,
    MaterialPageRoute(
      // builder = สูตรสร้างหน้าปลายทาง (route = เส้นทางไปหน้านั้น)
      builder: (context) => TaskDetailScreen(task: task),
    ),
  );
},

รันแล้วแตะงานดู: หน้ารายละเอียดสไลด์เข้ามา พร้อมปุ่ม ← ที่กดย้อนกลับได้ — คุณเพิ่งได้แอนิเมชันเปลี่ยนหน้าตามมาตรฐานของแต่ละแพลตฟอร์ม (iOS สไลด์จากขวา, Android เฟดขึ้น) ฟรีจาก MaterialPageRoute

ขั้นที่ 3 — รับผลลัพธ์กลับจากหน้าปลายทาง

ของจริงกว่านั้น: หน้ารายละเอียดมี ปุ่มลบ — แต่กระดานกลาง (_tasks) อยู่หน้าหลัก จะบอกกันยังไง? ใช้ท่า "pop พร้อมพัสดุ":

// ── ฝั่งหน้ารายละเอียด: ปุ่มลบ ──
ElevatedButton.icon(
  icon: const Icon(Icons.delete),
  label: const Text('ลบงานนี้'),
  onPressed: () {
    Navigator.pop(context, true);   // ยกจานออก + แนบผลลัพธ์ "true = ขอลบ" กลับไป
  },
)

// ── ฝั่งหน้าหลัก: push แบบรอฟังคำตอบ (บัตรคิวจากบทที่ 9 กลับมา!) ──
onTap: () async {
  final shouldDelete = await Navigator.push<bool>(
    context,
    MaterialPageRoute(builder: (context) => TaskDetailScreen(task: task)),
  );
  // โค้ดตรงนี้ทำงาน "หลังหน้ารายละเอียดถูก pop กลับมา"
  if (shouldDelete == true) {
    setState(() => _tasks.remove(task));   // ลบที่กระดานกลาง — จอทุกจุดอัปเดต
  }
},

สังเกต: push คืน Future — เพราะ "การรอผู้ใช้กลับมาจากอีกหน้า" ก็คืองานที่ต้องรอ (async บทที่ 9 ในบริบทจริงครั้งแรก!) และผลลัพธ์เป็น bool? เพราะผู้ใช้อาจกด ← เฉยๆ โดยไม่กดลบ (null safety บทที่ 4 ทำงานพอดี: เช็ค == true จึงปลอดภัยทั้ง null)

3. 🔧 เทคนิคการประยุกต์ใช้

เทคนิคที่ 1 — หนึ่งหน้า หนึ่งไฟล์ ใน lib/screens/ เสมอ

ตั้งชื่อลงท้าย Screen (TaskDetailScreen, SettingsScreen) และไฟล์ snake_case ตรงกัน — พอแอปมี 15 หน้า โฟลเดอร์ screens จะเป็น "สารบัญแอป" ที่มองปราดเดียวรู้ว่ามีหน้าอะไรบ้าง

เทคนิคที่ 2 — ส่ง object ทั้งก้อน ไม่ใช่แยกส่ง field

TaskDetailScreen(task: task) ✓ ไม่ใช่ TaskDetailScreen(title: ..., isDone: ...) ✗ — วันที่ Task มี field เพิ่ม (กำหนดส่ง, หมวดหมู่) แบบแรกไม่ต้องแก้อะไรเลย แบบหลังต้องไล่แก้ทุกจุดที่เรียก (เหตุผลเดียวกับที่สร้าง class ในบทที่ 8)

เทคนิคที่ 3 — ผลลัพธ์จาก pop ถือว่า "อาจเป็น null เสมอ"

ผู้ใช้มีทางกลับที่เราคุมไม่ได้เสมอ (ปุ่ม back ของเครื่อง, ปัดขอบจอ) ซึ่ง pop โดยไม่แนบค่า → เขียนเงื่อนไขแบบระบุชัด if (result == true) ไม่ใช่ if (result != null) แล้วเดาเอา

เทคนิคที่ 4 — อย่าลืมว่าหน้าเดิม "ยังมีชีวิต" ใต้กอง

จานล่างไม่ถูกทำลาย state ยังครบ — ดังนั้นไม่ต้องส่งข้อมูลกลับมา "สร้างหน้าเดิมใหม่" และในทางกลับกัน ระวังงานหนักที่หน้าเดิมยังรันต่อทั้งที่ถูกบังอยู่

4. 🏭 ตัวอย่างการเขียนจริงในงานจริง

🏭 สถานการณ์จริง: "เส้นทางสั่งซื้อของแอปช้อปปิ้ง"

ทุกการสั่งซื้อที่คุณเคยกด คือกองจานที่ออกแบบมาอย่างจงใจ:

หน้ารายการสินค้า → หน้าสินค้า → ตะกร้า → ที่อยู่จัดส่ง → จ่ายเงิน → สำเร็จ กองจาน ณ ขั้นจ่ายเงิน: การตัดสินใจออกแบบที่ทีมจริงถกกัน: ┌────────────┐ │ จ่ายเงิน │ · กด back จากหน้า "จ่ายเงิน" → กลับ "ที่อยู่" ✓ │ ที่อยู่จัดส่ง │ (ผู้ใช้แก้ที่อยู่ได้ ไม่ต้องเริ่มใหม่) │ ตะกร้า │ · แต่หน้า "สำเร็จ" ห้ามกด back กลับไป "จ่ายเงิน"! │ หน้าสินค้า │ (เดี๋ยวจ่ายซ้ำ) → ทีมใช้ท่า pop ทั้งกองกลับหน้าแรก │ หน้ารายการ │ · ปิด popup ยืนยัน = pop(false), กดตกลง = pop(true) └────────────┘ — ท่าเดียวกับปุ่มลบที่คุณเพิ่งเขียนเป๊ะ

การออกแบบ "กด back แล้วเกิดอะไร" ในแต่ละจุด คือหัวข้อที่ PM, ดีไซเนอร์ และ dev ประชุมร่วมกันจริงจัง — เพราะ back ที่พฤติกรรมเพี้ยนคือสาเหตุอันดับต้นๆ ที่ผู้ใช้ลบแอป และตอนนี้คุณมีเครื่องมือครบที่จะร่วมวงสนทนานั้น

5. 🧹 ตัวอย่าง Clean Code: การเดินทางที่อ่านออก

// ❌ ยัดหน้าใหม่เป็น Widget ก้อนใหญ่ใน main.dart + ส่งข้อมูลแบบแตกกระจาย
Navigator.push(context, MaterialPageRoute(builder: (context) => Scaffold(

  body: Column(children: [
    Text(title), Text(isDone ? 'เสร็จ' : 'ยัง'), // รับ field แยก 5 ตัว
    /* ...อีก 40 บรรทัด... */
  ]),

// ✅ หน้ามีชื่อ มีบ้าน — จุด push อ่านเป็นภาษาคน
final shouldDelete = await Navigator.push<bool>(

🧠 บันไดเดิม ขั้นใหญ่ขึ้น

สังเกตอีกครั้ง: กฎ "หนึ่งไฟล์หนึ่งหน้าที่" (บทที่ 2) → ฟังก์ชันผอม (บทที่ 6) → Widget ย่อย (บทที่ 11) → หนึ่งหน้าหนึ่งไฟล์ (บทนี้) — หลักการเดียวกันตลอดทาง

6. ⚠️ ข้อผิดพลาดที่เจอบ่อย

① "Navigator operation requested with a context that does not include a Navigator"

สาเหตุ: เรียก Navigator.push ด้วย context ที่อยู่ "เหนือ" MaterialApp — Navigator เกิดข้างใน MaterialApp การขอใช้จาก context ที่สูงกว่ามันเลยหาไม่เจอ (เจอบ่อยตอนใส่ปุ่มใน MyApp/build แรกสุดของแอป)
วิธีแก้: ย้ายปุ่ม/การ push ไปอยู่ใน Widget ที่เป็นลูกของ MaterialApp (เช่นในหน้า HomeScreen) — เจอ error นี้ = เช็คว่ากำลัง push จากตรงไหนของต้นไม้ Widget

② แก้ข้อมูลในหน้ารายละเอียด แล้วหน้าหลักไม่เปลี่ยน

สาเหตุ: เพื่อนเก่าจากบทที่ 12 — "ความจริงสองชุด": หน้ารายละเอียดแก้ตัวแปรของตัวเอง (สำเนา) แต่กระดานกลางอยู่หน้าหลัก
วิธีแก้: ยึดหลักเดิม: การแก้ข้อมูลต้องเกิดที่เจ้าของกระดาน — ส่งผลลัพธ์กลับด้วย pop(ค่า) แล้วให้หน้าหลัก setState เอง (แบบปุ่มลบในหัวข้อ 2) หรือส่ง callback ลงไปเหมือน TaskTile ก็ได้

③ คำเตือน "Do not use BuildContexts across async gaps"

สาเหตุ: หลัง await (เช่นรอ push กลับมา) แล้วใช้ context ต่อ — ระหว่างที่รอ ผู้ใช้อาจปิดหน้านี้ไปแล้ว context เลยอาจ "ตายแล้ว" (ญาติของ "setState after dispose" ที่เจอชื่อในบทที่ 10)
วิธีแก้: เช็คก่อนใช้: if (!context.mounted) return; หลัง await ก่อนแตะ context — ตอนนี้จำท่านี้ไว้ใช้ตามได้เลย เหตุผลลึกๆ จะชัดมากตอนเฟส 4

7. 🏋️ แบบฝึกหัดท้ายบท (ลงมือทำจริง)

🥉 ระดับ 1 — ประตูบานแรก

🥇 ระดับ 3 — pop พร้อมพัสดุ

เพิ่มปุ่มลบในหน้ารายละเอียดตามขั้นที่ 3: pop(context, true) → หน้าหลัก await แล้วลบจริง + แสดง SnackBar "ลบแล้ว" (บทที่ 13) — โบนัสท้าทาย: เพิ่มปุ่ม "ติ๊กเสร็จ" ในหน้ารายละเอียดด้วย โดยห้ามสร้างความจริงสองชุด (คิดดูว่าจะใช้ pop ส่งผล หรือส่ง callback ลงไป — ทั้งคู่ถูก เลือกแล้วอธิบายได้ว่าทำไม) — commit ด้วย feat: delete and toggle from detail screen

8. ✅ เช็กลิสต์และสรุปบทเรียน

  • อธิบาย navigation stack ด้วยภาพกองจานได้ (push วางทับ / pop ยกออก)
  • สร้างหน้าใหม่ใน lib/screens/ และ push/pop ได้คล่อง
  • ส่งข้อมูลไปหน้าใหม่ผ่าน constructor (ทั้ง object ไม่แยก field)

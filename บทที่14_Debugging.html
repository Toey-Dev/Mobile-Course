1. 🖼️ ภาพเล่าเรื่อง: นักสืบสองแบบ

บั๊กเกิดขึ้นแล้วในแอปของคุณ — คุณคือนักสืบที่ต้องหาตัวคนร้าย มีเครื่องมือสองระดับ:

🔦 นักสืบสายสอบพยาน (print/debugPrint) 🕰️ นักสืบสายหยุดเวลา (debugger) ถามพยานทีละคำถาม: กดปุ่มหยุดเวลา ณ จุดเกิดเหตุ: "ตอนนั้น _tasks มีกี่ชิ้น?" ทุกอย่างค้างนิ่ง — แล้วเดินสำรวจ → รันใหม่ → อ่านคำตอบ ได้ทั้งห้อง: "แล้ว index เป็นเท่าไหร่?" · ตัวแปรทุกตัว มีค่าอะไร (Variables) → เพิ่ม print → รันใหม่ → อ่านอีก · โค้ดเดินมาทางไหน (Call Stack) · สั่งเดินต่อทีละบรรทัด (Step) เหมาะกับ: คำถามเล็กๆ เร็วๆ เหมาะกับ: คดีซับซ้อน ตัวแปรเยอะ จุดอ่อน: ถามได้ทีละคำถาม ต้องรันใหม่เรื่อยๆ ไม่ต้องรันใหม่ ไม่ต้องเดา — เห็นหมด

และหลักฐานชิ้นแรกของทุกคดี: stack trace (รอยเท้าคนร้าย)

══ Exception caught by widgets library ═══════════════════ RangeError (index): Invalid value: Not in inclusive range 0..2: 3 ▲ ① เกิดอะไร: หยิบช่อง 3 จาก List ที่มีถึงช่อง 2 #0 List.[] (dart:core-patch/array.dart) ← ของ Dart เอง — ข้ามไป #1 _HomeScreenState._removeTask (package:todo_app/main.dart:42) #2 TaskTile.build.<anonymous> (...) ▲ ② ที่เกิดเหตุ: ไฟล์เรา! ...อีก 30 บรรทัด (ของ framework — ไม่ต้องอ่าน) main.dart บรรทัด 42 วิธีอ่าน: ① บรรทัดบนสุด = เกิดอะไรขึ้น ② ไล่หาบรรทัดแรกที่เป็น "ไฟล์ของเรา" (package:todo_app/...) = ไปดูตรงนั้น ที่เหลือทั้งหมดข้ามได้ — อย่ายอมแพ้เพราะมันยาว มันยาวเพื่อคนอื่น ไม่ใช่เรา

2. 💻 ลงมือจริง: หยุดเวลาครั้งแรกด้วย Breakpoint

ขั้นที่ 1 — รันแบบ debug ด้วย F5 (ไม่ใช่ flutter run ธรรมดา)

ใน VS Code เปิดโปรเจกต์ todo_app → กด F5 (หรือเมนู Run → Start Debugging) — หน้าตาเหมือน flutter run แต่คราวนี้ VS Code "ถือกุญแจหยุดเวลา" อยู่

ขั้นที่ 2 — วางกับดัก (breakpoint)

คลิกซ้ายสุดของบรรทัด setState(() => task.toggle()); ใน _toggleTask → จุดแดง 🔴 ปรากฏ = กับดักพร้อม

ขั้นที่ 3 — รอคนร้ายเหยียบกับดัก

ไปที่แอป → ติ๊กงานหนึ่งชิ้น → เวลาหยุดทันที! VS Code เด้งกลับมาพร้อมไฮไลต์บรรทัดที่หยุด:

สิ่งที่เห็นตอนเวลาหยุด (Debug Sidebar ฝั่งซ้าย): ▾ VARIABLES แถบควบคุมด้านบน: task: Task ▶️ Continue (F5) = ปล่อยเวลาเดินต่อ title: "อ่านบทเรียนที่ 12" ⤵️ Step Over (F10) = เดินทีละบรรทัด isDone: false ← ยังไม่ toggle! ⤷ Step Into (F11) = มุดเข้าไปในฟังก์ชันที่กำลังเรียก _tasks: List (2 items) ▾ CALL STACK _toggleTask ← อยู่ตรงนี้ TaskTile.build ← ถูกเรียกมาจากตรงนี้ (รอยเท้าย้อนกลับ!)

กด Step Over (F10) หนึ่งครั้ง → ดูค่า isDone ใน VARIABLES พลิกเป็น true ต่อหน้าต่อตา — คุณเพิ่งเห็นโค้ดทำงานทีละบรรทัดด้วยตาตัวเองครั้งแรก ไม่ต้องเดา ไม่ต้อง print

ขั้นที่ 4 — DevTools: ศูนย์บัญชาการใหญ่

ตอน debug อยู่ VS Code จะมีลิงก์ "Open DevTools" (หรือ Command Palette → "Open DevTools") — เครื่องมือชุดใหญ่ในเบราว์เซอร์: Widget Inspector (เพื่อนเก่าบทที่ 3), กราฟการใช้แรม/CPU, บันทึก network — ตอนนี้แค่เปิดดูให้รู้ว่ามีอะไร จะได้ใช้จริงหนักๆ ตอนเฟส 4

3. 🧠 คอนเซปต์: ขั้นตอนไล่ล่า 4 ขั้นของนักสืบมืออาชีพ

การไล่บั๊กไม่ใช่ดวง — มันคือกระบวนการวิทยาศาสตร์ที่ทำซ้ำได้:

① ทำให้เกิดซ้ำได้ (Reproduce) "กดอะไร ตามลำดับไหน แล้วพังทุกครั้ง?" │ บั๊กที่ทำซ้ำได้ = จับได้แน่นอน ▼ ② ตั้งสมมติฐาน (Hypothesize) "น่าจะเป็นเพราะ index เกิน ตอนลบแถวสุดท้าย" │ ห้ามแก้อะไรก่อนมีสมมติฐาน! ▼ ③ พิสูจน์ (Verify) วาง breakpoint / print ดูค่า ณ จุดสงสัย │ สมมติฐานผิด? → กลับไปข้อ ② (ปกติมาก) ▼ ④ แก้ แล้วยืนยันว่าหายจริง (Fix) แก้ที่ "สาเหตุ" ไม่ใช่กลบ "อาการ" ทำซ้ำตามข้อ ① อีกรอบ — ต้องไม่พังแล้ว
⚠️ กับดักที่มือใหม่ตกบ่อยที่สุด

ข้ามข้อ ②③ ไป "ลองแก้ตรงนั้นตรงนี้ดู" จนบั๊กหายไปเอง — อันตรายที่สุด เพราะไม่รู้ว่าหายเพราะอะไร บั๊กแบบนี้จะกลับมาตอนสำคัญเสมอ (มักเป็นคืนก่อนส่งงาน)

4. 🔧 เทคนิคการประยุกต์ใช้

เทคนิคที่ 1 — เลือกเครื่องมือตามขนาดคดี

คำถามเล็ก ("ค่านี้เป็นอะไร") → debugPrint เร็วสุด · คดีใหญ่ (ค่าหลายตัว, โค้ดวิ่งงงๆ, ลูปพิลึก) → breakpoint คุ้มกว่าเพราะเห็นทุกอย่างในรอบเดียว — มือโปรใช้ทั้งคู่สลับกัน ไม่มีอันไหน "เก่งกว่า"

เทคนิคที่ 2 — เทคนิคผ่าครึ่ง (binary search) หาที่เกิดเหตุ

โค้ดยาวไม่รู้พังตรงไหน: วาง debugPrint('ถึงจุด A') กลางทาง — ถ้าพิมพ์ออกมา แปลว่าพังหลังจุด A → ย้ายไปกลางครึ่งหลัง ทำซ้ำ — หาจุดพังในโค้ด 1,000 บรรทัดได้ใน ~10 รอบ

เทคนิคที่ 3 — Conditional breakpoint: กับดักที่เลือกเหยื่อ

บั๊กเกิดเฉพาะแถวที่ 57 จาก 100? คลิกขวาที่จุดแดง → Edit Breakpoint → ใส่เงื่อนไข index == 57 — เวลาจะหยุดเฉพาะตอนเงื่อนไขจริง ไม่ต้องกด Continue 56 รอบ

เทคนิคที่ 4 — จดสมุดบั๊ก (bug journal)

เจอบั๊กที่ใช้เวลาเกิน 30 นาที ให้จดสั้นๆ: อาการ / สาเหตุจริง / วิธีเจอ — ผ่านไป 3 เดือนสมุดนี้คือคลังความรู้เฉพาะตัวที่หาจากไหนไม่ได้ และบั๊กหน้าตาเดิมจะจบใน 5 นาที

5. 🏭 ตัวอย่างการเขียนจริงในงานจริง

🏭 สถานการณ์จริง: "บั๊กจากรีวิวผู้ใช้ — ทีมจริงไล่ยังไง"

รีวิว 1 ดาวเข้ามา: "กดลบงานแล้วแอปเด้งตลอด!" — แต่ทีมลองกดลบแล้ว...ปกติดี?? นี่คือสถานการณ์จริงที่เจอทุกสัปดาห์ และขั้นตอน ①②③④ ทำงานแบบนี้:

① Reproduce: ทีมไล่ถามเงื่อนไข — ผู้ใช้ลบ "แถวสุดท้าย" ขณะที่เปิดตัวกรอง "เฉพาะงานเสร็จ" อยู่ → ลองตาม = เด้งจริง! (บั๊กเกิดเฉพาะเงื่อนไขซ้อนกัน) ② Hypothesize: "ตัวกรองทำให้ index บนจอ ≠ index ใน List จริง → removeAt(index) เลยลบผิดช่อง/เกินช่วง" ③ Verify: วาง conditional breakpoint ที่ removeAt → ค่า index = 3 แต่ List จริงหลังกรองมี 2 ชิ้น → สมมติฐานถูก ✓ ④ Fix: เปลี่ยนจากลบด้วย index → ลบด้วยตัว object (_tasks.remove(task)) แล้วทำซ้ำข้อ ① — ไม่เด้งแล้ว + เขียนกันไว้ไม่ให้ใครพลาดซ้ำ

สังเกตว่าเครื่องมือทุกตัวในเรื่องนี้ คุณมีครบแล้วในบทนี้ — ต่างแค่ประสบการณ์ ซึ่งสร้างได้ด้วยการฝึกล่าจริงในแบบฝึกหัด

6. 🧹 ตัวอย่าง Clean Code: โค้ดที่ debug ง่าย คือโค้ดที่เขียนดีมาแต่แรก

บั๊กเดียวกัน ในโค้ดสองแบบ — ดูว่า stack trace ช่วยเราได้ต่างกันแค่ไหน:

// ❌ โค้ดก้อนยักษ์ — stack trace ชี้ได้แค่ "พังในก้อนนี้แหละ 80 บรรทัด"
void _handleEverything(int index, bool? value, String? action) {
  // ...ติ๊กก็ทำ ลบก็ทำ เพิ่มก็ทำ กรองก็ทำ ปนกัน 80 บรรทัด...
  // stack trace บอกแค่: _handleEverything (main.dart:57)
  // → ยังต้องไล่อ่านทั้ง 80 บรรทัดว่าเส้นทางไหนพัง
}
// ✅ ฟังก์ชันเล็ก หน้าที่เดียว — stack trace กลายเป็นลายแทงชี้จุด
void _toggleTask(Task task) { ... }     // พังที่นี่ = เรื่องติ๊กเท่านั้น
void _removeTask(Task task) { ... }     // พังที่นี่ = เรื่องลบเท่านั้น
List<Task> _visibleTasks() { ... }      // พังที่นี่ = เรื่องกรองเท่านั้น

// stack trace: _removeTask (main.dart:42) → รู้ทันทีว่าคดีอยู่หมวดไหน
// ฟังก์ชัน 5 บรรทัด อ่านแป๊บเดียวเจอ — ชื่อฟังก์ชันคือหมวดคดีในตัวเอง
🧠 หลักทอง

ทุกเทคนิค clean code ที่เรียนมา (ฟังก์ชันเล็ก บทที่ 6, ชื่อสื่อความหมาย บทที่ 4, Widget ย่อย บทที่ 11) ล้วนจ่ายดอกเบี้ยตอน debug — โค้ดที่อ่านง่าย คือโค้ดที่บั๊กซ่อนตัวยาก

7. ⚠️ ข้อผิดพลาดที่เจอบ่อย (ของการ debug เอง)

① Breakpoint ไม่ยอมหยุด (จุดแดงกลายเป็นวงกลมโปร่ง)

สาเหตุ: แอปรันด้วย flutter run ใน terminal ธรรมดา — VS Code ไม่ได้ "ถือกุญแจ" เลยสั่งหยุดเวลาไม่ได้
วิธีแก้: หยุดแอปเดิม แล้วรันใหม่ด้วย F5 (Start Debugging) เท่านั้น — จำง่ายๆ: อยากใช้กับดัก ต้องเข้าเกมผ่านประตู F5

② อ่าน stack trace แล้วเจอแต่ไฟล์แปลกๆ ไม่ใช่โค้ดเรา

สาเหตุ: 90% ของบรรทัดใน stack trace คือทางผ่านของ framework — มือใหม่อ่าน 3 บรรทัดแรกไม่เข้าใจแล้วยอมแพ้
วิธีแก้: สแกนหาคำว่า package:ชื่อแอปเรา/ (เช่น package:todo_app/) บรรทัดแรกที่เจอ = ที่เกิดเหตุ — ใน VS Code บรรทัดพวกนี้เป็นลิงก์ คลิกเด้งไปถึงที่

③ แก้แล้ว "เหมือนหาย" แต่ไม่รู้เพราะอะไร

สาเหตุ: ข้ามขั้นตั้งสมมติฐาน ไปสุ่มแก้จนอาการหาย — บั๊กตัวจริงยังอยู่ แค่หลบฉาก
วิธีแก้: บังคับตัวเองตอบ 2 คำถามก่อนปิดคดีเสมอ: "สาเหตุจริงคืออะไร?" และ "ทำไมการแก้นี้ถึงตรงสาเหตุ?" — ตอบไม่ได้ = คดียังไม่ปิด (ย้อนดูบทที่ 3: แก้จุดเดียวต่อรอบ คือวินัยเดียวกันนี้)

8. 🏋️ แบบฝึกหัดท้ายบท (ลงมือทำจริง)

🥉 ระดับ 1 — หยุดเวลาครั้งแรก

ทำตามหัวข้อ 2 ครบทุกขั้น: F5 → breakpoint ใน _toggleTask → ติ๊กงาน → ดู task.isDone พลิกค่าใน VARIABLES ด้วย Step Over — แล้วลองวาง breakpoint ใน build() ดูว่าถูกเรียกบ่อยแค่ไหน (คำตอบจะทำให้เข้าใจ rebuild จากบทที่ 10 ลึกขึ้นมาก)

🥈 ระดับ 2 — คดีฝังไว้ 3 คดี

โค้ดข้างล่างมีบั๊ก 3 ตัว (ตั้งใจฝัง!) — คัดลอกไปวางในโปรเจกต์ทดลอง แล้วไล่ล่าด้วยขั้นตอน ①②③④ ให้ครบทุกคดี ห้ามเดา ต้องพิสูจน์ด้วย breakpoint หรือ print ก่อนแก้:

// 🐛 มีบั๊ก 3 ตัวซ่อนอยู่ — ล่าให้เจอ!
final List<String> items = ['งาน A', 'งาน B', 'งาน C'];

void removeLast() {
  items.removeAt(items.length);          // คดีที่ 1: กดแล้วเด้งทันที ทำไม?
}

void addItem(String name) {
  items.add(name);                       // คดีที่ 2: เพิ่มแล้วจอไม่เปลี่ยน ทำไม?
}                                        //           (สมมติว่านี่อยู่ใน State class)

int countLong() {
  int count = 0;
  for (int i = 1; i < items.length; i++) {   // คดีที่ 3: นับขาดไป 1 เสมอ ทำไม?
    if (items[i].length > 4) count++;
  }
  return count;
}
🥇 ระดับ 3 — ล่าโดยไม่ใช้ print เลย

กลับไปทำแบบฝึกหัดระดับ 3 ของบทที่ 13 (แยกโซนเสร็จ/ค้าง) ถ้ายังไม่ได้ทำ — คราวนี้ห้ามใช้ print/debugPrint ตลอดงาน ทุกข้อสงสัยต้องตอบด้วย breakpoint + Variables + Step เท่านั้น จบแล้วจดสมุดบั๊ก: เจอบั๊กกี่ตัว สาเหตุอะไร รู้ได้ยังไง — commit ด้วย docs: add bug journal (สร้างไฟล์ BUGS.md ใน repo)

  • ท่องขั้นตอน 4 ขั้นได้: Reproduce → Hypothesize → Verify → Fix
  • อธิบายได้ว่าทำไม "สุ่มแก้จนหาย" อันตรายกว่าบั๊กเอง
  • ปิดคดีฝังไว้ครบ 3 คดี พร้อมบอกสาเหตุจริงของแต่ละคดีได้

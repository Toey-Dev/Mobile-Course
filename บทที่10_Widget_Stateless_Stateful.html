1. 🖼️ ภาพเล่าเรื่อง: ป้ายหน้าร้านสองแบบ

กลับมาที่ร้านอาหารของเราจากบทที่ 2 — คราวนี้ถึงเวลาติด "ป้ายหน้าร้าน" มีให้เลือกสองแบบ:

🪧 ป้ายทาสี (StatelessWidget) 💡 ป้าย LED (StatefulWidget) ┌─────────────────────┐ ┌─────────────────────┐ │ "ร้านป้าหนู" │ │ คิวตอนนี้: [ 42 ] │ │ เปิด 9:00-18:00 │ │ ▲ ตัวเลขเปลี่ยนได้ │ └─────────────────────┘ └─────────────────────┘ ทาสีครั้งเดียว ไม่เปลี่ยนอีกเลย ข้างในมี "หน่วยความจำ" (state) อยากแก้ = รื้อทำใหม่ทั้งป้าย เก็บเลขคิวปัจจุบันไว้ กดปุ่ม → เลขในหน่วยความจำเปลี่ยน เหมาะกับ: ชื่อร้าน โลโก้ เมนูถาวร → ป้ายวาดตัวเองใหม่ทันที เหมาะกับ: อะไรก็ตามที่ "เปลี่ยนได้"

สมการที่สำคัญที่สุดใน Flutter — จำภาพนี้ภาพเดียวพอ

state (ข้อมูล) UI (หน้าจอ) ┌──────────────┐ Flutter ┌──────────────┐ │ queueNumber │ วาดจาก │ คิวตอนนี้: 42 │ │ = 42 │ ─────────► │ │ └──────────────┘ ข้อมูลเสมอ └──────────────┘ ▲ │ │ setState(() { ... }) │ ผู้ใช้กดปุ่ม └──────────────────────────────┘ UI = f(state): "หน้าจอคือผลลัพธ์จากการเอาข้อมูลไปวาด" อยากให้จอเปลี่ยน → อย่าไปแก้จอ ให้แก้ "ข้อมูล" แล้วบอก Flutter ว่าข้อมูลเปลี่ยน
🤔 นี่คือจุดที่ Flutter ต่างจากการคิดแบบสามัญสำนึก

มือใหม่มักคิดว่า "จะไปสั่งให้ข้อความบนจอเปลี่ยน" — ผิดครับ ใน Flutter เราแก้ ข้อมูล (state) แล้วประกาศผ่าน setState ว่า "ข้อมูลเปลี่ยนแล้วนะ" Flutter จะเรียก build() วาดหน้าจอใหม่จากข้อมูลล่าสุดให้เอง (เรียกว่า rebuild) — ทิศทางเดียวเสมอ: ข้อมูล → หน้าจอ

2. 💻 ลงมือจริง: เริ่มโปรเจกต์ To-Do + ปุ่มแรกที่กดแล้วจอเปลี่ยน

สร้างโปรเจกต์ใหม่ (โปรเจกต์ต่อเนื่องของเฟส 3 — จะพัฒนาต่อทุกบทจนจบเฟส):

flutter create todo_app
cd todo_app

แทนที่ lib/main.dart ทั้งหมดด้วยโค้ดนี้:

import 'package:flutter/material.dart';

void main() {
  runApp(const TodoApp());
}

/// ตัวแอป — Stateless เพราะการตั้งค่าธีม/ชื่อแอปไม่เปลี่ยน (ป้ายทาสี)
class TodoApp extends StatelessWidget {
  const TodoApp({super.key});

  @override
  Widget build(BuildContext context) {
    return MaterialApp(
      title: 'To-Do List',
      debugShowCheckedModeBanner: false,
      theme: ThemeData(colorSchemeSeed: Colors.teal, useMaterial3: true),
      home: const HomeScreen(),
    );
  }
}

/// หน้าจอหลัก — Stateful เพราะจำนวนงานจะ "เปลี่ยนได้" (ป้าย LED)
/// สังเกต: StatefulWidget ต้องเขียน 2 คลาสคู่กันเสมอ
class HomeScreen extends StatefulWidget {
  const HomeScreen({super.key});

  @override
  State<HomeScreen> createState() => _HomeScreenState();
  // คลาสที่ 1 (ตัว Widget): เบา สร้างใหม่ได้บ่อยๆ
  // คลาสที่ 2 (ตัว State): "หน่วยความจำ" ที่อยู่ทนข้าม rebuild
}

class _HomeScreenState extends State<HomeScreen> {
  // ⭐ นี่คือ state — ข้อมูลที่หน้าจอวาดจากมัน (ความรู้ List จากบทที่ 7!)
  final List<String> _tasks = ['อ่านบทเรียนที่ 10'];

  void _addTask() {
    // ⭐ setState = "ประกาศว่าข้อมูลเปลี่ยน จอช่วยวาดใหม่ที"
    setState(() {
      _tasks.add('งานใหม่ชิ้นที่ ${_tasks.length + 1}');
    });
  }

  @override
  Widget build(BuildContext context) {
    // build ถูกเรียกใหม่ทุกครั้งที่ setState — หน้าจอจึงตรงกับข้อมูลล่าสุดเสมอ
    return Scaffold(
      appBar: AppBar(title: const Text('To-Do List')),
      body: Center(
        child: Column(
          mainAxisAlignment: MainAxisAlignment.center,
          children: [
            const Icon(Icons.checklist, size: 72, color: Colors.teal),
            const SizedBox(height: 12),
            Text(
              'มีงานทั้งหมด ${_tasks.length} รายการ',
              style: const TextStyle(fontSize: 22, fontWeight: FontWeight.bold),
            ),
            Text('ล่าสุด: ${_tasks.last}'),
          ],
        ),
      ),
      // ปุ่มลอยมุมขวาล่าง — ของแถมจาก Scaffold
      floatingActionButton: FloatingActionButton(
        onPressed: _addTask,   // กดปุ่ม → เรียกฟังก์ชันของเรา
        child: const Icon(Icons.add),
      ),
    );
  }
}

รัน flutter run แล้วกดปุ่ม + ดูตัวเลขเปลี่ยน — นี่คือแอปแรกของคุณที่มีชีวิต! ลองคอมเมนต์ setState(...) ออกเหลือแค่ _tasks.add(...) แล้วกดปุ่มดู: ข้อมูลเปลี่ยนแต่จอนิ่ง — นี่แหละคือเหตุผลว่าทำไมต้องมี setState

3. 🧠 คอนเซปต์: เลือก Stateless หรือ Stateful ยังไง

ถามคำถามเดียว: "หน้าจอส่วนนี้ต้องเปลี่ยนตามการใช้งานไหม?"

ส่วนของแอปเปลี่ยนได้ไหมใช้อะไร
โลโก้, ชื่อแอป, หน้า About❌ นิ่งตลอดStatelessWidget
ตัวนับ, ฟอร์ม, รายการที่เพิ่ม/ลบได้✅ เปลี่ยนตามผู้ใช้StatefulWidget
การ์ดแสดงข้อมูล 1 ชิ้น (รับข้อมูลมาโชว์เฉยๆ)❌ ตัวมันไม่เปลี่ยนเองStatelessWidget (ให้แม่ส่งข้อมูลมา)
🧠 กฎมืออาชีพ: เริ่มจาก Stateless เสมอ

เปลี่ยนเป็น Stateful เฉพาะเมื่อจำเป็นจริง เหมือน final กับ var ในบทที่ 4: ของที่ไม่เปลี่ยนอ่านง่ายกว่า เร็วกว่า และผิดยากกว่า

4. 🔧 เทคนิคการประยุกต์ใช้

เทคนิคที่ 1 — พิมพ์ stful + Tab ใน VS Code

โครง StatefulWidget 2 คลาสไม่ต้องจำ — VS Code มี snippet ให้: พิมพ์ stful กด Tab แล้วพิมพ์ชื่อคลาส โครงขึ้นครบ (stless สำหรับ Stateless) — dev จริงใช้ snippet กันทั้งนั้น

เทคนิคที่ 2 — เก็บ state ให้เล็กที่สุด อย่าเก็บของที่คำนวณได้

เก็บ _tasks (List) พอ — จำนวนงานคือ _tasks.length คำนวณสดใน build ได้เลย ถ้าเก็บ _taskCount แยกอีกตัว = มีข้อมูล 2 ชุดที่ต้องคอยให้ตรงกัน วันไหนลืมอัปเดตตัวหนึ่ง จอก็โกหกผู้ใช้ทันที

เทคนิคที่ 3 — ใน setState ทำแค่ "แก้ข้อมูล"

งานหนัก (โหลดข้อมูล คำนวณยาว) ทำนอก setState แล้วค่อย setState สั้นๆ ตอนจบ — setState มีหน้าที่เดียวคือประกาศว่า "เปลี่ยนแล้ว" ไม่ใช่ที่ทำงาน

เทคนิคที่ 4 — ตัวแปร state ขึ้นต้นด้วย _ เสมอ

_tasks, _addTask — เครื่องหมาย _ ใน Dart แปลว่า "ของส่วนตัวในไฟล์นี้" คนนอกมาแก้ state ของเราตรงๆ ไม่ได้ ต้องผ่าน method ที่เราเปิดให้ — วินัยข้อนี้จะสำคัญมากตอนแอปโต

5. 🏭 ตัวอย่างการเขียนจริงในงานจริง

🏭 สถานการณ์จริง: "ปุ่มหัวใจใน Instagram"

กดหัวใจ → เปลี่ยนเป็นสีแดงทันที + ตัวเลขไลก์บวกหนึ่ง — เบื้องหลังคือ pattern เดียวกับที่คุณเพิ่งเขียนเป๊ะ

class LikeButton extends StatefulWidget { /* ... */ }

class _LikeButtonState extends State<LikeButton> {
  bool _isLiked = false;   // state จิ๋วๆ ตัวเดียว (bool จากบทที่ 4)
  int _likeCount = 1024;

  void _toggleLike() {
    setState(() {
      _isLiked = !_isLiked;                     // สลับ จริง↔เท็จ
      _likeCount += _isLiked ? 1 : -1;
    });
  }

  @override
  Widget build(BuildContext context) {
    return IconButton(
      icon: Icon(
        _isLiked ? Icons.favorite : Icons.favorite_border,  // UI วาดจาก state
        color: _isLiked ? Colors.red : Colors.grey,
      ),
      onPressed: _toggleLike,
    );
  }
}

ในทีมจริง หน้าจอหนึ่งหน้าประกอบจาก Widget เล็กๆ แบบนี้หลายสิบตัว แต่ละตัวถือ state เฉพาะของตัวเอง — ปุ่มไลก์รู้แค่เรื่องไลก์ ไม่ต้องรู้เรื่องคอมเมนต์ นี่คือ Single Responsibility ระดับ UI

6. 🧹 ตัวอย่าง Clean Code

// ❌ state ซ้ำซ้อน ชื่อกำกวม ไม่มี _
class _HomeScreenState extends State<HomeScreen> {
  List<String> tasks = [];   // ไม่มี _ = คนนอกแก้ได้ อันตราย
  int taskCount = 0;         // ซ้ำกับ tasks.length — รอวันไม่ตรงกัน
  String title = 'To-Do';    // ค่านิ่งๆ ไม่ควรเป็น state เลย

  void add() {               // ชื่อกำกวม: add อะไร?
    tasks.add('งาน');
    // ลืมอัปเดต taskCount แล้ว... จอแสดงเลขผิดทันที
    setState(() {});
  }
}
// ✅ state น้อยที่สุด ชื่อชัด ของนิ่งเป็น const
class _HomeScreenState extends State<HomeScreen> {
  // state มีตัวเดียวจริงๆ: รายการงาน — ทุกอย่างที่เหลือคำนวณจากมัน
  final List<String> _tasks = [];

  void _addTask() {
    setState(() => _tasks.add('งานใหม่ชิ้นที่ ${_tasks.length + 1}'));
  }

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      appBar: AppBar(title: const Text('To-Do List')), // ค่านิ่ง → const ไปเลย
      body: Text('ทั้งหมด ${_tasks.length} งาน'),        // คำนวณสด ไม่มีวันเพี้ยน
      floatingActionButton: FloatingActionButton(
        onPressed: _addTask,
        child: const Icon(Icons.add),
      ),
    );
  }
}
🧠 หลักทอง

state ทุกตัวคือภาระที่ต้องดูแลให้ถูกต้องตลอดชีวิตของหน้าจอ — ยิ่งถือน้อย ยิ่งผิดยาก ถามตัวเองก่อนเพิ่ม state ทุกครั้ง: "ตัวนี้คำนวณจากตัวอื่นได้ไหม?"

7. ⚠️ ข้อผิดพลาดที่เจอบ่อย

① แก้ข้อมูลแล้วแต่จอไม่เปลี่ยน

สาเหตุ: แก้ตัวแปร state ตรงๆ โดยไม่ผ่าน setState — Flutter ไม่รู้ว่าต้องวาดใหม่
วิธีแก้: ครอบการแก้ด้วย setState(() { ... }); เสมอ — นี่คือบั๊กอันดับ 1 ของบทนี้ ถ้าจอนิ่งผิดปกติ หา setState ที่หายไปก่อนเลย

② งงว่าทำไม StatefulWidget ต้องมี 2 คลาส / ขีดล่างคืออะไร

วิธีเข้าใจ: คลาสแรก (HomeScreen) = "ป้ายชื่อ" เบาๆ สร้างทิ้งได้บ่อย · คลาสสอง (_HomeScreenState) = "หน่วยความจำ" ที่อยู่ทน ถือ state จริง · _ นำหน้า = ของส่วนตัว ไฟล์อื่นมองไม่เห็น (ตั้งใจ ไม่ใช่ความบังเอิญ) · ใช้ snippet stful สร้างเสมอ เดี๋ยวก็ชิน

③ "setState() called after dispose()"

สาเหตุ: สั่งวาดใหม่หลังหน้าจอถูกปิดไปแล้ว (มักเจอคู่กับงาน async — รอโหลดเสร็จแล้วค่อย setState แต่ผู้ใช้กดออกจากหน้าไปก่อน)
วิธีแก้: ตอนนี้แค่รู้จักชื่อมันพอ — จะเจอจริงและแก้จริงตอนบทที่ 19-20 (ถ้าเจอตอนนี้: เช็ค if (mounted) ก่อน setState)

8. 🏋️ แบบฝึกหัดท้ายบท (ลงมือทำจริง)

🥉 ระดับ 1 — ป้าย LED แรก

ทำแอปตัวนับตามหัวข้อ 2 ให้รันได้ แล้วทดลอง 2 อย่าง: (1) คอมเมนต์ setState ออกแล้วกดปุ่ม — อธิบายให้ตัวเองฟังว่าทำไมจอนิ่ง (2) ใส่ debugPrint('build ทำงาน!') ใน build แล้วดู Debug Console ว่า build ถูกเรียกใหม่ทุกครั้งที่กดจริงไหม (เครื่องมือจากบทที่ 3)

🥈 ระดับ 2 — ปุ่มหัวใจของตัวเอง

สร้าง Widget LikeButton แยกคลาสตามตัวอย่างหัวข้อ 5 ให้กดสลับหัวใจแดง/เทา + ตัวเลขขึ้นลงได้จริง แล้วเอาไปวางใน HomeScreen — ฝึกการมี Stateful Widget ตัวเล็กซ้อนในหน้าใหญ่

🥇 ระดับ 3 — โครง To-Do จริงจัง (ชิ้นส่วนแรกของโปรเจกต์เฟส!)

ต่อยอดแอปหัวข้อ 2: (1) แสดงรายการงานทุกชิ้นด้วย Column + .map() จากบทที่ 7: children: _tasks.map((t) => Text('• $t')).toList() (2) เพิ่มปุ่มลบงานล่าสุด (_tasks.removeLast() — อย่าลืมกัน List ว่างด้วย guard clause บทที่ 5!) (3) ถ้า List ว่าง แสดง "ยังไม่มีงาน 🎉" แทน — git init, commit ด้วย feat: todo skeleton with add/remove and empty state แล้ว push ขึ้น GitHub

9. ✅ เช็กลิสต์และสรุปบทเรียน

  • อธิบาย UI = f(state) ด้วยคำพูดตัวเองได้ (ทำไมเราแก้ข้อมูล ไม่แก้จอ)
  • บอกได้ว่าเมื่อไหร่ใช้ Stateless เมื่อไหร่ใช้ Stateful และทำไมควรเริ่มจาก Stateless
  • เขียน StatefulWidget + setState ได้ (ใช้ snippet stful เป็น)
  • เคยเห็นกับตาแล้วว่า "แก้ข้อมูลโดยไม่ setState = จอนิ่ง"
  • โครงแอป To-Do (เพิ่ม/ลบ/empty state) อยู่บน GitHub แล้ว

1. 🖼️ ภาพเล่าเรื่อง: บัตรคิวร้านกาแฟ

ลองนึกภาพร้านกาแฟสองแบบ:

☕ ร้านแบบ "รอหน้าเคาน์เตอร์" (synchronous) ☕ ร้านแบบ "บัตรคิว" (asynchronous) ลูกค้า: สั่งลาเต้ ลูกค้า: สั่งลาเต้ บาริสต้า: รอตรงนี้นะ 3 นาที บาริสต้า: รับบัตรคิวไปก่อนครับ 🎫 ลูกค้า: ยืนเฉยๆ บล็อกแถว... ลูกค้า: ไปนั่งเล่นมือถือ หาที่นั่ง คนข้างหลัง: 😡 แถวไม่ขยับเลย (ร้านยังรับออเดอร์คนถัดไปได้ตลอด) เสียงเรียก: "คิว 42 ลาเต้ได้แล้วครับ!" ลูกค้า: เดินไปรับ ☕

ในโลกโปรแกรม: งานที่ต้องรอ ถ้ารอแบบยืนบล็อกแถว = แอปค้างทั้งแอป ผู้ใช้กดอะไรไม่ได้เลย — Dart จึงใช้ระบบบัตรคิว:

สั่งงานที่ต้องรอ ──────► ได้ "บัตรคิว" มาทันที (นี่แหละคือ Future) │ โปรแกรมทำอย่างอื่นต่อได้ │ งานเสร็จเมื่อไหร่ ▼ ค่าจริงมาส่งตามบัตรคิว
💡 Future = บัตรคิว ไม่ใช่กาแฟ

Future<String> แปลว่า "บัตรคิวที่สัญญาว่าเดี๋ยวจะได้ String" — ผิดพลาดคลาสสิกของมือใหม่คือเอาบัตรคิวไปดื่มแทนกาแฟ (ใช้ Future ราวกับเป็นค่าจริง) เดี๋ยวเจอในหัวข้อข้อผิดพลาด

2. 💻 ลงมือจริง: จำลองการโหลดคำถามจาก "เซิร์ฟเวอร์ปลอม"

เกม Quiz จริงๆ ควรโหลดคำถามจากอินเทอร์เน็ต — ตอนนี้ยังไม่ถึงบทนั้น เราจะ "จำลองการรอ" ด้วย Future.delayed แทน:

// จำลองเซิร์ฟเวอร์: ขอคำถามแล้วต้องรอ 2 วินาที (เหมือนรอโหลดจากเน็ตจริง)
Future<List<String>> fetchQuestions() {
  return Future.delayed(
    const Duration(seconds: 2),                  // แกล้งช้า 2 วินาที
    () => ['Flutter ใช้ภาษาอะไร?', 'ไฟล์ package ชื่ออะไร?'],
  );
}

// ฟังก์ชันที่ "มีการรอ" ข้างใน ต้องติดป้าย async ที่หลังวงเล็บ
Future<void> main() async {
  print('เปิดเกม... กำลังโหลดคำถาม ⏳');

  // await = "รอตรงนี้จนบัตรคิวถูกเรียก แล้วค่อยไปบรรทัดถัดไป"
  final questions = await fetchQuestions();

  print('โหลดเสร็จ! ได้ ${questions.length} ข้อ:');
  for (final q in questions) {
    print('- $q');
  }
}

รันแล้วสังเกต: ข้อความแรกขึ้นทันที → เงียบ 2 วินาที → ผลลัพธ์มา — คุณเพิ่งเขียนโปรแกรมที่ "รอเป็น" ครั้งแรก

กฎ 3 ข้อที่ครอบคลุม 90% ของการใช้งาน

  1. งานที่ต้องรอ คืนค่าเป็น Future<ชนิดข้อมูล> — บัตรคิวที่ระบุว่ารออะไรอยู่
  2. อยากได้ค่าจริงจากบัตรคิว → await หน้ามัน
  3. ฟังก์ชันไหนมี await ข้างใน → ต้องติด async หลังวงเล็บพารามิเตอร์
🤔 ทำไมบทนี้สอนแค่นี้

เพราะ async จะ "เข้าใจจริง" ก็ต่อเมื่อมีของที่ต้องรอจริงๆ — ตอนนี้ให้จำหน้าตาโค้ดได้พอ พอถึงบทที่ 19 (เรียก API สภาพอากาศจริงจากอินเทอร์เน็ต) เราจะกลับมาเรียนซ้ำเชิงลึกพร้อมเหตุผลครบๆ การเจอสองรอบแบบนี้คือวิธีที่สมองจำได้ดีที่สุด

3. 🔧 เทคนิคการประยุกต์ใช้

เทคนิคที่ 1 — อ่านลายเซ็นฟังก์ชันก็รู้ว่าต้องรอไหม

เปิดเอกสารเจอ Future<String> loadName() → รู้ทันทีว่า "ฟังก์ชันนี้ต้องรอ และจะได้ String" — ชนิดข้อมูลเล่าเรื่องให้ฟังก่อนอ่านโค้ดสักบรรทัด

เทคนิคที่ 2 — ตั้งชื่อฟังก์ชัน async ด้วยกริยาที่สื่อการรอ

fetchQuestions, loadProfile, saveScore — วงการนิยมใช้ fetch/load/save/send กับงาน async พอเห็นชื่อก็เดาได้ว่ามี Future เกี่ยวข้อง

เทคนิคที่ 3 — print ก่อน-หลัง await คือเครื่องมือเรียนรู้ที่ดีที่สุด

สงสัยว่าโค้ดวิ่งลำดับไหน? แปะ print('ก่อนรอ') / print('หลังรอ') รอบ await แล้วดูจังหวะที่ข้อความขึ้น — ทำแบบฝึกหัดระดับ 1 แล้วคุณจะ "เห็น" การรอด้วยตาตัวเอง

เทคนิคที่ 4 — Future.delayed คือเพื่อนรักตอน dev

แอปจริงใช้จำลอง "เน็ตช้า" เพื่อทดสอบว่าหน้าจอ loading สวยไหม ผู้ใช้เน็ตอ่อนเห็นอะไร — เทคนิคนี้ทีมจริงใช้กันทุกวัน และคุณจะใช้มันตอนทำ Loading State ในบทที่ 20

4. 🏭 ตัวอย่างการเขียนจริงในงานจริง

🏭 สถานการณ์จริง: "หน้าฟีดที่ต้องโหลด 3 อย่างพร้อมกัน"

เปิดแอปโซเชียลหนึ่งครั้ง เบื้องหลังคือ async หลายงานพร้อมกัน: โหลดโปรไฟล์ + โหลดฟีด + โหลดแจ้งเตือน — ถ้าทำทีละอย่างต่อคิวกัน (รอ 1 วิ × 3 งาน = 3 วิ) ผู้ใช้หนีไปแล้ว ทีมจริงจึงยิงพร้อมกัน

// รอทั้ง 3 บัตรคิวพร้อมกัน — เสร็จเมื่องานที่ช้าสุดเสร็จ (1 วิ ไม่ใช่ 3 วิ)
final results = await Future.wait([
  fetchProfile(),
  fetchFeed(),
  fetchNotifications(),
]);

ทุกวินาทีที่ผู้ใช้ต้องรอ = โอกาสที่เขาจะปิดแอป — บริษัทใหญ่วัดกันจริงจังว่าหน้าแรกโหลดกี่มิลลิวินาที และเครื่องมือของสงครามความเร็วนี้คือ async ที่คุณเพิ่งรู้จักวันนี้ (Future.wait แค่โชว์ให้เห็นภาพ ยังไม่ต้องใช้เป็นตอนนี้)

5. 🧹 ตัวอย่าง Clean Code

// ❌ ลืม await — บั๊กเงียบที่เจอบ่อยที่สุดในโลก async
void main() {
  final questions = fetchQuestions();   // ลืม await!
  print(questions);                     // Instance of 'Future<List<String>>'
  // ได้ "บัตรคิว" มา print แทนที่จะได้คำถาม — ไม่ crash แต่ผลลัพธ์ผิดเงียบๆ
}
// ✅ await ครบ + ตั้งชื่อสื่อความหมาย + บอกผู้ใช้ว่ากำลังรอ
Future<void> main() async {
  print('กำลังโหลดคำถาม ⏳');            // บอกเสมอว่ากำลังเกิดอะไร (นิสัย UX ที่ดี
  final questions = await fetchQuestions(); // เริ่มปลูกได้ตั้งแต่บน Terminal!)
  print('พร้อมเล่น! (${questions.length} ข้อ)');
}
💡 ยามที่ตั้งไว้ตั้งแต่บทที่ 2

จำกฎ lint unawaited_futures ใน analysis_options.yaml ของเราได้ไหมครับ — มันคือยามที่คอยเตือนตอน "ลืม await" นี่แหละ ตั้งไว้ตั้งแต่วันแรกเพราะรู้ว่าวันนี้จะมาถึง

6. ⚠️ ข้อผิดพลาดที่เจอบ่อย

① print แล้วได้ "Instance of 'Future<...>'"

สาเหตุ: ลืม await — ได้บัตรคิวมาใช้แทนค่าจริง
วิธีแก้: เติม await หน้าการเรียกฟังก์ชันที่คืน Future และเช็คว่าฟังก์ชันเราติด async แล้ว — นี่คือ error ที่เจอบ่อยที่สุดของ async ทั่วโลก เจอเมื่อไหร่ หา await ที่หายไป

② "The await expression can only be used in an async function"

สาเหตุ: ใช้ await ในฟังก์ชันที่ไม่ได้ติดป้าย async
วิธีแก้: เติม async หลังวงเล็บ: Future<void> main() async { ... } — จำเป็นคู่กัน: มี await ที่ไหน ต้องมี async ที่หัวฟังก์ชันนั้น

③ โปรแกรมจบก่อนงานเสร็จ

สาเหตุ: main() ไม่ await งานสุดท้าย — โปรแกรมวิ่งถึงบรรทัดจบแล้วปิดตัว ทั้งที่บัตรคิวยังไม่ถูกเรียก
วิธีแก้: await ทุกงานที่อยากเห็นผลลัพธ์ก่อน main จบ

7. 🏋️ แบบฝึกหัดท้ายบท (ลงมือทำจริง)

🥉 ระดับ 1 — เห็นการรอด้วยตา

เขียนฟังก์ชัน Future<String> brewCoffee() ที่รอ 3 วินาทีแล้วคืน "ลาเต้ร้อน ☕" — ใน main ให้ print "สั่งกาแฟ", "รอ...", await, แล้ว print ผล จับเวลาดูว่าตรง 3 วินาทีจริงไหม แล้วลองลบ await ออกดูว่าเกิดอะไรขึ้น (สำคัญมาก — เจอบั๊กนี้ตอนฝึก ดีกว่าเจอตอนทำงาน)

🥈 ระดับ 2 — ลำดับการรอ

เขียน 3 ฟังก์ชัน async: washBeans() (1 วิ), grindBeans() (2 วิ), brew() (1 วิ) แต่ละตัว print ตอนเสร็จ — เรียกตามลำดับด้วย await ทั้งสามตัว แล้วสังเกตว่าเวลารวมคือ 4 วินาที (งานต่อคิวกัน) — โบนัส: ลองครอบด้วย Future.wait([...]) แล้วดูว่าเวลารวมเปลี่ยนเป็นเท่าไหร่ เพราะอะไร

🥇 ระดับ 3 — Quiz ฉบับสมบูรณ์ (ผลงานจบเฟส 2!)

รวมทุกบทเข้าด้วยกัน: Quiz แบบ OOP จากบทที่ 8 + เพิ่ม Future<List<Question>> loadQuestions() ที่จำลองโหลด 2 วินาทีก่อนเกมเริ่ม (แสดง "กำลังโหลด...") — จบเกมแสดงคะแนน + เกรด — commit, push, แล้วเขียน README สั้นๆ ให้ repo quiz_cli ตามแบบ README ของ my_first_app

8. ✅ เช็กลิสต์และสรุปบทเรียน

  • อธิบาย Future ด้วยคำเปรียบ "บัตรคิว" ให้เพื่อนฟังได้
  • ท่องกฎ 3 ข้อได้: คืน Future<T> / อยากได้ค่า → await / มี await → ติด async

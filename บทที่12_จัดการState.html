1. 🖼️ ภาพเล่าเรื่อง: กระดานคะแนนสนามกีฬา

นึกภาพสนามฟุตบอลที่มีจอทีวีย่อยกระจายอยู่ 20 จุดทั่วสนาม — คะแนนบนทุกจอต้องตรงกันเสมอ ทำยังไง?

❌ แบบที่พัง: ทุกจอจดคะแนนเอง ✅ แบบที่ใช้จริง: กระดานกลางอันเดียว จอ A: 2-1 จอ B: 2-2 🏟️ กระดานกลาง: 2-1 จอ C: 1-1 จอ D: 2-1 (single source of truth) 😱 ใครถูก?? คนดูตีกัน ┌──────┬──────┬──────┬──────┐ (แต่ละจอมีข้อมูล "สำเนา" ของตัวเอง จอ A จอ B จอ C จอ D พอมีประตูใหม่ อัปเดตไม่ครบทุกจอ) ทุกจอ "อ่านจาก" กระดานกลางเสมอ กระดานเปลี่ยน → ทุกจอตรงกันอัตโนมัติ

แล้วถ้ากรรมการอยู่ขอบสนามอยากแก้คะแนนล่ะ? — ฉากที่สอง: ส่งข่าวหาเจ้าของกระดาน

กรรมการ (Widget ลูก) เจ้าของกระดาน (Widget แม่ที่ถือ state) เห็นเหตุการณ์ประตูเข้า! │ │ │ │ 📣 "มีประตูที่นาทีที่ 88!" │ └────────── callback ────────────► แก้กระดาน (setState) │ กรรมการไม่แก้กระดานเอง ▼ แค่ "ส่งข่าว" ขึ้นไป ทุกจอวาดใหม่ตรงกันหมด
🤔 นี่คือ pattern ชื่อ "lifting state up" (ยก state ขึ้นไปไว้ที่แม่)

ข้อมูลอยู่ที่ Widget แม่ตัวเดียว → ลูกรับข้อมูลมา "แสดง" และรับ callback มา "ส่งข่าว" — ลูกไม่มีสำเนาข้อมูลของตัวเอง จึงไม่มีทางที่จอจะไม่ตรงกัน คุณจะใช้ pattern นี้ไปตลอดชีวิต Flutter (Provider ในเฟส 4 ก็คือแนวคิดนี้เวอร์ชันขยาย)

2. 💻 ลงมือจริง: Task Model + ติ๊กเสร็จได้จริง

ขั้นที่ 1 — สร้างพิมพ์เขียว Task (ความรู้บทที่ 8 + โครงสร้างโฟลเดอร์บทที่ 2)

สร้างไฟล์ใหม่ lib/models/task.dart:

/// พิมพ์เขียวของงาน 1 ชิ้น — จาก List<String> ธรรมดา อัปเกรดเป็น model จริง
/// เพราะงานหนึ่งชิ้นมีข้อมูลมากกว่าชื่อ: ต้องรู้ด้วยว่า "เสร็จหรือยัง"
class Task {
  Task({required this.title, this.isDone = false});

  final String title;
  bool isDone;          // ตั้งใจไม่ใส่ final — สถานะเสร็จ/ไม่เสร็จต้องเปลี่ยนได้

  void toggle() => isDone = !isDone;   // ความรู้เรื่องงาน อยู่กับตัวงาน (บทที่ 8)
}

ขั้นที่ 2 — เปลี่ยน state เป็น List<Task> และเพิ่ม Checkbox

import 'package:flutter/material.dart';
import 'models/task.dart';                 // ดึงพิมพ์เขียวจากบ้านของมัน

class _HomeScreenState extends State<HomeScreen> {
  // อัปเกรดจาก List<String> → List<Task> (กระดานกลางของทั้งหน้า)
  final List<Task> _tasks = [
    Task(title: 'อ่านบทเรียนที่ 12'),
    Task(title: 'ทำแบบฝึกหัด 3 ระดับ'),
  ];

  void _toggleTask(Task task) {
    setState(() => task.toggle());        // แก้กระดาน + ประกาศให้วาดใหม่
  }

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      appBar: AppBar(title: const Text('To-Do List')),
      body: Column(
        children: _tasks.map((task) {
          return CheckboxListTile(
            value: task.isDone,                       // จออ่านจากกระดานกลาง
            onChanged: (_) => _toggleTask(task),      // ติ๊ก = ส่งข่าวหาเจ้าของ
            title: Text(
              task.title,
              style: TextStyle(
                // เสร็จแล้ว = ขีดฆ่า + สีจาง (UI วาดจาก state เสมอ!)
                decoration: task.isDone ? TextDecoration.lineThrough : null,
                color: task.isDone ? Colors.grey : null,
              ),
            ),
          );
        }).toList(),
      ),
    );
  }
}

รันแล้วติ๊กดู: เครื่องหมายถูกขึ้น + ข้อความถูกขีดฆ่า — สังเกตว่าทุกอย่างไหลตามสมการ UI = f(state): เราแก้ task.isDone แล้วหน้าตา (ขีดฆ่า, สีจาง, เครื่องหมายถูก) เปลี่ยนตามเองทั้งหมด

ขั้นที่ 3 — แยกลูกออกไป แล้วคุยกันผ่าน callback

/// แถวงาน 1 แถว — Stateless! ไม่ถือข้อมูลเอง รับทุกอย่างมาจากแม่
class TaskTile extends StatelessWidget {
  const TaskTile({super.key, required this.task, required this.onToggle});

  final Task task;              // ข้อมูล: อ่านจากกระดานกลาง (ส่งผ่านแม่)
  final VoidCallback onToggle;  // ช่องทางส่งข่าว: "มีคนติ๊กฉันนะ!"

  @override
  Widget build(BuildContext context) {
    return CheckboxListTile(
      value: task.isDone,
      onChanged: (_) => onToggle(),   // ไม่แก้ข้อมูลเอง — ส่งข่าวขึ้นไป
      title: Text(task.title),
    );
  }
}

// ฝั่งแม่ ใช้แบบนี้:
TaskTile(task: task, onToggle: () => _toggleTask(task))

3. 🔧 เทคนิคการประยุกต์ใช้

เทคนิคที่ 1 — ตัดสินว่า state อยู่ที่ใคร ด้วยคำถาม "ใครต้องใช้บ้าง?"

ข้อมูลที่ใช้ใน Widget เดียว → เก็บใน Widget นั้น (เช่น _isLiked ของปุ่มหัวใจบทที่ 10) · ข้อมูลที่หลาย Widget ต้องเห็นตรงกัน (รายการงาน — ทั้ง header card และรายการใช้) → ยกขึ้นไปที่แม่ร่วมที่ใกล้ที่สุด

เทคนิคที่ 2 — ตั้งชื่อ callback ว่า onXxx เสมอ

onToggle, onDelete, onSubmit — ตรงกับที่ Flutter ใช้เอง (onPressed, onChanged) คนอ่านเห็นปุ๊บรู้ว่า "นี่คือช่องเสียบเหตุการณ์" ส่วนฝั่งแม่ตั้งชื่อ handler ว่า _handleXxx หรือ _toggleTask ตามงานที่ทำ

เทคนิคที่ 3 — initState คือที่ตั้งค่าเริ่มต้นของ state

อยากให้หน้าเริ่มมาพร้อมข้อมูล (เช่น โหลดงานตัวอย่าง) ให้ทำใน initState() — method พิเศษที่ Flutter เรียก ครั้งเดียว ตอน State ถูกสร้าง (ห้ามทำใน build เพราะ build ถูกเรียกซ้ำตลอด)

เทคนิคที่ 4 — callback ที่อยากส่งข้อมูลกลับ ใช้ ValueChanged<T>

VoidCallback = ส่งข่าวเปล่าๆ ("มีคนกด!") · ValueChanged<String> = ส่งข่าวพร้อมพัสดุ ("มีคนพิมพ์คำนี้!") — บทที่ 16 (Form) จะใช้ตัวหลังหนักมาก รู้จักไว้ก่อน

4. 🏭 ตัวอย่างการเขียนจริงในงานจริง

🏭 สถานการณ์จริง: "ปุ่มบวก-ลบในตะกร้าสินค้า"

หน้าตะกร้าของแอปช้อปปิ้ง: การ์ดสินค้าแต่ละใบมีปุ่ม + / − แต่ ยอดรวมด้านล่าง ต้องเปลี่ยนตามทันที — ถ้าการ์ดแต่ละใบเก็บจำนวนของตัวเอง ยอดรวมจะรู้ได้ยังไง?

CartScreen (แม่ — ถือกระดานกลาง: รายการสินค้า + จำนวน) ├── ProductCard(item, onAdd, onRemove) ← ลูกส่งข่าว ไม่แก้เอง ├── ProductCard(item, onAdd, onRemove) └── TotalBar(total: _calculateTotal()) ← คำนวณสดจากกระดานกลาง กด + ที่การ์ดใบแรก → callback ขึ้นไปหาแม่ → setState แก้กระดาน → การ์ดวาดจำนวนใหม่ + ยอดรวมล่างจอเปลี่ยน พร้อมกัน เป๊ะกันเสมอ

นี่คือเหตุผลที่ทีมจริงเข้มงวดเรื่อง lifting state up: บั๊กประเภท "จอสองจุดแสดงไม่ตรงกัน" เกือบทั้งหมดเกิดจากมีข้อมูลสำเนาซ้ำสองที่ — ออกแบบให้มีกระดานเดียวตั้งแต่แรก = ตัดบั๊กทั้งตระกูลทิ้ง

5. 🧹 ตัวอย่าง Clean Code: สำเนาข้อมูล คือบ่อเกิดของบั๊ก

// ❌ ลูกเก็บ state สำเนาเอง — กระดานสองอัน รอวันไม่ตรงกัน
class _TaskTileState extends State<TaskTile> {
  bool _isDone = false;   // 😱 สำเนาของ task.isDone — ตอนนี้มีความจริง 2 ชุด!

  @override
  Widget build(BuildContext context) {
    return CheckboxListTile(
      value: _isDone,
      onChanged: (_) => setState(() => _isDone = !_isDone),
      // ติ๊กแล้วแถวนี้เปลี่ยน... แต่ header card นับจำนวนเสร็จจาก task.isDone
      // ซึ่งไม่เคยถูกแก้ → ตัวเลขบนจอโกหกผู้ใช้ทันที
      title: Text(widget.task.title),
    );
  }
}
// ✅ ลูก Stateless รับข้อมูล + callback — ความจริงมีชุดเดียว
class TaskTile extends StatelessWidget {
  const TaskTile({super.key, required this.task, required this.onToggle});
  final Task task;              // อ่านจากกระดานกลางเท่านั้น
  final VoidCallback onToggle;  // อยากเปลี่ยน? ส่งข่าวหาเจ้าของกระดาน

  @override
  Widget build(BuildContext context) {
    return CheckboxListTile(
      value: task.isDone,
      onChanged: (_) => onToggle(),
      title: Text(task.title),
    );
  }
}
🧠 หลักทอง

ก่อนพิมพ์ StatefulWidget ให้ถามว่า "ข้อมูลนี้มีเจ้าของอยู่แล้วหรือเปล่า?" — ถ้ามี (อยู่ในกระดานของแม่) ลูกควรเป็น Stateless ที่รับของมาแสดง อย่าสร้างสำเนา

6. ⚠️ ข้อผิดพลาดที่เจอบ่อย

① "setState() or markNeedsBuild() called during build"

สาเหตุ: เผลอเรียกฟังก์ชันทันทีแทนที่จะส่งฟังก์ชันให้ปุ่ม: onPressed: _addTask() ← มีวงเล็บ = เรียกเลยระหว่างวาดจอ!
วิธีแก้: ส่ง "ตัวฟังก์ชัน" ไม่ใช่ "ผลของฟังก์ชัน": onPressed: _addTask (ไม่มีวงเล็บ) หรือห่อ: onPressed: () => _addTask(task) เมื่อต้องส่ง argument

② Checkbox เป็นสีเทา กดไม่ได้

สาเหตุ: onChanged เป็น null — ใน Flutter การส่ง null ให้ onChanged/onPressed มีความหมายพิเศษ: "ปิดการใช้งาน (disabled)"
วิธีแก้: ใส่ฟังก์ชันให้ onChanged เสมอเมื่ออยากให้กดได้ — ความรู้นี้ใช้ทำปุ่ม disabled แบบตั้งใจได้ด้วย (เช่น ปุ่ม submit ที่กดไม่ได้จนกว่าจะกรอกครบ)

③ ติ๊กแล้วเด้งกลับ / ติ๊กแถวหนึ่ง อีกจุดไม่อัปเดต

สาเหตุ: อาการคลาสสิกของ "ความจริงสองชุด" — มีสำเนา state ซ้ำ หรือ setState อยู่ผิดตัว (ลูกเรียก setState ของตัวเอง แต่ข้อมูลจริงอยู่ที่แม่ → จอแม่ไม่วาดใหม่)
วิธีแก้: ไล่หาว่า "กระดานกลางตัวจริงอยู่ไหน" แล้วให้ setState เกิดที่เจ้าของกระดานผ่าน callback — ลูกห้ามถือสำเนา

7. 🏋️ แบบฝึกหัดท้ายบท (ลงมือทำจริง)

🥉 ระดับ 1 — ติ๊กได้จริง

ทำตามหัวข้อ 2 ให้ครบ: สร้าง lib/models/task.dart, เปลี่ยน state เป็น List<Task>, ติ๊กเสร็จแล้วข้อความขีดฆ่า+สีจาง — ทดสอบ: ติ๊กแล้ว hot reload หน้าจอต้องคงสถานะติ๊กไว้ (ทำไมถึงคง? กลับไปดูตาราง Hot Reload บทที่ 3)

🥈 ระดับ 2 — แยกลูก ส่งข่าวผ่าน callback

แยก TaskTile เป็น StatelessWidget ตามขั้นที่ 3 (ไฟล์ใหม่ lib/widgets/task_tile.dart — โครงสร้างโฟลเดอร์จากบทที่ 2) เพิ่มปุ่มถังขยะท้ายแถวพร้อม callback onDelete ให้ลบงานได้ — commit ด้วย refactor: extract TaskTile with callbacks

🥇 ระดับ 3 — กระดานกลางโชว์พลัง

ทำให้ header card จากบทที่ 11 แสดง "เสร็จแล้ว x / ทั้งหมด y" โดยคำนวณสดจาก _tasks.where((t) => t.isDone).length (ห้ามเก็บตัวนับแยก! — เทคนิคบทที่ 10) และเพิ่มปุ่ม "ล้างงานที่เสร็จแล้ว" ที่ใช้ .where() กรองงานค้างเก็บไว้ — ติ๊กงานแล้วดูตัวเลขบนการ์ดเปลี่ยนพร้อมกันทันที นั่นคือ single source of truth ทำงาน — commit ด้วย feat: live done counter and clear-completed button

8. ✅ เช็กลิสต์และสรุปบทเรียน

  • อธิบาย lifting state up ด้วยภาพกระดานคะแนนสนามกีฬาได้
  • ตอบได้ว่า state ก้อนหนึ่งควรอยู่ที่ Widget ไหน ("ใครต้องใช้บ้าง?")
  • เขียน Widget ลูกแบบ Stateless ที่รับข้อมูล + callback ได้
  • อธิบายได้ว่าทำไม onPressed: _fn() (มีวงเล็บ) ถึงพัง
  • To-Do ติ๊กเสร็จได้ ลบได้ และ header นับงานเสร็จแบบคำนวณสด

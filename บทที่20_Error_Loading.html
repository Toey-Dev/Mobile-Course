1. 🖼️ ภาพเล่าเรื่อง: ร้านที่เงียบหาย กับร้านที่สุภาพ

วัตถุดิบหมด ไฟดับ ครัวล่ม — เกิดขึ้นได้กับทุกร้าน สิ่งที่แยกร้านดีออกจากร้านแย่ไม่ใช่ "ไม่มีปัญหา" แต่คือ วิธีบอกลูกค้าเมื่อมีปัญหา:

😶 ร้านที่เงียบหาย (แอปที่ไม่จัดการ error) 🙏 ร้านที่สุภาพ (แอปมืออาชีพ) ลูกค้า: สั่งข้าวผัดครับ ลูกค้า: สั่งข้าวผัดครับ ร้าน: ........... ร้าน: "กำลังปรุงครับ ⏳" (loading) ลูกค้า: ยังอยู่ไหม?? สั่งติดหรือเปล่า? ครัว: น้ำมันหมด! ร้าน: ...........(ครัวไฟไหม้อยู่ข้างใน) ร้าน: "ขออภัยครับ วัตถุดิบขาด 🙏 ลูกค้า: เดินออกจากร้าน ไม่กลับมาอีก รับเป็นเมนูอื่น หรือรอ 5 นาที (= ผู้ใช้ลบแอป) ให้เราลองใหม่ไหมครับ?" (error + retry) กฎของร้านสุภาพ: ลูกค้าต้องรู้สถานะออเดอร์ตัวเอง "ตลอดเวลา"

และนี่คือความจริงที่ครอบทุกแอปในโลก — ทุกหน้าจอที่โหลดข้อมูล มีได้แค่ 4 หน้า

เริ่มโหลด │ ▼ ⏳ LOADING ◄──────────────────┐ "กำลังปรุงครับ" │ │ │ กดปุ่ม "ลองอีกครั้ง" ┌────────────┴────────────┐ │ สำเร็จ ✓ ล้มเหลว ✗ │ │ └─────► ❌ ERROR ┌────────┴────────┐ "ขออภัย + วิธีแก้ + ปุ่มลองใหม่" ▼ ▼ ✅ SUCCESS 📭 EMPTY (มีข้อมูล แสดงเลย) (สำเร็จแต่ไม่มีของ เช่น "ยังไม่มีงาน 🎉" — คุณทำแล้วในบท 13!)
🤔 ทำไมภาพนี้เปลี่ยนวิธีทำงานของคุณ

ต่อไปนี้เมื่อได้โจทย์ "ทำหน้าจอแสดง X" คำถามแรกคือ "ทั้ง 4 หน้าของจอนี้หน้าตาเป็นยังไง?" — ดีไซเนอร์มืออาชีพออกแบบครบ 4 หน้าเสมอ dev มืออาชีพเขียนครบ 4 หน้าเสมอ และคนสัมภาษณ์งานชอบถามเรื่องนี้มาก

2. 💻 ลงมือจริง: อัปเกรด weather_app ให้สุภาพ (3 ขั้น)

ขั้นที่ 1 — รู้จักตาข่ายนิรภัย: try / catch

บทที่แล้ว service ของเรา throw Exception(...) เมื่อใบเสร็จไม่ใช่ 200 — คำถามคือ เผือกร้อนที่โยนขึ้นมา ใครรับ? ถ้าไม่มีใครรับ = แอปพัง try/catch คือมือที่รอรับ:

try {
  // "ลองทำดู" — โค้ดที่อาจโยนเผือกร้อน (เน็ตหลุด, เซิร์ฟเวอร์ล่ม, JSON เพี้ยน)
  final weather = await fetchWeather(lat: 13.75, lon: 100.5);
  print('ได้ข้อมูลแล้ว: ${weather.temperature}');
} catch (e) {
  // "ถ้าพลาด มาลงตรงนี้" — แอปไม่พัง เราได้ e มาดูว่าเกิดอะไร
  print('พลาด: $e');   // เช่น SocketException: Failed host lookup
}

และเติมนาฬิกาจับเวลาให้ service (บทที่แล้วถ้าเน็ตอืด เราจะรอ...ตลอดกาล):

// ใน weather_service.dart — รอเกิน 10 วิ = โยนเผือกร้อน TimeoutException เอง
final response = await http.get(url).timeout(const Duration(seconds: 10));

ขั้นที่ 2 — ประกาศ 4 หน้าให้ชัดด้วย enum

enum คือของใหม่ตัวเดียวของบทนี้ — การประกาศ "ชุดค่าที่เป็นไปได้ตายตัว" เหมือนสวิตช์ที่มีแค่ 3 ตำแหน่ง บิดไปตำแหน่งอื่นไม่ได้:

/// สถานะของหน้าจอ — เป็นได้ทีละอย่างเท่านั้น ไม่มีทางกำกวม
enum ViewState { loading, success, error }
// (weather ไม่มีเคส empty เพราะอากาศมีค่าเสมอ — จอที่เป็น "รายการ" ถึงจะมีครบ 4)

ขั้นที่ 3 — ประกอบเข้า WeatherScreen

class _WeatherScreenState extends State<WeatherScreen> {
  ViewState _state = ViewState.loading;   // เปิดหน้ามา = กำลังปรุง
  Weather? _weather;
  String _errorMessage = '';

  @override
  void initState() {
    super.initState();
    _loadWeather();
  }

  Future<void> _loadWeather() async {
    setState(() => _state = ViewState.loading);   // ⭐ ทุกครั้งที่เริ่มโหลด กลับสู่ "กำลังปรุง"
    try {
      final weather = await fetchWeather(lat: 13.75, lon: 100.5);
      if (!mounted) return;
      setState(() {
        _weather = weather;
        _state = ViewState.success;
      });
    } catch (e) {
      debugPrint('โหลดอากาศพลาด: $e');            // 🔧 log ภาษา dev — ไว้ให้เราตามคดี (บท 14)
      if (!mounted) return;
      setState(() {
        _errorMessage = 'โหลดข้อมูลไม่สำเร็จ\nตรวจสอบอินเทอร์เน็ตแล้วลองอีกครั้ง';
        _state = ViewState.error;                 // 🙏 จอภาษาคน — ไว้ให้ผู้ใช้เข้าใจ
      });
    }
  }

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      appBar: AppBar(title: const Text('อากาศกรุงเทพฯ')),
      // switch expression (บท 5) เลือกหน้าตาม state — อ่านครบทุกหน้าในสายตาเดียว
      body: switch (_state) {
        ViewState.loading => const Center(child: CircularProgressIndicator()),
        ViewState.error   => _ErrorView(message: _errorMessage, onRetry: _loadWeather),
        ViewState.success => _WeatherView(weather: _weather!),
        // _weather! ปลอดภัยตรงนี้ที่เดียว — เพราะเข้า success ได้ก็ต่อเมื่อมีข้อมูลแล้ว
      },
    );
  }
}

/// หน้าตอนพลาด — ขอโทษ + บอกวิธีแก้ + "ทางไปต่อ" เสมอ (Widget ย่อย บท 11)
class _ErrorView extends StatelessWidget {
  const _ErrorView({required this.message, required this.onRetry});
  final String message;
  final VoidCallback onRetry;          // callback บท 12 — ปุ่มนี้สั่งโหลดใหม่

  @override
  Widget build(BuildContext context) {
    return Center(
      child: Column(
        mainAxisAlignment: MainAxisAlignment.center,
        children: [
          const Icon(Icons.cloud_off, size: 72, color: Colors.grey),
          const SizedBox(height: 16),
          Text(message, textAlign: TextAlign.center),
          const SizedBox(height: 16),
          FilledButton.icon(
            onPressed: onRetry,
            icon: const Icon(Icons.refresh),
            label: const Text('ลองอีกครั้ง'),
          ),
        ],
      ),
    );
  }
}
💡 ทดสอบให้ครบทุกหน้า

รันปกติ → เห็น loading แวบแล้ว success · ปิด WiFi/ข้อมูลมือถือ → กดรีเฟรช → เห็นหน้า error สุภาพพร้อมปุ่ม · เปิดเน็ตกลับ → กด "ลองอีกครั้ง" → กลับสู่ loading → success — แอปของคุณไม่มีทางตันอีกต่อไป

3. 🔧 เทคนิคการประยุกต์ใช้

เทคนิคที่ 1 — สองภาษาเสมอ: จอพูดภาษาคน, log พูดภาษา dev

ผู้ใช้เห็น "ตรวจสอบอินเทอร์เน็ตแล้วลองอีกครั้ง" (ทำตามได้) — ไม่ใช่ SocketException (ตกใจเปล่าๆ) ส่วนของจริงเก็บลง debugPrint/ระบบ log ไว้ให้เราไล่คดี — จำหลักจากบทที่ 16: ข้อความ error ที่ดี "บอกวิธีแก้ ไม่ใช่ดุ" ตอนนี้ขยายผล: และต้องเป็นภาษาที่ผู้อ่านคนนั้นเข้าใจ

เทคนิคที่ 2 — ทุกหน้า error ต้องมี "ทางไปต่อ"

ปุ่มลองอีกครั้ง, ปุ่มกลับหน้าแรก, ลิงก์ติดต่อ — อะไรก็ได้ที่ไม่ใช่ทางตัน ผู้ใช้ที่เจอกำแพงแล้วไม่มีประตู = ผู้ใช้ที่ปิดแอปถาวร (สังเกตว่า _ErrorView เราบังคับ required this.onRetry — โครงสร้างบังคับให้ลืมไม่ได้)

เทคนิคที่ 3 — timeout ทุกการรอเน็ต ไม่มีข้อยกเว้น

การรอที่ไม่มีเส้นตาย = loading หมุนตลอดกาล = ทางตันแบบหนึ่ง — 10 วินาทีคือค่ามาตรฐานที่ดีสำหรับ GET ทั่วไป (เน็ตช้าจริงยังทัน แต่ไม่ปล่อยให้รอจนสิ้นหวัง)

เทคนิคที่ 4 — RefreshIndicator: ดึงลงเพื่อโหลดใหม่ในบรรทัดเดียว

ท่า "ดึงหน้าจอลงเพื่อรีเฟรช" ที่ผู้ใช้มือถือทุกคนคุ้นมือ — Flutter ให้ฟรี: ครอบเนื้อหาด้วย RefreshIndicator(onRefresh: _loadWeather, child: ...) (child ต้องเลื่อนได้ เช่น ListView/SingleChildScrollView) — ได้ UX ระดับแอปดังด้วยโค้ดหนึ่งบรรทัด

5. 🧹 ตัวอย่าง Clean Code: enum เดียว ดีกว่า bool สามตัว

// ❌ จัดการสถานะด้วย bool หลายตัว — เปิดประตูให้ "สถานะที่เป็นไปไม่ได้"
bool _isLoading = false;
bool _hasError = false;
Weather? _weather;
// 3 ตัวแปร = 8 ชุดค่าที่เป็นไปได้... แต่หลายชุด "ไม่มีจริงในโลก":
//   _isLoading=true + _hasError=true พร้อมกัน?? (กำลังโหลดแต่พังแล้ว?)
//   _hasError=true แต่ _weather มีค่า?? (พังแต่มีข้อมูล?)
// วันไหนลืมรีเซ็ตตัวใดตัวหนึ่ง → จอแสดงสองอย่างซ้อนกัน → บั๊กที่งงที่สุดในโลก

if (_isLoading) return Spinner();
if (_hasError) return ErrorView();     // ← ลำดับ if ก็มีผลอีก... เปราะไปหมด
return WeatherView(_weather!);
// ✅ enum เดียว — สถานะเดียวเสมอ สลับกันไม่ได้ ลืมเคสไม่ได้
ViewState _state = ViewState.loading;   // เป็นได้ทีละอย่าง จบ

body: switch (_state) {
  ViewState.loading => ...,
  ViewState.error   => ...,
  ViewState.success => ...,
  // ลืมเคสไหน คอมไพเลอร์เตือนทันที (switch expression บังคับครบทุกค่า!)
},
🧠 หลักทอง

นี่คือหลัก "state น้อยที่สุด" (บทที่ 10) ในร่างที่ลึกขึ้น — อย่าแค่ลดจำนวน state ให้ออกแบบชนิดของ state จนค่าที่เป็นไปไม่ได้ ไม่มีทางเขียนขึ้นมาได้เลย ประโยคติดปากของวงการคือ "make impossible states impossible"

6. ⚠️ ข้อผิดพลาดที่เจอบ่อย

① catch แล้วเงียบ — บาปมหันต์ของการจัดการ error

สาเหตุ: เขียน catch (e) {} ว่างเปล่า "กันแอปพังไว้ก่อน" — แอปไม่พังจริง แต่ปัญหาหายเข้ากลีบเมฆ: ผู้ใช้เห็นจอค้างเฉยๆ เราก็ไม่มี log ให้ไล่ (บั๊กที่มองไม่เห็น แก้ไม่ได้ตลอดกาล — แย่กว่าแอปพังตรงๆ เสียอีก)
วิธีแก้: catch ทุกครั้งต้องทำอย่างน้อย 2 อย่าง: ① debugPrint/log ตัว e ② เปลี่ยน state ให้จอบอกผู้ใช้ — "รับเผือกร้อนแล้วต้องประกาศ ไม่ใช่ซุกทิ้ง"

② กด "ลองอีกครั้ง" แล้วจอค้างที่หน้า error เดิม

สาเหตุ: ใน _loadWeather ลืมบรรทัด setState(() => _state = ViewState.loading); ตอนต้นฟังก์ชัน — โหลดรอบสองทำงานอยู่จริง แต่จอยังแสดง error เก่า ผู้ใช้คิดว่าปุ่มพัง กดรัวๆ (ยิงซ้ำหลาย request ฟรีๆ)
วิธีแก้: กฎ: ทุกครั้งที่เริ่มโหลด กลับสู่ loading ก่อนเสมอ — วงจรต้องครบ: loading → success/error → (retry) → loading → ...

③ ใช้ _weather! นอกเคส success แล้ว crash

สาเหตุ: เผลอเข้าถึง _weather! ในที่ที่ state อาจยังไม่ success — ตอน error _weather เป็น null → "Null check operator used on a null value" (เพื่อนเก่าบทที่ 4 ทวงหนี้ !)
วิธีแก้: กติกาเดียว: แตะ _weather! ได้เฉพาะใน branch success ของ switch ที่อื่นถ้าจำเป็นใช้ _weather?.x ?? ค่าสำรอง — ! ทุกตัวต้องอธิบายได้ (บท 4)

7. 🏋️ แบบฝึกหัดท้ายบท (ลงมือทำจริง)

🥉 ระดับ 1 — ครบทุกหน้า

อัปเกรด weather_app ตามหัวข้อ 2 ทั้ง 3 ขั้น แล้วทดสอบครบวงจรด้วยตัวเอง: เห็น loading → success, ปิด WiFi → เห็น error สุภาพ, เปิดเน็ต + กดลองใหม่ → กลับมา success — โบนัส: ใส่ Future.delayed 3 วินาทีใน service ชั่วคราว เพื่อดูหน้า loading ชัดๆ (เทคนิคบทที่ 9) แล้วเอาออก

🥈 ระดับ 2 — error ที่ฉลาดขึ้น

(1) แยกประเภทเผือกร้อนด้วย on ... catch: on TimeoutException → "เครือข่ายช้า ลองใหม่อีกครั้ง" / อื่นๆ → ข้อความทั่วไป (แต่ละแบบ log ต่างกัน) (2) ครอบหน้า success ด้วย RefreshIndicator ให้ดึงลงรีเฟรชได้ (3) แสดงเวลา "อัปเดตล่าสุด" เพื่อให้ผู้ใช้รู้ว่าข้อมูลสดแค่ไหน — commit ด้วย feat: typed errors and pull-to-refresh

🥇 ระดับ 3 — ย้อนเกล็ดปิดหนี้เก่า

กลับไป todo_app: (1) ครอบ loadTasks ด้วย try-catch — จำ error ③ ของบทที่ 18 ได้ไหม (FormatException จากข้อมูลเสีย)? ตอนนี้ปิดหนี้: ข้อมูลในเครื่องเสียหาย แอปต้องเปิดขึ้นพร้อม SnackBar "ข้อมูลเดิมเสียหาย เริ่มรายการใหม่" แทนที่จะ crash (2) ทดสอบจริง: ใช้ prefs.setString('tasks', 'ไม่ใช่ json!') ชั่วคราวเพื่อจำลองข้อมูลเสีย (3) จดลง BUGS.md ว่าซ้อมเคสนี้แล้ว — commit ทั้งสอง repo ด้วย fix: survive corrupted local data

8. ✅ เช็กลิสต์และสรุปบทเรียน

  • วาดผัง 4 หน้า (loading/success/empty/error) พร้อมลูกศร retry ได้จากความเข้าใจ
  • เขียน try-catch ที่ "ไม่เงียบ": log + เปลี่ยน state เสมอ
  • อธิบายได้ว่าทำไม enum เดียวดีกว่า bool หลายตัว (impossible states)
  • จอพูดภาษาคน log พูดภาษา dev — แยกสองภาษาเป็นนิสัยแล้ว
  • ทุกการรอเน็ตมี timeout และทุกหน้า error มีทางไปต่อ
  • ทดสอบด้วยการ "ทำให้พังจริง" (ปิด WiFi, ข้อมูลเสีย) แล้วแอปรอดทุกเคส

📖 สรุปศัพท์ใหม่บทนี้

ศัพท์ความหมายสั้นๆภาพจำ

UI state4 หน้าของทุกจอโหลดข้อมูลกำลังปรุง/ได้จาน/จานว่าง/ขอโทษ

🎓 สรุปบทนี้ใน 3 บรรทัด

ทุกจอโหลดข้อมูลมี 4 หน้า — ออกแบบครบก่อนเขียน · try-catch ต้องไม่เงียบ: log + บอกผู้ใช้เสมอ · enum เดียวปิดประตูสถานะที่เป็นไปไม่ได้ · error ทุกหน้าต้องมีทางไปต่อ — แอปที่สุภาพตอนพัง คือแอปที่ผู้ใช้ให้อภัย

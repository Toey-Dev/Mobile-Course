1. 🖼️ ภาพเล่าเรื่อง: สั่งอาหารจากครัวของคนอื่น

📱 แอปเรา (ลูกค้า) 🌐 เซิร์ฟเวอร์ (ครัวของเขา) │ │ │ ① สั่งอาหารตามเมนู (HTTP GET request) │ │ "ขอสภาพอากาศ กรุงเทพฯ 1 ที่ครับ" │ ├──────────────► 🤵 API ────────────────────►│ │ (พนักงานเสิร์ฟ │ ② ครัวปรุง │ รับออเดอร์ตามเมนูเท่านั้น) │ (ดึงจากฐานข้อมูล) │ │ │◄───────────── 🤵 ◄───────────────────────┤ │ ③ จานอาหารมาส่ง (response) │ │ พร้อมใบเสร็จบอกผล (status code): │ │ 200 = ได้จานครบ ✓ │ │ 404 = เมนูนี้ไม่มีนะครับ │ │ 500 = ขอโทษครับ ครัวไฟไหม้ 🔥 │ ▼ จานที่ได้ = ข้อความ JSON (เพื่อนเก่าจากบทที่ 18!)

แล้ว "เมนู" อยู่ไหน? — กายวิภาคของ endpoint

https://api.open-meteo.com/v1/forecast?latitude=13.75&longitude=100.5&current_weather=true ─────────┬────────────── ────┬────── ─────────────────────┬───────────────────────── ที่อยู่ร้าน ชื่อเมนู รายละเอียดออเดอร์ (query parameters) (โดเมนของเซิร์ฟเวอร์) (endpoint) "พิกัดนี้นะ ขอสภาพอากาศปัจจุบันด้วย" ทั้งเส้นนี้เรียกว่า URL — เหมือนสั่ง: "ร้าน open-meteo ครับ เมนู forecast พิกัด 13.75, 100.5 เอาแบบ current weather" — เปลี่ยนพิกัด = ได้เมืองอื่น
🤔 ระหว่างรอจานมาส่ง แอปทำอะไร

นี่แหละที่สัญญากันไว้ตั้งแต่บทที่ 9 — การสั่งอาหารข้ามอินเทอร์เน็ตใช้เวลา 0.2–3 วินาที (นานมากในโลกโปรแกรม!) ถ้ายืนเฝ้าเคาน์เตอร์ = แอปค้างทั้งแอป จึงต้องรับบัตรคิว (Future) แล้ว await — ระหว่างรอ ผู้ใช้ยังเลื่อนจอ กดปุ่มอื่นได้ปกติ วันนี้คุณจะได้เห็นมันทำงานกับการรอ "ของจริง" ครั้งแรก

2. 💻 ลงมือจริง: แอปสภาพอากาศตัวแรก (5 ขั้น)

เราใช้ Open-Meteo — API สภาพอากาศจริงที่ฟรีและไม่ต้องสมัคร/ไม่ต้องมี key (ศูนย์อุปสรรค เริ่มได้เลย)

ขั้นที่ 0 — กฎเหล็ก: เปิด URL ในเบราว์เซอร์ก่อนเสมอ!

ก่อนเขียนโค้ดแม้แต่บรรทัดเดียว วางลิงก์นี้ในเบราว์เซอร์:

https://api.open-meteo.com/v1/forecast?latitude=13.75&longitude=100.5&current_weather=true

คุณจะเห็น JSON สดๆ จากเซิร์ฟเวอร์ (พิกัดกรุงเทพฯ):

{
  "latitude": 13.75,
  "longitude": 100.5,
  "current_weather": {
    "temperature": 32.5,
    "windspeed": 8.4,
    "weathercode": 1,
    "time": "2026-07-19T10:00"
  }
}

อ่านออกใช่ไหมครับ — นี่คือผลของการบ้านบทที่ 18: {} คือ Map, ข้างในมี key current_weather ที่เป็น Map ซ้อนอีกชั้น (JSON ซ้อนกันได้เหมือน Map ซ้อน Map บทที่ 7) — การเห็นโครงจริงก่อน คือเข็มทิศของทุกบรรทัดที่กำลังจะเขียน

ขั้นที่ 1 — สร้างโปรเจกต์ + ติดตั้งโทรศัพท์สั่งอาหาร

ขั้นที่ 2 — สร้าง model จากโครง JSON ที่เห็น (lib/models/weather.dart)

/// พิมพ์เขียวสภาพอากาศ — สร้างจากการ "อ่านโครง JSON จริง" ในขั้นที่ 0
class Weather {
  const Weather({required this.temperature, required this.windSpeed});

  final double temperature;
  final double windSpeed;

  factory Weather.fromJson(Map<String, dynamic> json) {
    // JSON ซ้อนชั้น: เปิดล็อกเกอร์ current_weather ก่อน แล้วค่อยหยิบของข้างใน
    final current = json['current_weather'] as Map<String, dynamic>;
    return Weather(
      // (as num).toDouble() = ท่ากันกับดักตัวเลข JSON (ดูหัวข้อข้อผิดพลาด ②)
      temperature: (current['temperature'] as num).toDouble(),
      windSpeed: (current['windspeed'] as num).toDouble(),
    );
  }
}

ทักษะเดิมจากบทที่ 18 เป๊ะ — ต่างแค่คราวนี้ เราไม่ได้เป็นคนเขียน JSON เอง เลยต้องอ่านโครงจากของจริงก่อน (ขั้นที่ 0 สำคัญด้วยเหตุนี้)

ขั้นที่ 3 — ฟังก์ชันสั่งอาหาร (lib/services/weather_service.dart)

import 'dart:convert';
import 'package:http/http.dart' as http;   // as http = ตั้งชื่อเล่นให้ package
import '../models/weather.dart';

/// โทรสั่ง "สภาพอากาศปัจจุบัน" จากครัว Open-Meteo
Future<Weather> fetchWeather({required double lat, required double lon}) async {
  // Uri.parse = จดที่อยู่ร้านให้ถูกรูปแบบ (http ต้องการ Uri ไม่ใช่ String ดิบ)
  final url = Uri.parse(
    'https://api.open-meteo.com/v1/forecast'
    '?latitude=$lat&longitude=$lon&current_weather=true',   // interpolation บท 4!
  );

  final response = await http.get(url);   // ☎️ สั่ง → รับบัตรคิว → รอจานมาส่ง

  // เช็คใบเสร็จก่อนกินเสมอ! 200 = สำเร็จ (guard clause บท 5)
  if (response.statusCode != 200) {
    throw Exception('เซิร์ฟเวอร์ตอบ ${response.statusCode}');
  }

  // จานอาหาร (response.body) คือ "ข้อความ JSON" → พองกลับ → ปั้นเป็น object
  final json = jsonDecode(response.body) as Map<String, dynamic>;
  return Weather.fromJson(json);
}

อ่านไล่ดูสิครับ — ทุกบรรทัดคือความรู้เก่า: Future+async+await (บท 9), interpolation (บท 4), guard clause (บท 5), jsonDecode+fromJson (บท 18) — บทนี้ไม่ได้สอนของใหม่มาก มันคือสนามที่ของเก่าทุกชิ้นมาเล่นด้วยกัน

ขั้นที่ 4 — หน้าจอแสดงผล (lib/main.dart — ใช้ Stateful ที่คุ้นมือ)

class WeatherScreen extends StatefulWidget {
  const WeatherScreen({super.key});
  @override
  State<WeatherScreen> createState() => _WeatherScreenState();
}

class _WeatherScreenState extends State<WeatherScreen> {
  Weather? _weather;   // null = ยังไม่มีข้อมูล (nullable บท 4 — ตรงตามความหมายเป๊ะ)

  @override
  void initState() {
    super.initState();
    _loadWeather();    // เปิดหน้าปุ๊บ สั่งอาหารเลย (initState บท 12)
  }

  Future<void> _loadWeather() async {
    final weather = await fetchWeather(lat: 13.75, lon: 100.5);  // กรุงเทพฯ
    if (!mounted) return;               // ผู้ใช้อาจปิดหน้าไปแล้วระหว่างรอ (บท 15)
    setState(() => _weather = weather);
  }

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      appBar: AppBar(title: const Text('อากาศกรุงเทพฯ')),
      body: Center(
        child: _weather == null
            ? const CircularProgressIndicator()          // ยังไม่มา = วงกลมหมุน
            : Column(
                mainAxisAlignment: MainAxisAlignment.center,
                children: [
                  Text('${_weather!.temperature} °C',
                      style: const TextStyle(fontSize: 56, fontWeight: FontWeight.bold)),
                  Text('ลม ${_weather!.windSpeed} กม./ชม.',
                      style: const TextStyle(fontSize: 18, color: Colors.grey)),
                ],
              ),
      ),
      floatingActionButton: FloatingActionButton(
        onPressed: _loadWeather,                          // กดรีเฟรชได้
        child: const Icon(Icons.refresh),
      ),
    );
  }
}

3. 🔧 เทคนิคการประยุกต์ใช้

เทคนิคที่ 1 — เบราว์เซอร์ก่อนโค้ดเสมอ (ขั้นที่ 0 คือนิสัย ไม่ใช่ขั้นตอนครั้งเดียว)

ทุกครั้งที่จะใช้ endpoint ใหม่: วางใน browser → อ่านโครง → ค่อยเขียน model — เดาโครง JSON เอาเองคือสาเหตุอันดับหนึ่งของการนั่งงมครึ่งวัน (มืออาชีพใช้เครื่องมืออย่าง Postman/Bruno แต่ browser ก็พอสำหรับ GET)

เทคนิคที่ 2 — แปลงเป็น model ทันทีที่ "ชายแดน" — Map ห้ามเดินทางเข้าแอป

jsonDecode เกิดที่ service แล้ว fromJson ทันที — ห้ามส่ง Map<String, dynamic> ดิบๆ ให้หน้าจอเด็ดขาด (เหตุผลเต็มๆ อยู่บทที่ 8: สะกด key ผิด = บั๊กเงียบ) ทีมจริงเรียกจุดนี้ว่า boundary: ข้อมูลข้ามชายแดนเข้าประเทศเมื่อไหร่ ต้องแปลงสัญชาติเป็น object ทันที

เทคนิคที่ 3 — (as num).toDouble() คือท่าประจำกับตัวเลขจาก JSON

JSON ไม่แยก int/double — 32.5 มาเป็น double แต่ 32 (เลขกลมพอดี) มาเป็น int! โค้ดที่เขียน as double ตรงๆ จะผ่านตอนทดสอบ แล้วพังกลางคืนวันที่อุณหภูมิ 32° พอดี — บั๊กตามฤดูกาลของแท้ ปิดประตูด้วย (x as num).toDouble() เสมอ

เทคนิคที่ 4 — query parameters คือ "ช่องปรับแต่งออเดอร์"

ลอง &current_weather=true เอาออก, เปลี่ยน latitude — แล้วดูผลใน browser เอกสารของ API (API docs) คือเมนูเล่มเต็มที่บอกว่าปรับอะไรได้บ้าง — ทักษะ "อ่าน docs แล้วประกอบ URL" นี้ใช้กับ API ทุกตัวในโลก

4. 🏭 ตัวอย่างการเขียนจริงในงานจริง

🏭 สถานการณ์จริง: "วันแรกของ Flutter dev ในทีม — สิ่งแรกที่ได้ไม่ใช่โค้ด แต่คือเมนู"

ในบริษัทจริง แอปมือถือกับเซิร์ฟเวอร์เขียนโดยคนละทีม — สิ่งที่เชื่อมสองทีมคือเอกสารที่เรียกว่า API contract (สัญญาเมนู):

ทีม Backend (ครัว) ทีม Mobile (เรา) │ │ └────── 📋 API Contract ────────────────┘ "GET /orders/{id} ตอบ 200 พร้อม JSON: { id, status, total, items: [...] }" · สองทีมทำงาน "พร้อมกัน" ได้เพราะตกลงเมนูก่อน — mobile เขียน model จาก contract ได้เลยโดยครัวจริงยังสร้างไม่เสร็จ (ระหว่างรอ ใช้ข้อมูลปลอมที่หน้าตาตรง contract) · ครัวอยากเปลี่ยนเมนู (เพิ่ม/ลบ field)? ต้องแจ้งก่อน — ไม่งั้นแอปที่อยู่ในมือ ผู้ใช้ล้านคนพังพร้อมกัน (นี่คือเหตุผลที่ fromJson ต้องมีเกราะ ?? จากบทที่ 18)

การประชุมที่คุณจะเข้าบ่อยที่สุดในชีวิตการทำงานสาย mobile คือ "คุย API กับทีม backend" — และภาษาที่ใช้คุยกันคือทุกคำที่คุณเรียนวันนี้: endpoint, GET, status code, response body

5. 🧹 ตัวอย่าง Clean Code: สามชั้นที่แยกกันชัด

✅ สามชั้น: model / service / หน้าจอ — แต่ละชั้นมีหน้าที่เดียว lib/ ├── models/weather.dart ① รูปร่างข้อมูล + วิธีปั้นจาก JSON (fromJson) ├── services/weather_service.dart ② วิธีสั่งจากครัวไหน เมนูอะไร เช็คใบเสร็จยังไง └── main.dart ③ หน้าจอ: ขอข้อมูลจาก ② แล้วแสดง — ไม่รู้จัก http เลย · หน้าจอทดสอบได้โดยไม่ต้องมีเน็ต (ส่ง Weather ปลอมให้) · เปลี่ยนเจ้า API (ย้ายครัว)? แก้ไฟล์ ② ไฟล์เดียว — หน้าจอไม่สะเทือน · โครงนี้คือตัวอย่างแรกของ "สถาปัตยกรรม" ที่บทที่ 21 จะจัดเต็มทั้งแอป
🧠 เส้นเรื่องยาวของหลักสูตร

ความรู้อยู่กับข้อมูล (บท 8) → กฎอยู่ใน provider (บท 17) → การเก็บอยู่ใน provider (บท 18) → การสั่งจากเน็ตอยู่ใน service (บทนี้) — ทุกบทคือประโยคเดียวกัน: แต่ละเรื่องมีบ้านของมัน

6. ⚠️ ข้อผิดพลาดที่เจอบ่อย

① "SocketException: Failed host lookup" / ClientException

สาเหตุ: ติดต่อเซิร์ฟเวอร์ไม่ได้ — เน็ตหลุด, พิมพ์ URL ผิด (โดเมนสะกดผิด), หรือ (เจอบ่อยตอน build จริง) Android ฉบับ release ไม่ได้ประกาศขอสิทธิ์อินเทอร์เน็ตใน AndroidManifest.xml (ตอน debug ได้สิทธิ์ฟรี)
วิธีแก้: ไล่ตามลำดับ: เปิด URL เดียวกันใน browser ของเครื่อง/emulator ได้ไหม → ได้ = โค้ดเราผิด (เทียบ URL ทีละตัวอักษร) · ไม่ได้ = เน็ต/เครื่องมีปัญหา (การจัดการ "เน็ตหลุดกลางทาง" แบบสุภาพ = เนื้อหาเต็มบทของบทที่ 20)

② "type 'int' is not a subtype of type 'double'"

สาเหตุ: กับดักตัวเลข JSON — อุณหภูมิ 32 (เลขกลม) ถูกแปลงเป็น int แต่ model ประกาศ double ไว้ → พังเฉพาะ "บางวัน" ตามค่าจริงของข้อมูล! (บั๊กที่ทดสอบผ่านแต่พังในมือผู้ใช้ — ตระกูลที่เจ็บที่สุด)
วิธีแก้: ตัวเลขจาก JSON ใช้ (json['x'] as num).toDouble() เสมอ — num คือแม่ของทั้ง int และ double (inheritance บท 8!) เลยรับได้ทั้งคู่

③ "NoSuchMethodError: '[]' was called on null" ตอนแกะ JSON

สาเหตุ: เส้นทางเจาะ JSON ผิดชั้น — เช่น json['current']['temperature'] แต่ของจริงชื่อ 'current_weather' → ชั้นแรกได้ null แล้วชั้นสองพยายามเปิดล็อกเกอร์บนความว่างเปล่า (เพื่อนเก่า null บท 4)
วิธีแก้: กลับไปขั้นที่ 0: debugPrint(response.body) แล้วอ่านโครงจริงทีละชั้นเทียบกับโค้ด — 90% คือสะกดชื่อ key ผิดหรือนับชั้นผิด อย่าเดา ให้ดูของจริง

7. 🏋️ แบบฝึกหัดท้ายบท (ลงมือทำจริง)

🥉 ระดับ 1 — จานแรกจากอินเทอร์เน็ต

ทำครบ 5 ขั้นจนเห็นอุณหภูมิจริงบนจอ — แล้วพิสูจน์ความเข้าใจ: debugPrint(response.body) ใน service เทียบกับที่เห็นใน browser ว่าตรงกัน, ลองปิด WiFi แล้วกดรีเฟรชดูว่าเกิดอะไร (อ่าน error ให้ออก — บทหน้าเราจะจัดการมันอย่างสุภาพ)

🥈 ระดับ 2 — ปรับแต่งออเดอร์

(1) เปลี่ยนพิกัดเป็นเมืองของคุณ (หา lat/lon จาก Google Maps — กดขวาบนแผนที่) (2) อ่าน docs ของ Open-Meteo แล้วเพิ่มข้อมูลอีก 1 อย่างที่สนใจลงจอ เช่น weathercode แปลงเป็นอีโมจิสภาพอากาศด้วย switch (บท 5): แดด ☀️ เมฆ ☁️ ฝน 🌧️ (3) แสดงเวลาอัปเดตล่าสุดใต้ตัวเลข — commit ด้วย feat: my city weather with condition emoji

🥇 ระดับ 3 — หลายเมือง (โครงโปรเจกต์ portfolio)

สร้าง City model (name, lat, lon) + List เมือง 4–5 เมือง (กรุงเทพฯ เชียงใหม่ ภูเก็ต โตเกียว ลอนดอน) → ผู้ใช้เลือกเมืองจาก DropdownButton หรือแถบ chip แล้วแอปโหลดอากาศเมืองนั้นใหม่ — โครงสร้างต้องเป็นสามชั้นตามหัวข้อ 5 (models/ services/ screens/) — git init + commit + push ขึ้น GitHub เป็น repo ที่สามของคุณ ด้วยข้อความ feat: multi-city weather app

8. ✅ เช็กลิสต์และสรุปบทเรียน

  • อธิบายภาพ "พนักงานเสิร์ฟ API" ครบวงจร: request → status code → response ได้
  • แยกส่วนประกอบของ URL ได้: โดเมน / endpoint / query parameters
  • มีนิสัย "เปิด URL ใน browser ก่อนเขียนโค้ด" แล้ว
  • เขียน service ที่ get → เช็ค statusCode → decode → fromJson ได้เอง
  • อธิบายกับดัก int/double ของ JSON และท่า (as num).toDouble() ได้
  • แอปสภาพอากาศแสดงข้อมูลจริง และโครงเป็นสามชั้น model/service/หน้าจอ

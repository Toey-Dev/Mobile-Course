🎯 ภาพรวมบทเรียน (อ่านก่อนเริ่ม)

บทที่แล้วเรารันแอปได้ แต่ยังเหมือน "ขับรถโดยไม่รู้ว่าใต้ฝากระโปรงมีอะไร" — บทนี้จะเปิดฝากระโปรงให้ดูทุกชิ้น และสอนเครื่องมือที่ dev ทั้งโลกใช้เก็บงานทุกวัน: Git

จบบทนี้คุณจะ:

  • อธิบายได้ว่าไฟล์/โฟลเดอร์แต่ละตัวในโปรเจกต์มีไว้ทำไม อะไรแตะได้ อะไรอย่าแตะ

1. 🖼️ ภาพเล่าเรื่อง: โปรเจกต์ Flutter คือ "ร้านอาหาร" ทั้งร้าน

จำได้ไหมครับ บทที่แล้วเราเปรียบ Flutter เป็น ครัวกลางที่เสิร์ฟได้ทั้งหน้าร้านและเดลิเวอรี — บทนี้เราจะเดินสำรวจ "ร้าน" ทั้งร้านกัน ว่าแต่ละห้องมีไว้ทำอะไร

ฉากที่ 1 — เดินเข้าร้าน

🏪 my_first_app/ (ร้านอาหารของเรา) │ ├── 👨‍🍳 lib/ = ครัวกลาง — ที่ที่เรา "ทำอาหาร" (เขียนโค้ด) 95% ของเวลา │ └── main.dart = เตาหลักของครัว ทุกจานเริ่มจากตรงนี้ │ ├── 📱 android/ 🍎 ios/ = หน้าร้าน 2 สาขา — โครงสร้างเฉพาะของแต่ละแพลตฟอร์ม │ (ปกติไม่ต้องเข้าไปยุ่ง เหมือนระบบไฟฟ้า-ประปาของอาคาร) │ ├── 📋 pubspec.yaml = สมุดสั่งวัตถุดิบ — อยากใช้ของจากตลาด เขียนสั่งที่นี่ │ ├── 🧪 test/ = ห้องชิมอาหาร — ตรวจรสชาติก่อนเสิร์ฟลูกค้า │ └── 🚫 .gitignore = รายการของที่ "ไม่ต้องขนย้าย" ตอนย้ายร้าน (ของที่สร้างใหม่ได้เสมอ เช่น กล่องโฟม ถุงพลาสติก)

ฉากที่ 2 — เมื่อสั่ง flutter run เกิดอะไรขึ้น

คุณกด flutter run │ ▼ ① Flutter เปิดสมุด pubspec.yaml ── "จานนี้ใช้วัตถุดิบอะไรบ้าง?" │ ▼ ② ไปหยิบวัตถุดิบ (packages) จากตลาด pub.dev มาเตรียมไว้ │ ▼ ③ เข้าครัว lib/main.dart ── เริ่มปรุงจากฟังก์ชัน main() │ ▼ ④ เสิร์ฟผ่าน "หน้าร้าน" ที่ตรงกับเครื่องคุณ (android/ หรือ ios/) │ ▼ 📱 แอปขึ้นจอ!
🤔 ทำไมต้องเห็นภาพนี้

เวลาเกิด error คุณจะรู้ทันทีว่าปัญหาอยู่ "ฉากไหน" — วัตถุดิบขาด (ฉาก ② ปัญหา pubspec) หรืออาหารปรุงผิด (ฉาก ③ ปัญหาโค้ดเรา) หรือหน้าร้านมีปัญหา (ฉาก ④ ปัญหา build) การรู้ว่าพังที่ฉากไหน = แก้ได้เร็วขึ้น 10 เท่า และนี่คือเหตุผลที่หลักสูตรนี้ใช้ "ภาพเล่าเรื่อง" นำก่อนโค้ดเสมอ: สมองจำเรื่องราวได้ดีกว่ารายการข้อเท็จจริง

2. 🧠 เจาะแต่ละไฟล์/โฟลเดอร์ — อะไรแตะได้ อะไรอย่าแตะ

ไฟล์/โฟลเดอร์คืออะไรคุณจะแตะบ่อยแค่ไหน
lib/โค้ด Dart ทั้งหมดของแอป🔥 ทุกวัน — บ้านหลักของคุณ
lib/main.dartจุดเริ่มต้นของแอป🔥 ทุกวัน (หัวข้อ 7 จะสอนวิธีแยกไฟล์)
pubspec.yamlรายการ package, ชื่อแอป, เวอร์ชัน, assets🟡 บ่อย — ทุกครั้งที่เพิ่มของใหม่
test/โค้ดทดสอบอัตโนมัติ🟡 บ่อยขึ้นเรื่อยๆ ตามโปรเจกต์โต
android/, ios/โครงสร้างเฉพาะแพลตฟอร์ม🟢 นานๆ ครั้ง (ตั้งค่า permission, ไอคอน)
build/ผลลัพธ์จากการ build — สร้างใหม่ได้เสมอ⛔ ไม่ต้องแตะ ไม่ต้องเก็บ
.dart_tool/ไฟล์ภายในของเครื่องมือ Dart⛔ ไม่ต้องแตะ
analysis_options.yamlกฎตรวจคุณภาพโค้ด (lint)🟢 ตั้งครั้งเดียว แก้เมื่ออยากเข้มขึ้น
🧠 หลักคิด dev: "ของที่เราเขียน" vs "ของที่เครื่องมือสร้าง"

แยกให้ออกระหว่าง ของที่เราเขียน (lib, test, pubspec) กับ ของที่เครื่องมือสร้าง (build, .dart_tool) — กลุ่มหลังพังก็ลบทิ้งแล้วสร้างใหม่ได้เสมอ นี่คือเหตุผลที่คำแนะนำแก้ปัญหายอดฮิต flutter clean (ลบของที่สร้าง) ถึงได้ผลบ่อย

3. 💻 ลงมือจริง: เพิ่ม package แรกจากตลาด pub.dev

pub.dev คือ "ตลาดวัตถุดิบ" ของ Flutter — โค้ดสำเร็จรูปนับหมื่นตัวที่คนทั่วโลกแบ่งปันไว้ ใช้ฟรี เราจะลองเพิ่มฟอนต์สวยๆ ด้วย package google_fonts

ขั้นที่ 1 — สั่งวัตถุดิบ

flutter pub add google_fonts

คำสั่งนี้เขียน google_fonts: ^x.x.x เพิ่มใน pubspec.yaml ให้อัตโนมัติ และดาวน์โหลดมาไว้ในเครื่อง

ขั้นที่ 2 — ใช้งานใน main.dart (แก้ 2 จุด)

import 'package:flutter/material.dart';
import 'package:google_fonts/google_fonts.dart'; // ① import วัตถุดิบใหม่

// ...ภายใน Text เดิมของเรา แก้ style:
Text(
  'สวัสดีชาวโลก! 🌏',
  style: GoogleFonts.kanit(              // ② ใช้ฟอนต์ Kanit
    fontSize: 24,
    fontWeight: FontWeight.bold,
  ),
),

ขั้นที่ 3 — ดูผล

กดบันทึก → Hot Restart (กด R) → ฟอนต์เปลี่ยน! 🎉

🤔 ทำไมครั้งนี้ต้อง Hot Restart ไม่ใช่ Hot Reload

จำตารางบทที่ 1 ได้ไหมครับ — เพิ่ม package ใหม่คือการเปลี่ยน "วัตถุดิบตั้งต้น" ของแอป (บางกรณีต้องถึงขั้นหยุดแล้ว flutter run ใหม่) Hot Reload อัปเดตเฉพาะโค้ดที่ปรุงอยู่ ไม่ได้ไปหยิบวัตถุดิบใหม่จากตลาด

4. 🖼️ ภาพเล่าเรื่อง: Git คือ "จุดเซฟในเกม"

เคยเล่นเกมที่มี save point ไหมครับ — ก่อนสู้บอส เราเซฟไว้ก่อน ถ้าแพ้ก็ย้อนกลับมาที่จุดเซฟ ไม่ต้องเริ่มเกมใหม่ทั้งหมด Git ทำแบบเดียวกันกับโค้ดของคุณ

การทำงานของคุณบนไฟล์ ประวัติที่ Git จำไว้ (repository) ┌──────────────────────────┐ ┌────────────────────────────────┐ │ แก้ไฟล์ A, B ... │ │ ● commit 3 "เพิ่มฟอนต์ Kanit" │ │ (ยังไม่เซฟเข้า Git) │ │ │ │ │ │ │ commit │ ● commit 2 "เพิ่มไอคอน" │ │ ▼ │ ────────► │ │ │ │ git add (เลือกของที่จะ │ │ ● commit 1 "Hello World" │ │ เซฟ = จัดของขึ้นโต๊ะ) │ │ │ └──────────────────────────┘ │ ⏪ ย้อนกลับไปจุดไหนก็ได้เสมอ │ └────────────────────────────────┘

เรื่องเล่า 3 ฉากที่เกิดขึ้นทุกวันในชีวิต dev

ฉาก ① แก้โค้ดจนพอใจ → งานอยู่ใน "โต๊ะทำงาน" (ยังไม่ปลอดภัย) ฉาก ② git add . → ยกงานขึ้น "โต๊ะเตรียมเซฟ" (stage — เลือกได้ว่าเอาชิ้นไหน) ฉาก ③ git commit -m "..." → 📸 ถ่ายรูปเก็บเข้าอัลบั้มถาวร พร้อมป้ายบอกว่ารูปนี้คืออะไร
🤔 ทำไมต้องมีขั้น stage (git add) คั่นกลาง

เพื่อให้เลือกได้ว่า commit นี้จะรวมไฟล์ไหน — เหมือนถ่ายรูปหมู่ที่เราจัดคนเข้าเฟรมก่อนกดชัตเตอร์ งานจริงเราแก้หลายเรื่องพร้อมกันบ่อยมาก ขั้นนี้ทำให้แยก commit เป็นเรื่องๆ ได้ (1 commit = 1 เรื่อง คือมาตรฐานมืออาชีพ)

คำสั่งที่ใช้จริง 90% ของชีวิต มีแค่ 5 ตัวนี้

git status    # ตอนนี้มีอะไรเปลี่ยนบ้าง? (ใช้บ่อยสุด — สั่งก่อนทำอะไรเสมอ)
git add .     # ยกทุกไฟล์ที่แก้ ขึ้นโต๊ะเตรียมเซฟ
git commit -m "feat: add Kanit font"   # เซฟถาวร พร้อมข้อความอธิบาย
git log --oneline   # ดูอัลบั้มประวัติทั้งหมด
git diff      # ดูว่าแก้อะไรไปบ้างแบบบรรทัดต่อบรรทัด (ก่อน commit ควรดูเสมอ)

.gitignore — รายการของที่ไม่ต้องขนย้าย

เปิดไฟล์ .gitignore ในโปรเจกต์ดู จะเห็นรายชื่ออย่าง /build/, .dart_tool/ — นี่คือการบอก Git ว่า "ของพวกนี้ไม่ต้องจำ" เพราะเป็นของที่สร้างใหม่ได้เสมอและมีขนาดใหญ่มาก

⚠️ ทำไมสำคัญ

repo ที่ commit โฟลเดอร์ build/ เข้าไปจะบวมหลายร้อย MB, โคลนช้า, และคนในทีมตีกันเพราะไฟล์ build ของแต่ละเครื่องไม่เหมือนกัน — ข่าวดีคือ flutter create เขียน .gitignore มาตรฐานให้แล้ว หน้าที่คุณคือ อย่าลบมัน และรู้ว่ามันมีไว้ทำไม

5. 🔧 เทคนิคการประยุกต์ใช้

เทคนิคที่ 1 — commit เล็กและบ่อย ดีกว่า commit ใหญ่วันละครั้ง

ทำฟีเจอร์เสร็จ 1 เรื่อง → commit ทันที เช่น เปลี่ยนฟอนต์เสร็จ = 1 commit, เพิ่มไอคอน = อีก 1 commit ถ้าพังจะย้อนกลับได้ละเอียดระดับ "เรื่อง" ไม่ใช่ย้อนทีเสียงานทั้งวัน

เทคนิคที่ 2 — ใช้ git status เป็น "เข็มทิศ" เวลาหลงทาง

ไม่แน่ใจว่าแก้อะไรไปบ้าง? ลืมว่าทำถึงไหน? git status + git diff ตอบได้เสมอ นิสัยนี้ช่วยแม้กระทั่งตอนกลับมาเปิดโปรเจกต์หลังหยุดไปหลายวัน

เทคนิคที่ 3 — ก่อนทดลองอะไรเสี่ยงๆ ให้ commit ก่อน

อยากลองรื้อ layout ใหม่ทั้งหน้า? commit สถานะที่ทำงานได้ไว้ก่อน แล้วรื้อได้เต็มที่ ถ้าเละก็กลับมาได้ด้วย git checkout -- . (ทิ้งการแก้ทั้งหมด กลับไปจุดเซฟล่าสุด) — นี่คือ "สู้บอสแบบมี save point"

เทคนิคที่ 4 — เขียนข้อความ commit ให้ "ตัวเองในอนาคต" อ่าน

git log ของคุณคือไดอารี่การทำงาน ข้อความแบบ "แก้บั๊ก" บอกอะไรไม่ได้เลย แต่ "fix: ปุ่มเพิ่มงานไม่ทำงานเมื่อช่องข้อความว่าง" อ่านปุ๊บรู้ปั๊บแม้ผ่านไป 6 เดือน

6. 🏭 ตัวอย่างการเขียนจริงในงานจริง

🏭 สถานการณ์จริง: "บั๊กด่วนตอนบ่ายสาม"

คุณกำลังทำฟีเจอร์ใหม่ค้างอยู่ครึ่งทาง จู่ๆ หัวหน้าทักมา: "ลูกค้าเจอบั๊กบนแอปจริง แก้ด่วนเลย!" — งานที่ค้างอยู่ยังไม่เสร็จ commit ก็ยังไม่ได้ จะทิ้งก็เสียดาย ทำไงดี?

git stash          # 📦 เก็บงานค้างใส่กล่องชั่วคราว — ไฟล์กลับไปสะอาดเหมือนจุดเซฟล่าสุด
# ...แก้บั๊กด่วน ทดสอบ commit เรียบร้อย...
git stash pop      # 📤 หยิบงานค้างออกจากกล่อง กลับมาทำต่อเหมือนไม่มีอะไรเกิดขึ้น

นี่คือเหตุการณ์ที่เกิดขึ้น ทุกสัปดาห์ ในทีมจริง และเป็นเหตุผลว่าทำไม dev ที่ใช้ Git คล่องถึงทำงานในทีมได้ราบรื่นกว่า

ภาพการทำงานเป็นทีมจริง (จะได้เจอเมื่อฝึกงาน/ทำงาน)

งานจริงไม่มีใครแก้โค้ดบนเส้นหลัก (main) ตรงๆ: main ──●──────────────●─────────●──► (เส้นนี้ต้อง "ใช้งานได้เสมอ") \ ▲ merge \ / (ผ่านการรีวิวจากเพื่อนร่วมทีมก่อน = Pull Request) ●────●───● branch "feature/add-login" (ห้องทดลองส่วนตัว พังได้ไม่กระทบใคร)

ตอนนี้คุณทำงานคนเดียว ยังไม่ต้องใช้ branch ทุกวันก็ได้ แต่จำภาพนี้ไว้ — บทท้ายๆ เราจะฝึกจริง และวันแรกที่เข้าทีม คุณจะขอบคุณตัวเองที่เข้าใจภาพนี้มาก่อน

7. 🧹 ตัวอย่าง Clean Code: แยกไฟล์ก่อนที่ main.dart จะกลายเป็น "ลิ้นชักรวมของ"

โปรเจกต์มือใหม่เกือบทุกคนเจอปัญหาเดียวกัน: เขียนทุกอย่างใน main.dart จนยาว 800 บรรทัด หาอะไรไม่เจอ — มาดูวิธีป้องกันตั้งแต่ตอนนี้

❌ แบบรีบๆ — ทุกอย่างกองใน main.dart

✅ แบบมืออาชีพ — หนึ่งไฟล์ หนึ่งหน้าที่

main.dart เหลือแค่จุดสตาร์ท ~15 บรรทัด หน้าจอแต่ละหน้ามี "บ้าน" ของตัวเองใน lib/screens/

lib/ ├── main.dart # เหลือแค่จุดสตาร์ท (สั้นมาก ~15 บรรทัด) └── screens/ └── home_screen.dart # หน้าจอหลัก แยกบ้านของตัวเอง
// lib/screens/home_screen.dart
import 'package:flutter/material.dart';

/// หน้าจอหลัก — ย้ายมาจาก main.dart ทั้งก้อน ไม่ต้องแก้อะไรข้างใน
class HomeScreen extends StatelessWidget {
  const HomeScreen({super.key});

  @override
  Widget build(BuildContext context) {
    return Scaffold(/* ...โค้ดเดิมจากบทที่ 1... */);
  }
}
// lib/main.dart — สะอาด อ่านจบใน 10 วินาที
import 'package:flutter/material.dart';
import 'screens/home_screen.dart'; // ดึงหน้าจอจากไฟล์ของมันเอง

void main() {
  runApp(const MyApp());
}

class MyApp extends StatelessWidget {
  const MyApp({super.key});

  @override
  Widget build(BuildContext context) {
    return MaterialApp(
      title: 'แอปแรกของฉัน',
      theme: ThemeData(colorSchemeSeed: Colors.indigo),
      home: const HomeScreen(),
    );
  }
}

8. ⚠️ ข้อผิดพลาดที่เจอบ่อย (พร้อมวิธีอ่านและวิธีแก้)

① commit แล้วแต่ไฟล์ไม่ไปด้วย

อาการ: git commit บอก "nothing to commit" หรือ commit แล้วแต่ GitHub ไม่เห็นไฟล์ใหม่
สาเหตุ: ลืมขั้น git add (ลืมยกของขึ้นโต๊ะก่อนถ่ายรูป)
วิธีแก้: git status ดูก่อน — ไฟล์สีแดง = ยังไม่ add → git add . → commit ใหม่

② flutter pub get ล้มเหลว: "version solving failed"

อาการ: เพิ่ม package แล้วขึ้น error ยาวๆ มีคำว่า "because ... depends on ..."
สาเหตุ: package ที่ขอ เข้ากันไม่ได้กับเวอร์ชัน Flutter/Dart ของเครื่อง หรือกับ package อื่น
วิธีแก้: อ่านบรรทัดแรกของ error หา "ชื่อ package คู่กรณี" → ไปดูหน้า package นั้นบน pub.dev แท็บ Versions เลือกเวอร์ชันที่รองรับ SDK ของเรา (นี่คือทักษะอ่าน error จากบทที่ 1 มาใช้จริงครั้งแรก!)

③ แก้ pubspec.yaml แล้วแอปพังทันที

อาการ: ขึ้น "Error on line 15: expected ..." ทั้งที่แค่เพิ่มบรรทัดเดียว
สาเหตุ: ไฟล์ .yaml เข้มงวดเรื่อง "ย่อหน้า" มาก — เว้นวรรคผิด 1 ช่องก็พัง (ห้ามใช้ Tab!)
วิธีแก้: ดูบรรทัดที่ error บอก เทียบย่อหน้ากับบรรทัดข้างเคียงให้ตรงกันเป๊ะ · เคล็ดลับ: ใช้ flutter pub add ชื่อpackage แทนการพิมพ์เองในไฟล์ จะไม่พลาดเลย

9. 🏋️ แบบฝึกหัดท้ายบท (ลงมือทำจริง)

🥉 ระดับ 1 — สำรวจ

เปิด git log --oneline ดูประวัติ commit ของ my_first_app แล้วสั่ง git status, แก้ข้อความ 1 จุดใน main.dart, ดู git diff ว่า Git เห็นการแก้ของเรายังไง แล้ว commit ด้วยข้อความมาตรฐาน conventional commits

🥈 ระดับ 2 — เพิ่มวัตถุดิบ

เพิ่ม package google_fonts ตามหัวข้อ 3 เปลี่ยนฟอนต์แอปเป็น Kanit (หรือฟอนต์ไทยตัวอื่นที่ชอบ) แล้ว commit ด้วยข้อความ feat: use Kanit font from google_fonts

🥇 ระดับ 3 — จัดบ้าน

แยก HomeScreen ออกจาก main.dart ไปไว้ที่ lib/screens/home_screen.dart ตามหัวข้อ 7 ให้แอปยังรันได้เหมือนเดิม และ flutter test ยังผ่าน แล้ว commit ด้วยข้อความ refactor: move HomeScreen to its own file — push ทั้งหมดขึ้น GitHub

10. ✅ เช็กลิสต์และสรุปบทเรียน

ก่อนไปบทต่อไป ตรวจว่าทำครบแล้ว:

  • อธิบายได้ว่า lib/, pubspec.yaml, android//ios/, .gitignore มีไว้ทำไม
  • เพิ่ม package จาก pub.dev และใช้งานได้จริง
  • ใช้ git status → add → commit ได้โดยไม่ต้องเปิดดูตัวอย่าง
  • อธิบายได้ว่าทำไมห้าม commit โฟลเดอร์ build/
  • แยกไฟล์หน้าจอออกจาก main.dart สำเร็จ (แบบฝึกหัดระดับ 3)
  • push ขึ้น GitHub แล้วเห็น commit ใหม่บนเว็บ

📖 สรุปศัพท์ใหม่บทนี้

ศัพท์ความหมายสั้นๆภาพจำ
pubspec.yamlไฟล์รายการของที่โปรเจกต์ใช้สมุดสั่งวัตถุดิบ
packageโค้ดสำเร็จรูปจากคนอื่นวัตถุดิบจากตลาด
dependencypackage ที่โปรเจกต์เรา "พึ่งพา"รายการในสมุดสั่งของ
repository (repo)โฟลเดอร์ที่ Git จำประวัติให้อัลบั้มรูปถาวร
commitจุดเซฟถาวร 1 จุด พร้อมคำอธิบายรูปถ่าย 1 ใบในอัลบั้ม
stage (git add)ขั้นเลือกของก่อนเซฟจัดคนเข้าเฟรมก่อนกดชัตเตอร์
.gitignoreรายการไฟล์ที่ Git ไม่ต้องจำของที่ไม่ต้องขนตอนย้ายบ้าน
🎓 สรุปบทนี้ใน 3 บรรทัด

โปรเจกต์ Flutter = ร้านอาหารที่เราทำงานในครัว (lib/) สั่งวัตถุดิบผ่านสมุด (pubspec.yaml) โดยไม่ต้องยุ่งกับระบบอาคาร (android/, ios/) · Git = จุดเซฟในเกมที่ทำให้กล้าทดลอง · แยกไฟล์ตั้งแต่โปรเจกต์ยังเล็ก = ของขวัญให้ตัวเองในอนาคต

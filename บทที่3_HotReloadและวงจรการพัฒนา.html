1. 🖼️ ภาพเล่าเรื่อง: ช่างปั้นดินเหนียว vs ช่างแกะสลักหิน

ลองนึกภาพช่างสองคนที่ต้องปั้นรูปหน้าคนให้เหมือนแบบ:

🗿 ช่างแกะสลักหิน (dev ยุคก่อน) 🏺 ช่างปั้นดินเหนียว (dev ยุค Flutter) สกัดผิดทีเดียว = เริ่มใหม่ทั้งก้อน บีบ ดู ปรับ บีบ ดู ปรับ... ต้องวางแผนเป๊ะก่อนลงมือ ลองผิดได้ไม่จำกัด แก้ได้ตลอด กว่าจะเห็นผลรวม = อีกหลายวัน เห็นผลทุก 3 วินาที แก้โค้ด ──► กดเซฟ ──► จอเปลี่ยนทันที ▲ │ └────── ดูผล + คิดต่อ ◄───┘ ← วงจรนี้คือ "Dev Loop" (วนรอบละไม่กี่วินาที)

Dev Loop คือจังหวะการทำงานที่คุณจะใช้ตลอดชีวิต dev: ยิ่งรอบสั้น ยิ่งเรียนรู้เร็ว ยิ่งกล้าทดลอง — Flutter ออกแบบมาให้รอบนี้สั้นระดับวินาทีด้วย Hot Reload

🤔 ทำไมต้องฝึก "จังหวะ" ไม่ใช่แค่ "คำสั่ง"

มือใหม่ส่วนใหญ่เขียนโค้ดยาวๆ แล้วค่อยรันทีเดียว (แบบช่างแกะหิน) พอพังก็หาไม่เจอว่าผิดตรงไหน มืออาชีพแก้ทีละนิดแล้วดูผลตลอด — บั๊กจึงถูกจับได้ตั้งแต่ยังตัวเล็ก

2. 🧠 สามระดับของการรีเฟรช — ใช้ตัวไหนเมื่อไหร่

คำสั่งทำอะไรใช้เมื่อเวลา
Hot Reload (กด r หรือแค่กดเซฟใน VS Code)ฉีดโค้ดใหม่เข้าแอปที่รันอยู่ คงสถานะเดิมแก้ UI, ข้อความ, สี, layout~1 วิ
Hot Restart (กด R)รันแอปใหม่ ล้างสถานะ แต่ไม่ build ใหม่แก้ main(), ตัวแปร global, สถานะเพี้ยน~5 วิ
Full Restart (หยุดแล้ว flutter run ใหม่)build ใหม่ทั้งหมดเพิ่ม package, แก้ไฟล์ native, เพิ่ม assets~1–2 นาที

วิธีจำแบบภาพ: Hot Reload = เปลี่ยนเสื้อผ้าตัวละครขณะละครกำลังเล่น · Hot Restart = เริ่มละครใหม่จากฉากแรก · Full Restart = สร้างโรงละครใหม่

🤔 "คงสถานะเดิม" สำคัญยังไง

สมมติคุณกำลังแก้หน้าที่ต้องกดเข้าไป 3 ชั้น (หน้าแรก → ตั้งค่า → โปรไฟล์) — Hot Reload แก้แล้ว ยังอยู่หน้าโปรไฟล์เหมือนเดิม ไม่ต้องกดเข้าไปใหม่ 3 ชั้นทุกรอบ นี่แหละที่ประหยัดเวลามหาศาล

3. 💻 ลงมือจริง: ฝึกจังหวะ Dev Loop 5 รอบ

เปิด my_first_app แล้วสั่ง flutter run จากนั้นทำตามนี้ ทีละรอบ อย่ารวบ — เป้าหมายคือฝึกจังหวะ ไม่ใช่ผลลัพธ์:

// รอบ 1: เปลี่ยนข้อความหลัก แล้วกดเซฟ → ดูจอเปลี่ยน
Text('ฝึก Dev Loop รอบแรก! 🔁', ...)

// รอบ 2: เปลี่ยนสีธีมทั้งแอปจุดเดียว แล้วเซฟ
theme: ThemeData(colorSchemeSeed: Colors.teal),   // จาก indigo → teal

// รอบ 3: ขยายไอคอน แล้วเซฟ
Icon(Icons.flutter_dash, size: 120, ...)          // จาก 80 → 120

// รอบ 4: ลองทำให้ "พัง" ดูบ้าง — ลบวงเล็บปิดตัวหนึ่งทิ้ง แล้วเซฟ
// → อ่าน error หาคำว่า "Expected ..." และเลขบรรทัด → แก้กลับ

// รอบ 5: เพิ่มบรรทัดนี้ไว้บนสุดของ build() แล้วเซฟ
debugPrint('build ทำงานแล้ว!');                    // ดูข้อความใน Debug Console

รอบ 5 คือการรู้จัก debugPrint — เครื่องมือ "ส่องความคิด" ของแอป: อยากรู้ว่าโค้ดบรรทัดไหนทำงานไหม ตัวแปรมีค่าอะไร ก็ print ออกมาดูใน Debug Console (แถบล่างของ VS Code) นี่คือเครื่องมือ debug ชิ้นแรกและถูกใช้บ่อยที่สุดในโลก

💡 ทำไมใช้ debugPrint ไม่ใช่ print

ทำหน้าที่เหมือนกัน แต่ debugPrint กันข้อความยาวถูกตัดหาย และถูกปิดอัตโนมัติในแอปเวอร์ชันจริงที่ส่งถึงมือผู้ใช้

4. 🔧 เทคนิคการประยุกต์ใช้

เทคนิคที่ 1 — "แก้จุดเดียวต่อรอบ" คือกฎเหล็ก

ถ้าแก้ 5 จุดพร้อมกันแล้วพัง จะไม่รู้ว่าจุดไหนคือตัวปัญหา แก้ทีละจุด → เซฟ → ดูผล → รู้ทันทีว่าอะไรทำให้เกิดอะไร (นี่คือหลักวิทยาศาสตร์: เปลี่ยนตัวแปรทีละตัว)

เทคนิคที่ 2 — ใช้ Widget Inspector ส่องหน้าจอ

ใน VS Code เปิด Command Palette (Ctrl+Shift+P) → พิมพ์ "Widget Inspector" → คลิกที่องค์ประกอบบนจอ จะเห็นเลยว่ามันคือ Widget ตัวไหนในโค้ด ซ้อนอยู่ใต้อะไร — เหมือนเอกซเรย์หน้าจอ มีประโยชน์มากตอน layout ไม่เป็นอย่างที่คิด

เทคนิคที่ 3 — จบ "หนึ่งเรื่อง" ให้ commit ทันที (จังหวะใหญ่)

Dev Loop คือจังหวะเล็ก (วินาที) ส่วน commit คือจังหวะใหญ่ (ครึ่งชั่วโมง–ชั่วโมง): ปรับสีเสร็จเป็นที่พอใจ = commit เพราะจุดเซฟที่ดีคือ "สถานะที่แอปทำงานได้" เสมอ

เทคนิคที่ 4 — debugPrint คือคำถาม ไม่ใช่ขยะ

พิมพ์ debugPrint เพื่อ "ถาม" แอปว่าเกิดอะไรขึ้น พอได้คำตอบแล้ว ลบทิ้ง ก่อน commit — โค้ดที่เต็มไปด้วย print เก่าๆ เหมือนกระดาษทดที่ไม่เคยเก็บโต๊ะ

5. 🏭 ตัวอย่างการเขียนจริงในงานจริง

🏭 สถานการณ์จริง: "ประชุมรีวิว UI วันศุกร์"

ทีมของคุณมีนัดรีวิวหน้าจอใหม่กับดีไซเนอร์และ Product Manager ทุกวันศุกร์ — dev ที่คล่อง Dev Loop จะเปิดแอปบนจอใหญ่ แล้วแก้ตามคำขอ สดๆ ในห้องประชุม

PM: "ปุ่มใหญ่ไปนิด ลดลงหน่อยได้ไหม" คุณ: แก้ size → เซฟ → จอเปลี่ยนใน 1 วิ ดีไซเนอร์: "สีเขียวนี้ไม่ใช่เขียวแบรนด์ ใช้ #2E7D32" คุณ: แก้สี → เซฟ → "แบบนี้ใช่ไหมครับ" ทีม: เคาะผ่านในที่ประชุม ✓

สิ่งที่เคยใช้เวลา "รับ feedback → กลับไปแก้ → นัดดูรอบใหม่สัปดาห์หน้า" จบใน 10 นาที — ความเร็วของ Dev Loop ไม่ใช่แค่เรื่องความสะดวกส่วนตัว แต่เปลี่ยนวิธีทำงานของทั้งทีม

6. 🧹 ตัวอย่าง Clean Code: จัดโค้ดให้ "แก้เร็ว" รับจังหวะ Dev Loop

Dev Loop จะเร็วได้ โค้ดต้อง "หาจุดแก้เจอเร็ว" ด้วย — เทียบสองแบบนี้:

❌ ค่ากระจัดกระจาย

อยากเปลี่ยนสีแบรนด์ต้องไล่หา 0xFF3F51B5 ให้ครบทุกจุด พลาด 1 จุด = สีเพี้ยนไม่รู้ตัว

✅ รวมค่าไว้ที่เดียว

ประกาศ _accent, _textSize ครั้งเดียวบนสุด — แก้จุดเดียว เปลี่ยนทั้งหน้า

// ❌ ค่าตัวเลข/สีกระจัดกระจาย — แก้ 1 เรื่องต้องไล่หา 5 ที่
body: Center(
  child: Column(children: [
    Icon(Icons.star, size: 40, color: Color(0xFF3F51B5)),
    Text('หัวข้อ', style: TextStyle(fontSize: 22, color: Color(0xFF3F51B5))),
    Text('รายละเอียด', style: TextStyle(fontSize: 22, color: Color(0xFF3F51B5))),
  ]),
),
// ✅ รวมค่าที่ใช้ซ้ำไว้ที่เดียว — แก้จุดเดียว เปลี่ยนทั้งหน้า
class HomeScreen extends StatelessWidget {
  const HomeScreen({super.key});

  // ค่าที่ใช้ซ้ำ ประกาศครั้งเดียวบนสุด — เห็นปุ๊บรู้ปั๊บว่าหน้านี้ใช้อะไรบ้าง
  static const Color _accent = Color(0xFF3F51B5);
  static const double _textSize = 22;

  @override
  Widget build(BuildContext context) {
    return Center(
      child: Column(children: const [
        Icon(Icons.star, size: 40, color: _accent),
        Text('หัวข้อ', style: TextStyle(fontSize: _textSize, color: _accent)),
        Text('รายละเอียด', style: TextStyle(fontSize: _textSize, color: _accent)),
      ]),
    );
  }
}
🧠 หลักทอง

โค้ดที่ clean ไม่ใช่แค่ "สวย" — มันคือโค้ดที่ ตอบสนอง Dev Loop ได้เร็ว: อยากแก้อะไร รู้ทันทีว่าไปแก้ตรงไหน จุดเดียวจบ

7. ⚠️ ข้อผิดพลาดที่เจอบ่อย

① กดเซฟแล้วจอไม่เปลี่ยน (Hot Reload ไม่ติด)

สาเหตุ: การแก้บางอย่าง Hot Reload ทำไม่ได้ เช่น แก้ใน main(), เปลี่ยนตัวแปร global, เปลี่ยน Stateless ↔ Stateful
วิธีแก้: กด R (Hot Restart) — ถ้ายังไม่เปลี่ยน หยุดแล้ว flutter run ใหม่ · จำง่ายๆ: "เซฟแล้วเฉย → R → ยังเฉย → รันใหม่"

② จอแดงทั้งจอ (Red Screen)

สาเหตุ: โค้ดมีข้อผิดพลาดตอนวาดหน้าจอ — จอแดงคือ Flutter บอกว่า "วาดต่อไม่ได้"
วิธีแก้: อย่าตกใจ อ่านบรรทัดแรก (มักบอกตรงๆ เช่น "RenderFlex overflowed") แก้แล้วเซฟ จอแดงหายเอง — จอแดงเกิดเฉพาะตอนพัฒนา ผู้ใช้จริงไม่เห็นแบบนี้

③ "Waiting for another flutter command to release the startup lock"

สาเหตุ: มีคำสั่ง flutter อีกตัวค้างอยู่ (มักเกิดจากปิด terminal กลางคัน) — ตัวเก่าล็อกไฟล์ (lock file) กันคำสั่งชนกันเอง
วิธีแก้: ปิด terminal/VS Code ทุกหน้าต่างแล้วเปิดใหม่ ถ้ายังไม่หายให้ restart เครื่อง

8. 🏋️ แบบฝึกหัดท้ายบท (ลงมือทำจริง)

🥉 ระดับ 1 — ฝึกจังหวะ

ทำ Dev Loop 5 รอบตามหัวข้อ 3 ให้ครบ รวมถึงรอบที่ตั้งใจทำพังแล้วอ่าน error เอง จับเวลาดูว่าแต่ละรอบใช้กี่วินาที

🥈 ระดับ 2 — เอกซเรย์จอ

เปิด Widget Inspector แล้วคลิกไล่ดูทุกองค์ประกอบบนหน้าจอ Hello World — วาดผังลงกระดาษว่า Widget อะไรซ้อนอยู่ใต้อะไร แล้วเทียบกับโค้ดจริงว่าตรงไหม

🥇 ระดับ 3 — Speed run แบบมืออาชีพ

ตั้งโจทย์เอง 3 การแก้ (เช่น เปลี่ยนธีม, เพิ่มไอคอนตัวที่สอง, แก้ข้อความ) ทำให้ครบภายใน 10 นาที โดยแต่ละเรื่อง commit แยกกันด้วยข้อความ conventional commits ที่ถูกต้อง — จบแล้ว git log --oneline ต้องเล่าเรื่องได้ว่าทำอะไรไปบ้าง

9. ✅ เช็กลิสต์และสรุปบทเรียน

  • อธิบายความต่างของ Hot Reload / Hot Restart / Full Restart พร้อมตัวอย่างว่าใช้เมื่อไหร่
  • ใช้ debugPrint ดูค่าใน Debug Console เป็น
  • เปิด Widget Inspector และชี้ได้ว่าองค์ประกอบบนจอคือ Widget ตัวไหน
  • เจอจอแดงแล้วไม่ตกใจ อ่านบรรทัดแรกเป็น
  • ทำ speed run ระดับ 3 สำเร็จ พร้อมประวัติ commit สวยงาม

เริ่มต้น Flutter และรันแอปแรกของคุณ

เข้าใจว่า Flutter ทำงานอย่างไร ติดตั้งเครื่องมือสำเร็จ และรันแอป Hello World บนมือถือได้จริง พร้อมหลักคิดของ developer มืออาชีพตั้งแต่บรรทัดแรก

⏱️ ใช้เวลา ~2–3 ชั่วโมง 📊 ระดับ: เริ่มต้นจากศูนย์ 🛠️ Flutter 3.44+ / Dart 3.12+

🎯 ภาพรวมบทเรียน (อ่านก่อนเริ่ม)

บทนี้คือ "ด่านแรก" ที่สำคัญที่สุด เพราะเป็นจุดที่คนเลิกเรียนเยอะที่สุด เป้าหมายของเราคือพาคุณข้ามกำแพงการติดตั้งให้ได้ และทำให้คุณเห็นแอปของตัวเองทำงานบนมือถือจริง ซึ่งจะสร้างแรงบันดาลใจให้เรียนต่อ

🧭 สิ่งที่คุณจะทำได้เมื่อจบบทนี้
  • อธิบายได้ว่า Flutter คืออะไร และต่างจากการเขียนแอปแบบเดิมอย่างไร
  • ติดตั้ง Flutter SDK + editor และผ่าน flutter doctor
  • สร้าง รัน และแก้ไขแอป Hello World ของตัวเอง
  • เข้าใจคอนเซปต์หัวใจ "Everything is a Widget"
  • เก็บผลงานชิ้นแรกขึ้น GitHub แบบมืออาชีพ
💡 วิธีเรียนให้ได้ผล

อย่าแค่อ่าน — ลงมือพิมพ์โค้ดเองทุกตัว แม้จะรู้สึกว่าก็อปวางเร็วกว่า การพิมพ์เองทำให้สมองจดจำ pattern และเจอ error เล็กๆ ที่สอนเราได้มากที่สุด นักพัฒนาเก่งทุกคนผ่านขั้นตอนนี้มาแล้ว

1. Flutter คืออะไร? (มองให้เห็นภาพก่อน)

ลองนึกถึงการเปิดร้านอาหารที่อยากขายทั้ง หน้าร้าน และ เดลิเวอรี พร้อมกัน

  • วิธีเดิม (Native): ต้องจ้างเชฟ 2 คน — คนหนึ่งทำอาหารสำหรับหน้าร้าน (iOS, ภาษา Swift) อีกคนทำสำหรับเดลิเวอรี (Android, ภาษา Kotlin) → แพง ช้า เมนูต้องอัปเดต 2 ที่ และมีโอกาสที่อาหารสองช่องทางหน้าตาไม่เหมือนกัน
  • วิธี Flutter: เชฟคนเดียว ครัวกลางครัวเดียว แล้วเสิร์ฟได้ทั้ง 2 ช่องทาง → เขียนโค้ดครั้งเดียว ออกได้ทั้ง iOS และ Android (และ Web/Desktop)
โค้ด Flutter (เขียนครั้งเดียว) │ ┌─────────────┼─────────────┐ ▼ ▼ ▼ 📱 iOS 🤖 Android 🌐 Web (iPhone) (Samsung...) (เบราว์เซอร์)

คีย์เวิร์ดที่ต้องจำ:

Cross-platform = ข้ามแพลตฟอร์ม Dart = ภาษาที่ใช้ Widget = ทุกชิ้นบนจอ SDK = ชุดเครื่องมือพัฒนา

🤔 ทำไม Flutter ถึงเร็วเทียบเท่าแอป Native?

ต่างจากเครื่องมือ cross-platform รุ่นเก่า (ที่ยัดเว็บไว้ในแอป) Flutter คอมไพล์โค้ด Dart เป็นภาษาเครื่อง (native machine code) ของแต่ละแพลตฟอร์มโดยตรง และ Flutter วาดทุก pixel เองด้วย engine ของตัวเอง (Skia/Impeller) จึงควบคุมหน้าตาและความลื่นได้เต็มที่ นี่คือเหตุผลทางเทคนิคที่ทำให้แอป Flutter รู้สึก "ลื่นเหมือน native"

ทำไมบริษัทถึงเลือก Flutter (ความสำคัญในโลกจริง)

ในมุมธุรกิจ Flutter ตอบโจทย์ 3 อย่างที่บริษัทแคร์ที่สุด:

มุมมองประโยชน์จริง
💰 ต้นทุนทีมเดียว โค้ดเดียว แทนที่จะมีทีม iOS + ทีม Android แยกกัน ลดคนและเวลาได้ราว 40–50%
⚡ ความเร็วออกฟีเจอร์ใหม่พร้อมกันทั้งสองระบบ ไม่ต้องรอทีมใดทีมหนึ่ง
🎨 ความสม่ำเสมอแบรนด์ดูเหมือนกันเป๊ะทุกเครื่อง ไม่ต้องดีไซน์ซ้ำ 2 รอบ

แอปที่ใช้ Flutter จริงในระดับโลก เช่น Google Pay, Alibaba (บางส่วน), BMW, eBay Motors และแอปธนาคาร/อีคอมเมิร์ซจำนวนมากในไทย ทำให้ทักษะนี้มี ตลาดงานรองรับจริง

🏭 มุมมองในโปรดักชันจริง

เวลาทำงานจริง คุณมักได้ยินคำว่า "write once, run anywhere" แต่ความจริงคือ "write once, test everywhere" — โค้ดส่วนใหญ่ใช้ร่วมกันได้ก็จริง แต่ยังต้องทดสอบบนทั้ง iOS และ Android เพราะพฤติกรรมเล็กๆ (คีย์บอร์ด, permission, การแจ้งเตือน) ต่างกัน อย่าหลงเชื่อว่าทำครั้งเดียวแล้วจบ 100%

2. คอนเซปต์หัวใจ: "Everything is a Widget"

นี่คือแนวคิดหลักที่ต้องฝังในหัวตั้งแต่วันแรก — ทุกอย่างที่คุณเห็นใน Flutter คือ Widget ปุ่มคือ Widget, ข้อความคือ Widget, ช่องว่างคือ Widget, แม้แต่การจัดวาง (ตรงกลาง/ซ้าย/ขวา) ก็เป็น Widget

🧠 ทำไมต้องเข้าใจตรงนี้ให้ลึก

ถ้าเข้าใจว่าทุกอย่างคือ Widget ที่นำมา ประกอบและซ้อนกัน (compose) ได้เหมือนตัวต่อเลโก้ คุณจะสร้างหน้าจอแบบไหนก็ได้ในอนาคต โดยไม่ต้องท่องจำ เพราะมันคือหลักการเดียวที่ใช้ซ้ำตลอดทั้งเฟรมเวิร์ก — เข้าใจหลักการเดียว แทนการจำร้อยกรณี

โครงสร้างของหน้าจอจึงเป็น "ต้นไม้ของ Widget" (Widget Tree) — Widget ซ้อนอยู่ใน Widget:

MaterialApp ← ตัวห่อทั้งแอป (ธีม, การนำทาง) └── Scaffold ← โครงหน้าจอ 1 หน้า ├── AppBar ← แถบบน │ └── Text ← "ชื่อแอป" └── Center ← จัดกึ่งกลาง └── Text ← "Hello World" ◀ Widget ในสุด

แนวคิดเบื้องหลัง: Declarative UI (สำคัญมาก)

Flutter ใช้แนวคิด Declarative ซึ่งต่างจากวิธีเดิม (Imperative) ที่มือใหม่หลายคนคุ้นเคย ลองเทียบ:

❌ Imperative (วิธีเดิม)

"ไปหาปุ่มนั้น แล้วเปลี่ยนสีมัน แล้วเปลี่ยนข้อความมัน" — เราสั่งทีละขั้นว่าจะ เปลี่ยน อย่างไร พอ UI ซับซ้อนจะพันกันยุ่ง

✅ Declarative (Flutter)

"หน้าจอควรหน้าตาแบบนี้ ตามข้อมูลปัจจุบัน" — เราบอกแค่ ผลลัพธ์ที่ต้องการ แล้ว Flutter จัดการอัปเดตเอง โค้ดคาดเดาง่ายและบั๊กน้อยกว่า

🤔 ทำไม Declarative ถึงดีกว่าในระยะยาว

หัวใจคือสมการ UI = f(state) — "หน้าตา = ฟังก์ชันของข้อมูล" เมื่อข้อมูลเปลี่ยน Flutter จะวาดหน้าจอใหม่ให้ตรงกับข้อมูลเสมอ เราไม่ต้องไล่แก้ทีละจุดเอง ทำให้โค้ดที่มี state ซับซ้อน (เช่น ตะกร้าสินค้า, ฟอร์มยาวๆ) จัดการได้ง่ายขึ้นมาก คุณจะเห็นพลังของสิ่งนี้ชัดในเฟส 3–4

3. ติดตั้งเครื่องมือ (Setup)

⚠️ หลักคิดสำคัญก่อนเริ่ม

ขั้นตอนติดตั้งคือด่านที่คนเลิกเยอะที่สุด ใจเย็นๆ ทำตามทีละขั้น ถ้าติด error ให้ คัดลอกข้อความ error ไปค้นหาใน Google — นี่ไม่ใช่การโกง แต่คือ "ทักษะ dev จริง" ที่มืออาชีพใช้ทุกวัน ไม่มีใครจำทุกอย่างได้

3.1 ติดตั้ง Flutter SDK

  1. ไปที่หน้า docs.flutter.dev/get-started/install
  2. เลือกระบบปฏิบัติการ (Windows / macOS / Linux) แล้วทำตามทีละขั้น
  3. ติดตั้ง Git ด้วย (Flutter ใช้ Git ทำงานเบื้องหลัง — ดาวน์โหลด/อัปเดต)
ระบบสิ่งที่ต้องลงเพิ่มใช้ build อะไรได้
WindowsAndroid Studio (สำหรับ Android SDK + emulator)Android, Web
macOSAndroid Studio + XcodeAndroid, iOS, Web (ครบที่สุด)
LinuxAndroid StudioAndroid, Web
💡 ยังไม่มี Mac ก็เริ่มได้

การ เขียนโค้ด Flutter ทำได้บนทุกระบบ ติดแค่ตอน build เป็นไฟล์ iOS ที่ต้องใช้ Mac (เฉพาะเฟส 5) ระหว่างเรียนทั้งหมดนี้ ใช้ Android emulator หรือรันบนเบราว์เซอร์ไปก่อนได้สบายมาก อย่าให้เรื่องนี้มาหยุดคุณ

3.2 ติดตั้ง Editor (โปรแกรมเขียนโค้ด)

แนะนำ VS Code (ฟรี เบา นิยมที่สุดสำหรับมือใหม่):

  1. โหลดจาก code.visualstudio.com
  2. ติดตั้ง Extension ชื่อ "Flutter" (มันจะลาก Dart extension มาให้อัตโนมัติ)
🏭 เกร็ดจากการทำงานจริง

ทีมส่วนใหญ่ใช้ VS Code หรือ Android Studio ก็ได้ — เลือกอันที่ถนัด แต่สิ่งที่ทุกทีมบังคับเหมือนกันคือ extension ที่ format โค้ดอัตโนมัติเมื่อกดบันทึก (ตั้งค่า "editor.formatOnSave": true) เพราะทำให้โค้ดทั้งทีมหน้าตาเหมือนกัน ลดการทะเลาะเรื่องสไตล์ใน code review — ตั้งค่านี้ตั้งแต่วันแรกเลย

3.3 ตรวจสอบความพร้อมด้วย flutter doctor

เปิด Terminal แล้วพิมพ์:

flutter doctor
🤔 flutter doctor คืออะไร และทำไมต้องใช้

มันคือ "หมอตรวจสุขภาพ" ของระบบ — บอกว่าเครื่องมือตัวไหนพร้อม (✓) ตัวไหนยังขาด (✗) พร้อมบอกวิธีแก้ ใช้คำสั่งนี้ทุกครั้งที่ setup มีปัญหา มันคือเครื่องมือ debug ตัวแรกที่ dev Flutter ทุกคนพึ่งพา เพิ่ม flutter doctor -v เพื่อดูรายละเอียดเต็ม

ผลลัพธ์ที่ดีจะมีติ๊กถูกครบ เช่น:

[✓] Flutter (Channel stable, 3.44.0)
[✓] Android toolchain - develop for Android devices
[✓] Xcode - develop for iOS and macOS   (เฉพาะ Mac)
[✓] VS Code
[✓] Connected device (1 available)
[✓] Network resources
💡 ถ้าเจอ ✗ ไม่ต้องตกใจ

บรรทัดที่ขึ้น ✗ จะมีข้อความบอกว่าต้องทำอะไร เช่น "Run flutter doctor --android-licenses" ให้ทำตามทีละบรรทัดจนเขียวครบ ขอแค่ Flutter เขียว + มี device 1 ตัว ก็เริ่มบทนี้ได้แล้ว (Xcode ค่อยจัดการตอนเฟส 5)

4. สร้างและรันแอปแรก

4.1 สร้างโปรเจกต์

เปิด Terminal ในโฟลเดอร์ที่ต้องการ แล้วพิมพ์:

flutter create my_first_app
cd my_first_app
flutter run
🤔 ทำไมต้องใช้ flutter create แทนการสร้างไฟล์เอง

คำสั่งนี้สร้าง "โครงสร้างมาตรฐาน" ให้ครบ — โฟลเดอร์สำหรับ iOS, Android, Web, ไฟล์ตั้งค่า, และตัวอย่างโค้ด ถ้าสร้างเองทีละไฟล์จะขาดไฟล์สำคัญ (เช่นไฟล์ตั้งค่าแพลตฟอร์ม) และรันไม่ได้ นี่คือมาตรฐานที่ทีมทั่วโลกใช้ และทำให้โปรเจกต์ของคุณ "คุยกัน" กับ dev คนอื่นได้ทันที

4.2 เขียน Hello World เวอร์ชันของเราเอง

เปิดไฟล์ lib/main.dart แล้วแทนที่ทั้งหมดด้วยโค้ดนี้:

// main.dart — แอป Hello World แบบเข้าใจง่าย
import 'package:flutter/material.dart';

// จุดเริ่มต้นของทุกแอป Flutter — เปรียบเหมือน "สวิตช์เปิดเครื่อง"
void main() {
  runApp(const MyApp());
}

// MyApp คือ Widget ตัวนอกสุด (root) ของทั้งแอป
class MyApp extends StatelessWidget {
  const MyApp({super.key});

  @override
  Widget build(BuildContext context) {
    return MaterialApp(
      title: 'แอปแรกของฉัน',
      home: Scaffold(
        // AppBar = แถบหัวด้านบน
        appBar: AppBar(
          title: const Text('แอปแรกของฉัน'),
        ),
        // Center = จัดเนื้อหาให้อยู่กึ่งกลางจอ
        body: const Center(
          child: Text(
            'สวัสดีชาวโลก! 🌏',
            style: TextStyle(fontSize: 24),
          ),
        ),
      ),
    );
  }
}

4.3 อธิบายโค้ดทีละบรรทัด (ทำไมแต่ละบรรทัดถึงจำเป็น)

โค้ดทำหน้าที่อะไรทำไมต้องมี
import '...material.dart';ดึงชุดเครื่องมือ UI สำเร็จรูปมาใช้ถ้าไม่ import จะไม่รู้จัก Text, Scaffold ฯลฯ — โปรแกรมไม่รู้ว่าของพวกนี้คืออะไร
void main()จุดเริ่มทำงานของแอปทุกแอปต้องมีจุดเริ่ม Flutter จะมองหาฟังก์ชันชื่อ main ก่อนเสมอ
runApp(...)สั่งให้แอปเริ่มแสดงผลเป็นคำสั่งที่ "เปิดเครื่อง" จริงๆ — รับ Widget ตัวแรกไปแสดง
constบอกว่าค่านี้ไม่เปลี่ยนช่วยให้ Flutter ไม่ต้องสร้าง object ใหม่ทุกครั้ง → แอปเร็วขึ้น (ดูหัวข้อ clean code)
extends StatelessWidgetสร้าง Widget ที่หน้าตาไม่เปลี่ยนเหมาะกับหน้าจอที่นิ่ง — บทหน้าจะเจอ StatefulWidget ที่เปลี่ยนได้
build(...)วาดหน้าจอออกมาFlutter เรียกฟังก์ชันนี้เพื่อรู้ว่าจะแสดงอะไร — หัวใจของทุก Widget
Scaffoldโครงหน้าจอมาตรฐานให้โครง AppBar + body + ปุ่มลอย ฯลฯ มาให้พร้อม ไม่ต้องสร้างเอง
🧠 หลักคิด dev: อ่านโค้ด "จากนอกเข้าใน"

เวลาเจอโค้ดที่ซ้อนกันหลายชั้นแบบนี้ อย่าอ่านทีละบรรทัดจากบนลงล่าง แต่ให้มองโครงใหญ่ก่อน: MaterialApp → Scaffold → (AppBar + body) แล้วค่อยเจาะเข้าไปแต่ละชั้น เหมือนดูแผนที่ก่อนดูตรอกซอย ทักษะ "อ่านโครงสร้างก่อนรายละเอียด" นี้จะช่วยคุณตลอดอาชีพ

5. Hot Reload — อาวุธลับของ Flutter

ขณะที่แอปกำลังรันอยู่ ลองแก้ข้อความจาก 'สวัสดีชาวโลก! 🌏' เป็น 'ผมเขียนแอปได้แล้ว! 🎉' แล้วกด บันทึก (Ctrl+S / Cmd+S) → หน้าจอจะเปลี่ยน ทันที โดยไม่ต้องปิด-เปิดแอปใหม่

🤔 ทำไม Hot Reload ถึงเปลี่ยนชีวิต dev

สมัยก่อนแก้โค้ด 1 บรรทัดต้อง build ใหม่ รอ 1–2 นาที วันหนึ่งเสียเวลาเป็นชั่วโมงไปกับการรอ Flutter ทำให้เห็นผลใน เสี้ยววินาที โดย คงสถานะแอปไว้ (เช่น ยังอยู่หน้าเดิม ข้อมูลในฟอร์มยังอยู่) ทำให้ลองผิดลองถูกได้เร็ว = เรียนรู้เร็ว นี่คือเหตุผลอันดับหนึ่งที่คนรัก Flutter

คำสั่งทำอะไรใช้เมื่อไหร่
Hot Reload (กด r)อัปเดต UI ทันที คงสถานะเดิมแก้หน้าตา/ข้อความ/สี (ใช้บ่อยสุด)
Hot Restart (กด R)รีเซ็ตแอปใหม่ทั้งหมด ล้างสถานะแก้ตัวแปร global, โครงสร้างหลัก หรือ Hot Reload ไม่ติด
Full Restart (flutter run ใหม่)build ใหม่หมดเพิ่ม package ใหม่ใน pubspec, แก้ไฟล์ native
🏭 ตัวอย่างใช้งานจริงในที่ทำงาน

ตอนประชุมกับดีไซเนอร์ เขาบอก "ขยับปุ่มลงนิด เปลี่ยนสีเป็นน้ำเงิน เพิ่มระยะห่าง" คุณแก้แล้วทุกคนเห็นผลทันทีบนจอในห้องประชุม — ตัดสินใจร่วมกันได้เลยใน 5 นาที แทนที่จะนัดรอบใหม่พรุ่งนี้ ความเร็วระดับนี้เปลี่ยนวิธีทำงานของทั้งทีม

6. หลักคิดของ developer (Developer Mindset)

นี่คือหัวข้อที่ตำราส่วนใหญ่ข้าม แต่สำคัญที่สุด เพราะ คอนเซปต์อยู่กับคุณตลอดชีวิต ส่วน syntax ลืมแล้วเปิดดูได้ ถ้าไม่เข้าใจวิธีคิด จะไปเขียนเองในอนาคตยากมาก

6.1 Error message คือเพื่อน ไม่ใช่ศัตรู

มือใหม่เห็นข้อความสีแดงแล้วตกใจ แต่มืออาชีพ อ่านมันอย่างใจเย็น เพราะ error ส่วนใหญ่บอกตรงๆ ว่าผิดตรงไหน บรรทัดไหน เทคนิคการอ่าน error:

  1. อ่าน บรรทัดแรก — มักบอกชนิดของปัญหา (เช่น "The method 'foo' isn't defined")
  2. หา ชื่อไฟล์ + เลขบรรทัด — บอกตำแหน่งที่เกิดเหตุ
  3. ถ้าไม่เข้าใจ → คัดลอกบรรทัดแรกไปค้นหา (เติมคำว่า "flutter" ต่อท้าย)
🧠 หลักคิด: Rubber Duck Debugging

เทคนิคคลาสสิกของ dev — เวลาติดปัญหา ให้ อธิบายโค้ดของคุณออกมาดังๆ ทีละบรรทัด (จะพูดกับเป็ดยางบนโต๊ะก็ได้) บ่อยครั้งคุณจะ "เห็น" จุดผิดเองระหว่างอธิบาย เพราะการเรียบเรียงเป็นคำพูดบังคับให้สมองคิดเป็นลำดับ

6.2 เขียนทีละนิด ทดสอบบ่อยๆ

อย่าเขียนยาว 100 บรรทัดแล้วค่อยกดรัน เพราะถ้าพังจะหาไม่เจอว่าผิดตรงไหน ให้เขียน 5–10 บรรทัด → รัน → เห็นว่าทำงาน → เขียนต่อ วิธีนี้เรียกว่า "incremental development" ทำให้ขอบเขตของบั๊กเล็กเสมอ

6.3 เข้าใจ > จำ

โค้ดและเฟรมเวิร์กเปลี่ยนทุกปี (Flutter ออกเวอร์ชันใหม่ตลอด) แต่หลักการอย่าง "ทุกอย่างคือ Widget", "UI = f(state)", "แยกหน้าที่ให้ชัด" อยู่กับเราตลอด ลงทุนเวลาเข้าใจหลักการ คุ้มกว่าท่องจำ syntax

💡 4 นิสัยที่ควรสร้างตั้งแต่วันนี้
  • อ่าน error ก่อนถาม/ค้นหาทุกครั้ง
  • commit งานขึ้น Git ทุกวันที่เขียนโค้ด
  • ตั้งชื่อตัวแปร/ฟังก์ชันให้คนอื่นอ่านเข้าใจ
  • เมื่อโค้ดทำงานแล้ว ใช้เวลาอีก 2 นาทีจัดให้สะอาด (refactor)

7. Clean Code vs โค้ดในโปรดักชันจริง

หลายคนสับสนว่า "clean code" กับ "โค้ดที่ใช้งานได้" คือสิ่งเดียวกัน — ไม่ใช่ โค้ดที่ รันได้ กับโค้ดที่ ทีมดูแลต่อได้ ต่างกันมาก ส่วนนี้จะให้เห็นภาพชัด

7.1 ตัวอย่างเดียวกัน เขียน 2 แบบ

❌ เขียนแบบรีบๆ (ใช้ได้ แต่ดูแลยาก)
✅ เขียนแบบ clean (มืออาชีพ)
// ❌ แบบรีบๆ — รันได้ แต่อ่านยาก
class a extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return Scaffold(body: Center(child: Text("hello",
      style: TextStyle(fontSize: 24, color: Color(0xFF3F51B5)))));
  }
}
// ✅ แบบ clean — ตั้งชื่อสื่อความหมาย, แยกค่าคงที่, ใช้ const
class WelcomeScreen extends StatelessWidget {
  const WelcomeScreen({super.key});

  // แยกค่าคงที่ออกมา -> แก้ที่เดียว มีผลทั้งไฟล์ (หลัก DRY)
  static const double _titleSize = 24;

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      body: Center(
        child: Text(
          'hello',
          // ใช้สีจากธีม แทนการ hardcode เลขสี
          style: TextStyle(
            fontSize: _titleSize,
            color: Theme.of(context).colorScheme.primary,
          ),
        ),
      ),
    );
  }
}
ประเด็น❌ แบบรีบ✅ แบบ clean
ชื่อคลาสa — ไม่รู้ว่าคืออะไรWelcomeScreen — รู้ทันที
สีhardcode 0xFF3F51B5ดึงจากธีม เปลี่ยนทีเดียวทั้งแอป
ค่าตัวเลขปนอยู่ในโค้ด (magic number)ตั้งชื่อค่าคงที่ชัดเจน
constไม่มี → แอปทำงานหนักขึ้นมี → ประสิทธิภาพดีกว่า
🏭 ความจริงในโปรดักชัน: "ดีพอ" กับ "สมบูรณ์แบบ"

ในงานจริงมีคำว่า deadline บางครั้งคุณต้องเขียนให้ "ใช้ได้ก่อน" แล้วค่อยกลับมาทำให้สะอาด (เรียกว่า technical debt — หนี้ทางเทคนิค) สิ่งสำคัญคือ รู้ตัวว่ากำลังสร้างหนี้ และจดไว้ (เช่นเขียน // TODO: refactor นี้ทีหลัง) ทีมที่ดีจะมีเวลากลับมาจ่ายหนี้ ส่วนทีมที่แย่จะปล่อยจนโค้ดพังทั้งระบบ — เป้าหมายคือสมดุล ไม่ใช่ perfect ทุกบรรทัด

7.2 กฎทองที่ทีมระดับโลกใช้

  • ตั้งชื่อให้อ่านแล้วเข้าใจ — โค้ดถูกอ่านมากกว่าถูกเขียน 10 เท่า
  • หนึ่งฟังก์ชัน หนึ่งหน้าที่ (Single Responsibility) — ถ้าอธิบายฟังก์ชันต้องใช้คำว่า "และ" หลายครั้ง แปลว่าควรแยก
  • comment บอก "ทำไม" ไม่ใช่ "ทำอะไร" — โค้ดบอกอยู่แล้วว่าทำอะไร
  • DRY (Don't Repeat Yourself) — ถ้าก็อปโค้ดเดิมเกิน 2 ครั้ง ให้แยกเป็นฟังก์ชัน/Widget
🧠 หลักทองที่ควรท่อง

"เขียนโค้ดเหมือนคนที่จะมาอ่านต่อ คือตัวคุณเองในอีก 6 เดือนข้างหน้า ที่ลืมหมดแล้วว่าเขียนอะไรไว้" — เขียนให้อ่านง่ายที่สุดเข้าไว้ คุณในอนาคตจะขอบคุณ

8. เก็บผลงานขึ้น GitHub (ทำให้เป็นผลงานจริง)

ทำให้แอปแรกเป็นผลงานมาตรฐานที่โชว์ได้ — รันคำสั่งเหล่านี้ในโฟลเดอร์โปรเจกต์:

# 1. เข้าโฟลเดอร์โปรเจกต์
cd my_first_app

# 2. เริ่ม track ด้วย git (flutter create สร้าง .gitignore ให้แล้ว)
git init
git add .
git commit -m "feat: create first Flutter Hello World app"

# 3. สร้าง repo บน GitHub แล้วเชื่อมต่อ
git remote add origin https://github.com/<ชื่อคุณ>/my_first_app.git
git branch -M main
git push -u origin main
🤔 ทำไมต้อง commit ตั้งแต่บทแรก

GitHub คือ "เรซูเม่ที่มีชีวิต" ของ dev — บริษัทดูประวัติ commit เพื่อเห็นว่าคุณทำงานสม่ำเสมอแค่ไหน และดูคุณภาพโค้ดจริง การเริ่มนิสัยนี้ตั้งแต่วันแรกทำให้ 6 เดือนต่อมาคุณมี portfolio เต็มไปด้วยผลงาน โดยไม่ต้องมานั่งทำย้อนหลัง

💡 มาตรฐาน Conventional Commits

การขึ้นต้น commit ด้วยคำเฉพาะเป็นมาตรฐานสากลที่ทีมมืออาชีพใช้ ทำให้ประวัติอ่านง่ายและดูเป็นมือโปร:

  • feat: = เพิ่มฟีเจอร์ใหม่
  • fix: = แก้บั๊ก
  • docs: = แก้เอกสาร
  • refactor: = ปรับโค้ดให้ดีขึ้นโดยพฤติกรรมเหมือนเดิม
🏭 README สำคัญกว่าที่คิด

ในงานจริง สิ่งแรกที่คนเปิดดูใน repo คือ README ไม่ใช่โค้ด README ที่ดีบอก: แอปทำอะไร, รันยังไง, โครงสร้างโปรเจกต์ — โปรเจกต์ my_first_app ที่แนบมามี README มาตรฐานพร้อม badge ให้แล้ว ลองเปิดดูเป็นแม่แบบ

9. แบบฝึกหัดท้ายบท (ลงมือทำจริง)

ความรู้จะติดตัวก็ต่อเมื่อได้ลองทำเอง ลองทำ 3 ระดับนี้:

🥉 ระดับ 1 — ปรับแต่ง

เปลี่ยนข้อความ, สีของ AppBar, และขนาดตัวอักษร แล้วใช้ Hot Reload ดูผล

🥈 ระดับ 2 — เพิ่มองค์ประกอบ

เพิ่ม Icon และข้อความบรรทัดที่สอง โดยใช้ Column ห่อไว้ (ดูเฉลยได้จากไฟล์ main.dart ในโปรเจกต์ตัวอย่าง)

🥇 ระดับ 3 — ท้าทาย

สร้างหน้า "นามบัตรดิจิทัล" ของตัวเอง: มีไอคอนรูปคน, ชื่อ, ตำแหน่ง, และอีเมล จัดให้กึ่งกลางสวยงาม แล้ว push ขึ้น GitHub เป็น repo ที่สอง

💡 ทำเสร็จแล้ว

commit ด้วยข้อความ feat: add personal business card screen — นี่คือผลงานชิ้นที่ 2 ใน portfolio ของคุณแล้ว

10. เช็กลิสต์และสรุปบทเรียน

ก่อนไปบทต่อไป ตรวจว่าทำครบแล้ว:

  • flutter doctor ขึ้น ✓ ครบ (อย่างน้อย Flutter + 1 device)
  • รันแอป Hello World บนมือถือ/emulator ได้
  • ลองแก้ข้อความแล้วเห็น Hot Reload ทำงาน
  • เข้าใจคอนเซปต์ "Everything is a Widget" และ "UI = f(state)"
  • อ่าน error เป็น และรู้จัก Rubber Duck Debugging
  • แยกความต่างระหว่างโค้ดแบบรีบ กับแบบ clean ได้
  • push โปรเจกต์ขึ้น GitHub ด้วย conventional commit สำเร็จ
🎓 สรุปบทนี้ใน 3 บรรทัด

Flutter = เขียนครั้งเดียวออกหลายระบบ เพราะคอมไพล์เป็น native และวาดทุก pixel เอง · ทุกอย่างคือ Widget ที่ประกอบกันแบบ declarative (UI = f(state)) · เริ่มสร้างนิสัยมืออาชีพตั้งแต่วันแรก: อ่าน error เป็น, เขียน clean, commit ทุกวัน

1. 🖼️ ภาพเล่าเรื่อง: แม่พิมพ์ขนม

ร้านขนมที่ต้องทำคุกกี้วันละพันชิ้น ไม่มีใครปั้นทีละชิ้นด้วยมือ — เขาสร้าง แม่พิมพ์ ครั้งเดียว แล้วกดออกมากี่ชิ้นก็ได้:

🍪 แม่พิมพ์ (class) 🍪🍪🍪 ขนมจริง (objects) ┌─────────────────────────┐ │ พิมพ์เขียว "Question" │ q1 = Question(text: '1+1=?', answer: '2') │ กำหนดว่าทุกชิ้นต้องมี: │ กดพิมพ์ → q2 = Question(text: 'เมืองหลวงไทย?', answer: 'กรุงเทพ') │ · text (โจทย์) │ q3 = Question(text: 'ภาษาของ Flutter?', answer: 'dart') │ · answer (เฉลย) │ │ · check() (วิธีตรวจ) │ แต่ละชิ้นหน้าตาเหมือนกัน (โครงเดียวกัน) └─────────────────────────┘ แต่ "ไส้" ต่างกัน (ข้อมูลของใครของมัน)

แล้ว inheritance (การสืบทอด) ล่ะ? — ฉากที่สอง

🍪 แม่พิมพ์แม่: "ขนม" (มี: อบ(), ราคา) │ extends = "สืบทอดทุกอย่างจากแม่ แล้วเพิ่มของตัวเอง" ┌──────────────┼──────────────┐ ▼ ▼ ▼ 🍪 คุกกี้ช็อกชิป 🥐 ครัวซองต์ 🎂 เค้กวันเกิด (เพิ่ม: ช็อกชิป) (เพิ่ม: ทาเนย) (เพิ่ม: จุดเทียน() และ "อบ()" แบบของตัวเอง = @override) ลูกทุกตัว "อบได้ มีราคา" โดยไม่ต้องเขียนซ้ำ — เพราะสืบทอดจากแม่
🤔 ทำไม OOP ถึงเป็นหัวใจของโค้ดที่ขยายได้

พิมพ์เขียวเขียนครั้งเดียว ใช้ผลิตได้ไม่จำกัด และต่อยอดจากของเดิมได้โดยไม่ต้องรื้อ — แอปใหญ่ๆ ที่มีโค้ดเป็นแสนบรรทัดอยู่รอดได้เพราะหลักการนี้

2. 💻 ลงมือจริง: สร้างพิมพ์เขียว Question

/// พิมพ์เขียวของ "คำถาม 1 ข้อ" — ทุกข้อจะมีโครงเหมือนกันเป๊ะ
class Question {
  // property (สมบัติประจำตัว): ข้อมูลที่ขนมแต่ละชิ้นถือไว้
  final String text;
  final String answer;

  // constructor: "เครื่องกดพิมพ์" — วิธีสร้างขนมชิ้นใหม่จากพิมพ์นี้
  // this.text = รับค่ามาใส่ property ชื่อ text ของชิ้นนี้
  const Question({required this.text, required this.answer});

  // method (ท่าประจำตัว): ฟังก์ชันที่ผูกติดกับขนมชิ้นนั้น
  bool check(String userAnswer) {
    return userAnswer.trim().toLowerCase() == answer.toLowerCase();
  }
}

void main() {
  // กดพิมพ์ออกมาเป็น object จริง 2 ชิ้น
  final q1 = Question(text: 'Flutter ใช้ภาษาอะไรเขียน?', answer: 'dart');
  final q2 = Question(text: 'ไฟล์รายการ package ชื่ออะไร?', answer: 'pubspec.yaml');

  print(q1.text);              // เข้าถึง property ด้วยจุด
  print(q1.check('DART'));     // true — เรียก method ด้วยจุด (normalize ในตัวแล้ว!)
  print(q2.check('pubspec'));  // false

  // พลังจริง: List ของ Question — Collection บทที่แล้ว + Class บทนี้
  final quiz = [q1, q2];
  for (final q in quiz) {
    print('${q.text} (เฉลย ${q.answer})');
  }
}

เทียบกับ Map จะเห็นความต่างชัดๆ: พิมพ์ q1.txet (สะกดผิด) → เส้นแดงขึ้นทันทีตั้งแต่ยังไม่รัน เพราะพิมพ์เขียวประกาศไว้ชัดว่ามีอะไรบ้าง ส่วน map['txet'] → เงียบๆ แล้วได้ null ไปพังที่อื่น

Inheritance — ต่อยอดจากพิมพ์แม่

/// คำถามพิเศษ: ให้คะแนน 2 เท่า — สืบทอดทุกอย่างจาก Question แล้วเพิ่ม/แก้บางจุด
class BonusQuestion extends Question {
  const BonusQuestion({required super.text, required super.answer});
  // super.text = ส่งต่อค่าไปให้ constructor ของแม่

  int get points => 2;   // ของใหม่ที่แม่ไม่มี
}

🤯 ไขปริศนาที่ค้างมา 7 บท

class MyApp extends StatelessWidget {   // ← บรรทัดนี้จากบทที่ 1!
  const MyApp({super.key});             // ← constructor + ส่งค่าให้แม่

  @override                             // ← "ฉันจะทำท่านี้แบบของฉันเอง"
  Widget build(BuildContext context) {  // ← method ที่แม่สั่งให้ลูกทุกตัวต้องมี
    return MaterialApp(...);
  }
}
🧠 อ่านออกแล้วใช่ไหมครับ

MyApp คือพิมพ์เขียวที่สืบทอดจาก StatelessWidget (พิมพ์แม่ที่ทีม Flutter เขียนไว้ให้) — แม่กำหนดว่า "ลูกทุกตัวต้องมีท่า build() บอกว่าหน้าตาตัวเองเป็นยังไง" แล้วเราก็ @override ใส่หน้าตาของเราลงไป ทั้งเฟรมเวิร์ก Flutter คือ OOP ที่คุณเพิ่งเรียนจบนี่เอง

3. 🔧 เทคนิคการประยุกต์ใช้

เทคนิคที่ 1 — เจอ "ข้อมูลที่เดินทางด้วยกันเสมอ" = สร้าง class

ชื่อ+ราคา+รูป ของสินค้า, โจทย์+เฉลย ของคำถาม, ชื่อ+อีเมล ของผู้ใช้ — ข้อมูลที่ไปไหนไปกันเป็นชุด ควรอยู่ในพิมพ์เขียวเดียวกัน ไม่ใช่ตัวแปรกระจัดกระจายหรือ Map หลวมๆ

เทคนิคที่ 2 — method คือ "ความรู้ที่อยู่กับข้อมูล"

วิธีตรวจคำตอบ (check) อยู่ใน Question ไม่ใช่ลอยอยู่ใน main — ใครหยิบ Question ไปใช้ที่ไหน ก็ได้วิธีตรวจที่ถูกต้อง (รวม normalize) ไปด้วยเสมอ ไม่ต้องเขียนใหม่และไม่มีวันลืม trim

เทคนิคที่ 3 — ใช้ const constructor เมื่อข้อมูลไม่เปลี่ยน

const Question({...}) + property เป็น final ทั้งหมด = object ที่สร้างแล้วแก้ไม่ได้ (immutable) — Flutter รักของแบบนี้มาก (จำ "ใช้ const แล้วแอปเร็วขึ้น" จากบทที่ 1 ได้ไหม นี่คือที่มา)

เทคนิคที่ 4 — สืบทอดเมื่อ "เป็นชนิดหนึ่งของ" เท่านั้น

BonusQuestion เป็น Question ชนิดหนึ่ง → extends ได้ · แต่ Question ไม่ได้ เป็น Quiz ชนิดหนึ่ง (มันเป็น "ส่วนหนึ่งของ") → ใช้การเก็บใน List แทน ถามตัวเองด้วยประโยค "A เป็น B ไหม?" ก่อน extends ทุกครั้ง

4. 🏭 ตัวอย่างการเขียนจริงในงานจริง

🏭 โฟลเดอร์ models/ — สิ่งแรกที่เปิดดูเมื่อเข้าทีมใหม่

แอปจริงทุกตัวมีโฟลเดอร์ lib/models/ เก็บพิมพ์เขียวของข้อมูลทั้งแอป — วันแรกที่เข้าทีม รุ่นพี่จะบอกว่า "อยากรู้ว่าแอปนี้ทำอะไร ไปอ่าน models ก่อน"

// lib/models/product.dart — หน้าตา model จริงในแอปอีคอมเมิร์ซ
class Product {
  final String id;
  final String name;
  final double price;
  final int stock;

  const Product({
    required this.id,
    required this.name,
    required this.price,
    required this.stock,
  });

  // ความรู้เรื่อง "ของหมดไหม" อยู่กับตัวสินค้าเอง — ทั้งแอปถามคำถามนี้ที่เดียว
  bool get isOutOfStock => stock <= 0;
}

พอถึงเฟส 4 ตอนดึงข้อมูลจากอินเทอร์เน็ต ข้อมูลดิบ (JSON) จะถูกแปลงเข้า model แบบนี้ทันทีที่รับมา — เพราะทีมจริงเรียนรู้มานานแล้วว่า ปล่อย Map หลวมๆ วิ่งทั่วแอป = บั๊กสะกดผิดตามมาเป็นฝูง

5. 🧹 ตัวอย่าง Clean Code: จาก Map หลวมๆ สู่ Class ที่ปลอดภัย

// ❌ Map ทั่วแอป — สะกดผิดคือบั๊กเงียบ
final q = {'text': 'Flutter ใช้ภาษาอะไร?', 'answer': 'dart'};
print(q['txet']);        // สะกดผิด → null เงียบๆ ไปพังตอนรัน ไม่มีใครเตือน
// ตรวจคำตอบ? ต้องเขียน trim().toLowerCase() เองทุกจุด — ลืมสักจุด = บั๊ก
// ✅ Class — คอมไพเลอร์เป็นบอดี้การ์ด
final q = Question(text: 'Flutter ใช้ภาษาอะไร?', answer: 'dart');
// print(q.txet);        // ← เส้นแดงทันที ยังไม่ทันรันก็รู้ว่าผิด
print(q.check(' DART ')); // วิธีตรวจ (รวม normalize) มากับตัว object เสมอ
ประเด็น❌ Map✅ Class
สะกด key/ชื่อผิดnull เงียบๆ ตอนรันเส้นแดงตั้งแต่ตอนเขียน
ชนิดข้อมูลdynamic เดาเอาเองประกาศชัดทุก property
ความรู้ (วิธีตรวจ ฯลฯ)กระจายทั่วโค้ดอยู่กับข้อมูลใน method
คนใหม่อ่านโค้ดต้องไล่หาว่ามี key อะไรเปิด class เดียวเห็นครบ

6. ⚠️ ข้อผิดพลาดที่เจอบ่อย

① สับสนระหว่างพิมพ์เขียวกับของจริง

สาเหตุ: property เป็นของขนมแต่ละชิ้น จะใช้ได้ต้องกดพิมพ์ (สร้าง object) ก่อน
วิธีแก้: จำว่า class คือแบบแปลน ยังกินไม่ได้ — ต้องสร้าง object (final q = Question(...)) ก่อนเสมอจึงจะ .text ได้

② "Missing concrete implementation of 'X'"

สาเหตุ: extends พิมพ์แม่ที่สั่งว่า "ลูกทุกตัวต้องมี method นี้" แต่เราไม่ได้เขียน (เจอบ่อยสุดตอนลืมเขียน build() ใน Widget)
วิธีแก้: อ่านชื่อ method ใน error แล้วเขียนเติมพร้อม @override · ใน VS Code: คลิกหลอดไฟ 💡 → "Create missing override" ได้เลย

③ ลืม required แล้ว constructor บ่น "must have a default"

สาเหตุ: named parameter ใน constructor ที่ไม่ใส่ required และไม่มีค่า default — Dart ไม่รู้ว่าถ้าคนสร้างไม่ส่งค่ามา จะเอาอะไรใส่
วิธีแก้: property ที่ต้องมีเสมอ → required this.x · property ที่มีค่าตั้งต้นได้ → ใส่ default เช่น this.points = 1

7. 🏋️ แบบฝึกหัดท้ายบท (ลงมือทำจริง)

🥉 ระดับ 1 — พิมพ์เขียวแรก

สร้าง class Book (title, author, price) พร้อม constructor แบบ named + required, method describe() คืนข้อความแนะนำหนังสือ — สร้าง object 3 เล่ม เก็บใน List แล้ววนเรียก describe()

🥈 ระดับ 2 — สืบทอด

สร้าง class Animal มี name และ method speak() แล้วสร้าง Dog และ Cat ที่ extends Animal พร้อม @override speak() เป็นเสียงของตัวเอง — เก็บทั้งหมดใน List<Animal> วนเรียก speak() แล้วสังเกตว่าแต่ละตัวส่งเสียงต่างกันทั้งที่อยู่ List เดียวกัน (นี่คือ polymorphism — ยังไม่ต้องจำชื่อ แค่เห็นความเจ๋งพอ)

🥇 ระดับ 3 — Quiz เวอร์ชัน OOP (ผลงานจบเฟส!)

refactor quiz.dart ครั้งใหญ่: สร้าง class Question (พร้อม check()) และ class Quiz ที่ถือ List<Question> + score + method play() — main() เหลือแค่สร้าง Quiz แล้วเรียก play() (2–3 บรรทัด!) เพิ่ม BonusQuestion ที่ให้ 2 คะแนนอย่างน้อย 1 ข้อ — commit ด้วย refactor: convert quiz to OOP และ push ขึ้น GitHub ให้เรียบร้อย

8. ✅ เช็กลิสต์และสรุปบทเรียน

  • อธิบายความต่าง class (พิมพ์เขียว) กับ object (ของจริง) ด้วยคำพูดตัวเองได้
  • เขียน class พร้อม property, constructor (named + required), method ได้เอง
  • อธิบายได้ว่า extends และ @override ทำอะไร
  • อ่าน class MyApp extends StatelessWidget แล้วอธิบายได้ทุกคำ
  • Quiz เป็น OOP เต็มตัว และ push ขึ้น GitHub แล้ว

📖 สรุปศัพท์ใหม่บทนี้

ศัพท์ความหมายสั้นๆภาพจำ
classพิมพ์เขียว/แม่พิมพ์แม่พิมพ์ขนม
objectของจริงที่สร้างจากพิมพ์ขนมแต่ละชิ้น
propertyข้อมูลประจำตัวของ objectไส้ของขนมชิ้นนั้น
methodฟังก์ชันประจำตัวของ objectท่าประจำตัว
constructorวิธีสร้าง object ชิ้นใหม่เครื่องกดพิมพ์
extendsสืบทอดจากพิมพ์แม่สูตรต่อยอดจากสูตรแม่
@overrideทำ method ของแม่ในแบบของตัวเอง"ท่านี้ฉันขอทำเอง"
🎓 สรุปบทนี้ใน 3 บรรทัด

class = แม่พิมพ์ object = ขนม · ข้อมูลที่เดินทางด้วยกัน + ความรู้เกี่ยวกับมัน ควรอยู่ในพิมพ์เดียวกัน · extends = ต่อยอดโดยไม่รื้อของเดิม · Flutter ทั้งเฟรมเวิร์กคือ OOP ที่คุณเพิ่งอ่านออกแล้ว

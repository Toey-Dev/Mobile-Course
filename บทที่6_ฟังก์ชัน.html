1. 🖼️ ภาพเล่าเรื่อง: เครื่องปั่นน้ำผลไม้

ฟังก์ชันคือ "เครื่อง" ที่สร้างครั้งเดียว แล้วใช้ได้ตลอด: ใส่วัตถุดิบเข้าไป → เครื่องทำงาน → ได้ผลลัพธ์ออกมา

วัตถุดิบเข้า (parameter) เครื่องทำงาน ผลลัพธ์ออก (return) ┌─────────────────┐ 🍓 สตรอว์เบอร์รี ──────► │ 🌀 เครื่องปั่น │ ──────► 🥤 น้ำปั่น 🥭 มะม่วง ──────► │ makeSmoothie() │ ──────► 🥤 น้ำปั่น (รสใหม่) └─────────────────┘ สร้างเครื่องครั้งเดียว → ปั่นได้ล้านแก้ว ไม่ต้องสร้างเครื่องใหม่ทุกแก้ว

ฉากที่สอง — ทำไมต้องมีเครื่อง? ลองนึกภาพครัวที่ไม่มี

❌ ครัวที่ไม่มีเครื่อง (ไม่มีฟังก์ชัน) ✅ ครัวที่มีเครื่อง (มีฟังก์ชัน) ออเดอร์ 1: หยิบผลไม้→ล้าง→หั่น→ปั่น→เท ออเดอร์ 1: makeSmoothie('สตรอว์เบอร์รี') ออเดอร์ 2: หยิบผลไม้→ล้าง→หั่น→ปั่น→เท ออเดอร์ 2: makeSmoothie('มะม่วง') ออเดอร์ 3: หยิบผลไม้→ล้าง→หั่น→ปั่น→เท ออเดอร์ 3: makeSmoothie('กล้วย') (ก็อปโค้ดเดิม 5 บรรทัดทุกออเดอร์ (สูตรอยู่ที่เดียว — แก้สูตรครั้งเดียว แก้สูตร = ไล่แก้ทุกจุด พลาดแน่นอน) ทุกออเดอร์ได้สูตรใหม่ทันที)
🤔 นี่คือหลัก DRY (Don't Repeat Yourself) ตัวจริง

โค้ดซ้ำไม่ใช่แค่ "ไม่สวย" — มันคือระเบิดเวลา เพราะวันที่ต้องแก้ คุณจะแก้ไม่ครบทุกจุด

2. 💻 ลงมือจริง: กายวิภาคของฟังก์ชัน

// ┌ ชนิดผลลัพธ์  ┌ ชื่อเครื่อง       ┌ วัตถุดิบเข้า (parameter)
// ▼             ▼                 ▼
   String        greetPlayer(String name) {
     return 'สวัสดีคุณ $name ยินดีต้อนรับสู่เกม Quiz!';
//   ▲ return = ส่งผลลัพธ์ออกจากเครื่อง (จบการทำงานทันที)
   }

void main() {
  // เรียกใช้เครื่อง — ค่าที่ส่งเข้าไปจริงๆ เรียกว่า argument
  print(greetPlayer('มายด์'));   // สวัสดีคุณมายด์ ยินดีต้อนรับ...
  print(greetPlayer('บีม'));     // เครื่องเดิม วัตถุดิบใหม่
}

// ฟังก์ชันที่ "ทำอย่างเดียว ไม่คืนค่า" ใช้ void (แปลว่า ว่างเปล่า)
void showWelcomeBanner() {
  print('====================');
  print('   QUIZ GAME 2026   ');
  print('====================');
}

// ฟังก์ชันสั้นบรรทัดเดียว เขียนย่อด้วย arrow (=>) ได้
int doubleScore(int score) => score * 2;

Named Parameter — ท่าเดียวกับ Flutter เป๊ะ

พารามิเตอร์หลายตัวจำลำดับยาก Dart จึงมี named parameter: ระบุชื่อตอนเรียก อ่านออกทันทีว่าค่าไหนคืออะไร

// {required ...} = ต้องส่งเสมอ · มีค่า default ได้ (เช่น timeLimit = 30)
void askQuestion({
  required String question,
  required String answer,
  int timeLimit = 30,
}) {
  print('($timeLimit วินาที) $question');
}

void main() {
  // เรียกแบบระบุชื่อ — สลับลำดับได้ อ่านรู้เรื่องเสมอ
  askQuestion(question: 'Flutter ใช้ภาษาอะไร?', answer: 'dart');
  askQuestion(answer: 'pubspec', question: 'ไฟล์รายการ package ชื่ออะไร?', timeLimit: 60);
}

🤯 ความลับที่รอมาถึงบทนี้

ย้อนดูโค้ดบทที่ 1 สิครับ —

Text('สวัสดีชาวโลก', style: TextStyle(fontSize: 24))
//   ▲ argument ตามลำดับ  ▲ named argument ชื่อ style!
🧠 Widget ทุกตัวคือการเรียกฟังก์ชัน

Widget ทุกตัวที่คุณเขียนมาตลอด คือการเรียกฟังก์ชัน (constructor) ที่มี named parameter — title:, body:, child: ล้วนเป็น named parameter ทั้งนั้น คุณใช้คอนเซปต์นี้มาตั้งแต่วันแรกโดยไม่รู้ตัว ตอนนี้คุณอ่านโค้ด Flutter "ออก" แล้วจริงๆ

3. 🔧 เทคนิคการประยุกต์ใช้

เทคนิคที่ 1 — ตั้งชื่อฟังก์ชันด้วยกริยา

calculateTotal(), showResult(), isAnswerCorrect() — ฟังก์ชันคือ "การกระทำ" ชื่อจึงควรขึ้นต้นด้วยกริยา (ยกเว้นที่คืนค่า bool นิยมขึ้นด้วย is/has/can)

เทคนิคที่ 2 — กฎ "หนึ่งฟังก์ชัน หนึ่งงาน" ทดสอบด้วยคำว่า "และ"

ถ้าอธิบายฟังก์ชันแล้วต้องพูดว่า "มันถามคำถาม และ ตรวจคำตอบ และ สรุปคะแนน" = ควรแยกเป็น 3 ฟังก์ชัน

เทคนิคที่ 3 — ใช้ named parameter เมื่อพารามิเตอร์เกิน 2 ตัว

createUser('มายด์', 25, true, false) — true false คืออะไร? ไม่มีใครรู้ · createUser(name: 'มายด์', age: 25, isVip: true, isBanned: false) — อ่านออกทุกคน นี่คือเหตุผลที่ Flutter เลือกทางนี้ทั้งเฟรมเวิร์ก

เทคนิคที่ 4 — return เร็ว = guard clause เวอร์ชันฟังก์ชัน

ความรู้บทที่แล้วใช้ต่อได้เลย: เช็คเงื่อนไขไม่ผ่านแล้ว return ทันทีตั้งแต่บรรทัดแรกๆ ให้โค้ดหลักอยู่ตื้นเสมอ

4. 🏭 ตัวอย่างการเขียนจริงในงานจริง

🏭 สถานการณ์จริง: "สูตรคำนวณราคาที่ใช้ 5 ที่ในแอป"

แอปขายของมีจุดแสดงราคา 5 จุด: หน้าสินค้า, ตะกร้า, หน้าจ่ายเงิน, อีเมลยืนยัน, ใบเสร็จ — ทุกจุดต้องคิด "ราคาหลังส่วนลด + VAT" เหมือนกันเป๊ะ

// ฟังก์ชันเดียว เป็นแหล่งความจริงเดียว (single source of truth) ของสูตรราคา
double finalPrice({required double base, double discountRate = 0}) {
  final discounted = base * (1 - discountRate);
  return discounted * 1.07; // VAT 7%
}

วันหนึ่ง VAT เปลี่ยน หรือบริษัทเปลี่ยนนโยบายส่วนลด — แก้ที่เดียว ถูกทั้ง 5 จุดทันที ถ้าไม่มีฟังก์ชันนี้ เคสจริงที่เจอบ่อยคือแก้ครบแค่ 4 จุด → ใบเสร็จแสดงราคาไม่ตรงกับหน้าจ่ายเงิน → ลูกค้าร้องเรียน นี่คือบั๊กประเภทที่ทำให้บริษัทเสียเงินจริง และป้องกันได้ด้วยฟังก์ชันเดียว

5. 🧹 ตัวอย่าง Clean Code: ผ่าตัด main() ยักษ์

❌ main() ยักษ์

แบนเนอร์ + ถาม 3 ข้อ (ก็อปโค้ดเดิม 3 รอบ) + สรุปผล กองรวมกัน 40+ บรรทัด

✅ แยกตามหน้าที่

main() กลายเป็น "สารบัญ" ของโปรแกรม — อ่าน 7 บรรทัดรู้ flow ทั้งเกม

import 'dart:io';

void main() {
  showWelcomeBanner();                       // อ่าน main() แล้วรู้ flow ทั้งเกม
  int score = 0;                             // โดยไม่ต้องอ่านรายละเอียดสักบรรทัด
  score += playRound(question: 'Flutter ใช้ภาษาอะไรเขียน?', answer: 'dart');
  score += playRound(question: 'ไฟล์รายการ package ชื่ออะไร?', answer: 'pubspec.yaml');
  score += playRound(question: 'คำสั่งดูสถานะ git คืออะไร?', answer: 'git status');
  showFinalResult(score: score, total: 3);
}

void showWelcomeBanner() {
  print('=== QUIZ GAME 2026 ===');
}

/// เล่น 1 ข้อ: ถาม → รับคำตอบ → ตรวจ → คืนคะแนน (1 = ถูก, 0 = ผิด)
int playRound({required String question, required String answer}) {
  print('\nคำถาม: $question');
  final userAnswer = (stdin.readLineSync() ?? '').trim().toLowerCase();
  if (userAnswer == answer) {
    print('ถูกต้อง! 🎉');
    return 1;
  }
  print('เฉลย: $answer');
  return 0;
}

void showFinalResult({required int score, required int total}) {
  print('\nคุณได้ $score จาก $total คะแนน');
  print(score == total ? 'เพอร์เฟกต์! 🏆' : 'ฝึกต่อไป สู้ๆ 💪');
}

สังเกตว่า เพิ่มคำถามใหม่ = เพิ่ม 1 บรรทัดใน main() — ไม่ต้องก็อป 8 บรรทัดอีกแล้ว และแต่ละฟังก์ชันทดสอบแยกกันได้ (ประโยชน์จะชัดมากตอนบทที่ 22)

6. ⚠️ ข้อผิดพลาดที่เจอบ่อย

① เรียกฟังก์ชันแล้วได้ null / ค่าไม่มา

สาเหตุ: ลืม return — ฟังก์ชันทำงานครบแต่ไม่ได้ "ส่งของออกมา" หรือเขียน return ไว้ใน if แต่ลืมเคส else
วิธีแก้: Dart จะเตือน "The body might complete normally..." = มีเส้นทางที่ไม่ return — ตรวจว่าทุกเส้นทางมี return ครบ

② "The named parameter 'x' is required, but there's no corresponding argument"

สาเหตุ: ฟังก์ชันประกาศ required แต่ตอนเรียกไม่ได้ส่งค่านั้นมา
วิธีแก้: อ่านชื่อ parameter ใน error แล้วเติมให้ครบ — error นี้จะเจอบ่อยมากตอนเขียน Flutter (ลืมส่ง child:, builder: ฯลฯ)

③ ใช้ตัวแปรนอกฟังก์ชันไม่ได้ — "Undefined name 'userAnswer'"

สาเหตุ: ตัวแปรที่ประกาศในฟังก์ชัน มีชีวิตอยู่แค่ในฟังก์ชันนั้น (เรียกว่า scope) เหมือนอุปกรณ์ในครัว ร้านอื่นมาหยิบใช้ไม่ได้
วิธีแก้: อยากส่งค่าออก → return · อยากส่งค่าเข้า → parameter (อย่าแก้ด้วยการประกาศ global ไว้บนสุด — นั่นคือจุดเริ่มของโค้ดสปาเกตตี)

7. 🏋️ แบบฝึกหัดท้ายบท (ลงมือทำจริง)

🥉 ระดับ 1 — สร้างเครื่อง

เขียนฟังก์ชัน bmi({required double weightKg, required double heightM}) คืนค่า BMI (น้ำหนัก ÷ ส่วนสูง²) แล้วเขียน bmiCategory(double bmi) คืนข้อความ ผอม/ปกติ/เกิน ด้วย guard clause — เรียกใช้จาก main() กับข้อมูล 3 คน

🥈 ระดับ 2 — เครื่องคิดเลข

เขียน calculate({required double a, required double b, required String operator}) รองรับ + - * / โดยใช้ switch และมี guard clause กันหารด้วยศูนย์ (คืนข้อความเตือนแทน crash) — ทำเมนูวนถามด้วย while จากบทที่แล้ว

🥇 ระดับ 3 — ผ่าตัด Quiz

refactor quiz.dart ของคุณให้เหมือนหัวข้อ 5: main() ต้องไม่เกิน 10 บรรทัด, มี playRound() / showFinalResult() เป็นอย่างน้อย, เพิ่มระบบชีวิตจากบทที่ 5 กลับเข้าไปโดยให้ playRound คืนค่าผลลัพธ์ — commit ด้วย refactor: extract functions from main

8. ✅ เช็กลิสต์และสรุปบทเรียน

  • เขียนฟังก์ชันรับ parameter และ return ค่าได้เอง
  • อธิบายความต่าง parameter (ตอนประกาศ) กับ argument (ตอนเรียก) ได้
  • ใช้ named parameter + required เป็น และบอกได้ว่าทำไม Flutter ใช้ท่านี้ทั้งเฟรมเวิร์ก
  • อธิบาย scope ได้ว่าทำไมตัวแปรในฟังก์ชันหยิบใช้ข้างนอกไม่ได้
  • main() ของ Quiz สั้นกว่า 10 บรรทัดแล้ว

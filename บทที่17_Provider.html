1. 🖼️ ภาพเล่าเรื่อง: จากส่งเอกสารมือต่อมือ สู่เสียงตามสายโรงเรียน

ฉากที่ 1 — ปัญหาที่เราแบกมาตั้งแต่เฟส 3 (แค่ยังไม่เจ็บ เพราะแอปยังเล็ก)

วิธีปัจจุบัน: กระดานกลางอยู่ที่ HomeScreen — ใครอยากใช้ต้องส่ง "มือต่อมือ" HomeScreen (ถือ _tasks) │ ส่ง tasks + ส่ง callback ลงไป... ├──► TaskListSection │ │ ...ส่งต่ออีกชั้น... │ └──► TaskTile │ │ ...ส่งต่ออีกชั้น! │ └──► Checkbox └──► Navigator.push + await + pop(ค่า) เพื่อคุยกับหน้ารายละเอียด แอป 2 หน้า: พอไหว 😅 แอป 10 หน้าที่ 8 หน้าต้องใช้ _tasks: ทุกชั้นที่คั่นกลาง ต้องรับของที่ "ตัวเองไม่ได้ใช้ แค่ส่งต่อ" — ฝันร้ายของการแก้โค้ด (ปัญหานี้มีชื่อเล่นในวงการว่า prop drilling — เจาะอุโมงค์ส่งของ)

ฉากที่ 2 — วิธีใหม่: ย้ายกระดานไปห้องประชาสัมพันธ์ แล้วติดเสียงตามสายทั้งอาคาร

📢 ห้องประชาสัมพันธ์ (TaskProvider) · ถือกระดานกลาง _tasks ตัวจริง · มีปุ่มประกาศ notifyListeners() │ ┌─────────────────┼─────────────────┐ ▼ ▼ ▼ 🔊 HomeScreen 🔊 หน้ารายละเอียด 🔊 header card (ติดลำโพง watch) (ติดลำโพง watch) (ติดลำโพง watch) · ห้องไหนอยากรู้ข้อมูล → ฟังลำโพง (watch) — ประกาศเมื่อไหร่ วาดใหม่ทันที · ห้องไหนอยากสั่งแก้ข้อมูล → โทรศัพท์ภายใน (read) ไปสั่งห้องประชาสัมพันธ์ · ไม่มีการส่งมือต่อมืออีกต่อไป — ทุกห้องต่อตรงถึงศูนย์กลางเอง
💡 ข่าวดีที่สุดของบทนี้

คุณเข้าใจหัวใจของ Provider แล้วตั้งแต่บทที่ 12 — มันคือ "กระดานกลาง + single source of truth" ตัวเดิมเป๊ะ แค่ย้ายที่อยู่จาก Widget ไปเป็นคลาสอิสระ แล้วเปลี่ยนวิธีกระจายข่าวจาก callback มือต่อมือ เป็นเสียงตามสาย — แนวคิดไม่ใหม่เลย ใหม่แค่ "ท่อส่ง"

2. 💻 ลงมือจริง: ย้าย To-Do เข้าระบบเสียงตามสาย (4 ขั้น)

ขั้นที่ 1 — ติดตั้ง package (บทที่ 2 สอนไว้แล้ว)

flutter pub add provider

ขั้นที่ 2 — สร้างห้องประชาสัมพันธ์: lib/providers/task_provider.dart

import 'package:flutter/foundation.dart';
import '../models/task.dart';

/// ห้องประชาสัมพันธ์ของข้อมูลงานทั้งแอป
/// extends ChangeNotifier = "คลาสนี้มีระบบเสียงตามสายในตัว" (inheritance บทที่ 8!)
class TaskProvider extends ChangeNotifier {
  // กระดานกลางตัวจริง — private (_) คนนอกแก้ตรงๆ ไม่ได้ (บทที่ 10)
  final List<Task> _tasks = [];

  // ประตูอ่านอย่างเดียว: ห้องอื่นดูได้ แต่แก้ต้องผ่าน method ข้างล่างเท่านั้น
  List<Task> get tasks => List.unmodifiable(_tasks);

  // ค่าที่คำนวณได้ ไม่เก็บซ้ำ — กฎเดิมจากบทที่ 10 ยังศักดิ์สิทธิ์
  int get doneCount => _tasks.where((t) => t.isDone).length;

  void addTask(String title) {
    _tasks.add(Task(title: title));
    notifyListeners();          // 📢 กดปุ่มประกาศ: "ข้อมูลเปลี่ยนแล้ว ทุกลำโพงวาดใหม่!"
  }

  void toggleTask(Task task) {
    task.toggle();
    notifyListeners();          // ญาติแท้ๆ ของ setState — แค่ประกาศทั้งอาคารแทนห้องเดียว
  }

  void removeTask(Task task) {
    _tasks.remove(task);
    notifyListeners();
  }
}

สังเกตให้ดี: ไฟล์นี้ไม่มี Widget ไม่มี BuildContext เลย — เป็น Dart ล้วนแบบเฟส 2 นี่คือความงามข้อแรก: ตรรกะธุรกิจแยกขาดจากหน้าจอ (จะสำคัญมากตอนเขียนเทสต์บทที่ 22)

ขั้นที่ 3 — ติดตั้งระบบทั้งอาคาร: ครอบแอปใน main.dart

import 'package:provider/provider.dart';
import 'providers/task_provider.dart';

void main() {
  runApp(
    // ครอบ "นอกสุด" — ทุก Widget ใต้ร่มนี้เข้าถึง TaskProvider ได้หมด
    ChangeNotifierProvider(
      create: (context) => TaskProvider(),
      child: const TodoApp(),
    ),
  );
}

ขั้นที่ 4 — เปลี่ยนหน้าจอจาก "ถือกระดาน" เป็น "ฟังลำโพง"

/// 🤯 HomeScreen กลายเป็น StatelessWidget ได้แล้ว!
/// เพราะมันไม่ถือ state เองอีกต่อไป — แค่ฟังลำโพงแล้ววาดตาม
class HomeScreen extends StatelessWidget {
  const HomeScreen({super.key});

  @override
  Widget build(BuildContext context) {
    // 🔊 watch = ติดลำโพง: อ่านค่า + สมัครรับประกาศ (ประกาศเมื่อไหร่ build ใหม่)
    final provider = context.watch<TaskProvider>();

    return Scaffold(
      appBar: AppBar(title: Text('งาน ${provider.doneCount}/${provider.tasks.length}')),
      body: ListView.builder(
        itemCount: provider.tasks.length,
        itemBuilder: (context, index) {
          final task = provider.tasks[index];
          return TaskTile(
            task: task,
            // ☎️ read = โทรศัพท์ภายใน: สั่งงานเฉยๆ ไม่สมัครรับประกาศ
            onToggle: () => context.read<TaskProvider>().toggleTask(task),
          );
        },
      ),
    );
  }
}

กฎจำง่ายที่ครอบคลุม 95% ของการใช้งาน

ใน build (อยากให้จอตามข้อมูล) → context.watch<T>() 🔊 ลำโพง ใน onPressed/callback (สั่งงาน) → context.read<T>() ☎️ โทรศัพท์

รันแล้วลองครบ: เพิ่ม/ติ๊ก/ลบ — ทุกอย่างทำงานเหมือนเดิม แต่เบื้องหลังเปลี่ยนไปสิ้นเชิง: ไม่มี setState เหลืออยู่เลย และตัวเลขบน AppBar อัปเดตเองทุกครั้งที่ประกาศ

3. 🧠 คอนเซปต์: แล้ว pop(ค่า) จากบทที่ 15 ล่ะ?

จำท่า "pop พร้อมพัสดุ" ที่หน้ารายละเอียดต้องส่ง true กลับมาบอกหน้าหลักให้ลบไหมครับ — ดูสิ่งที่เกิดขึ้นเมื่อทั้งสองหน้าต่อสายตรงถึงห้องประชาสัมพันธ์:

แบบเดิม (บท 15): แบบใหม่ (Provider): หน้ารายละเอียด ── pop(true) ──► หน้ารายละเอียด ── read().removeTask() ──► 📢 หน้าหลัก await → เช็ค → setState │ หน้าหลัก (watch) ◄── ประกาศถึงเอง ────────┘ หน้าหลักต้อง "รอฟังคำตอบ" ทุกครั้ง ลบเสร็จ กด back กลับมา — รายการหายไปแล้ว โดยหน้าหลักไม่ต้องรู้เลยว่าเกิดอะไรที่ไหน

pop(ค่า) ยังมีที่ใช้ (เช่น ฟอร์มที่ผลลัพธ์เป็นเรื่องเฉพาะของหน้าที่เรียก) แต่ ข้อมูลกลางของแอปไม่ต้องเดินทางแบบนั้นอีกแล้ว — นี่แหละคือคำว่า "จัดการข้อมูลทั้งแอปอย่างเป็นระบบ" ใน Roadmap

4. 🔧 เทคนิคการประยุกต์ใช้

เทคนิคที่ 1 — เปิดประตูอ่านอย่างเดียวเสมอ (getter + unmodifiable)

_tasks เป็น private แล้ว expose ผ่าน List.unmodifiable(_tasks) — ใครแอบ provider.tasks.add(...) จะ crash ทันทีแทนที่จะแก้กระดานเงียบๆ โดยไม่ประกาศ (บั๊กเงียบที่ตามยากที่สุด) — กติกาเดียวกับ "ลูกห้ามถือสำเนา" บทที่ 12 แต่บังคับด้วยโค้ดจริง

เทคนิคที่ 2 — แยก provider ตามโดเมน ไม่ใช่ยำรวมก้อนเดียว

TaskProvider ดูแลงาน, SettingsProvider ดูแลธีม/ภาษา — provider ละเรื่อง เหมือนโรงเรียนมีทั้งเสียงตามสายฝ่ายวิชาการและฝ่ายกิจกรรม (ใช้ MultiProvider ครอบหลายตัว) — อย่าสร้าง AppProvider ก้อนยักษ์ที่รู้ทุกเรื่อง มันคือ main() ยักษ์ (บทที่ 6) กลับชาติมาเกิด

เทคนิคที่ 3 — Consumer<T> เมื่ออยากประหยัดการวาด

watch ใน build ทำให้ ทั้งหน้า วาดใหม่ทุกประกาศ — ถ้าหน้าใหญ่แต่มีจุดเดียวที่เปลี่ยน (เช่น badge ตัวเลข) ครอบเฉพาะจุดนั้นด้วย Consumer<TaskProvider>(builder: ...) = วาดใหม่แค่ตรงนั้น (ตอนนี้แอปเราเล็ก watch ทั้งหน้าไปก่อนได้ รู้ไว้เพื่อวันที่แอปโต)

เทคนิคที่ 4 — เมื่อพร้อมแล้ว: Riverpod คือรุ่นถัดไปของแนวคิดนี้

ในตลาดงานคุณจะได้ยินชื่อ Riverpod บ่อย — มันคือวิวัฒนาการของ Provider จากผู้สร้างคนเดียวกัน (ปลอดภัยกว่าเรื่อง compile-time, ไม่ผูกกับ BuildContext) แนวคิด "ศูนย์กลาง + ประกาศ + ฟัง" เหมือนกันทุกประการ — เรียน Provider ให้แน่นก่อน แล้ว Riverpod จะใช้เวลาเรียนแค่ครึ่งวัน อย่าเพิ่งกระโดดไปตอนนี้

5. 🏭 ตัวอย่างการเขียนจริงในงานจริง

🏭 สถานการณ์จริง: "ตะกร้าสินค้าที่โผล่ 5 จุดในแอป"

แอปช้อปปิ้งทุกเจ้ามีข้อมูลตะกร้าปรากฏพร้อมกันหลายจุด — นี่คืองานที่ Provider เกิดมาเพื่อทำ:

📢 CartProvider (ถือรายการ + ยอดรวม) │ notifyListeners ครั้งเดียว... ┌──────────┬──────────┬─────┴────┬─────────────┐ ▼ ▼ ▼ ▼ ▼ 🔊 badge 🔊 ปุ่ม 🔊 หน้า 🔊 แถบยอดรวม 🔊 หน้า เลขบนไอคอน "ใส่ตะกร้า" ตะกร้า ล่างจอ checkout ตะกร้า (เปลี่ยนเป็น "อยู่ในตะกร้า ✓") ผู้ใช้กด "ใส่ตะกร้า" ที่หน้าสินค้า → ทั้ง 5 จุดอัปเดตพร้อมกันเป๊ะ ในบรรทัดเดียว: context.read<CartProvider>().add(product);

ลองนึกภาพทำสิ่งนี้ด้วย callback มือต่อมือ: ปุ่มในหน้าสินค้า (ลึก 4 ชั้น) ต้องคุยกับ badge บน AppBar ของอีกหน้า — แทบเป็นไปไม่ได้ นี่คือเหตุผลที่ ทุกประกาศรับสมัครงาน Flutter เขียนคำว่า state management และคุณเพิ่งข้ามกำแพงนั้นแล้ว

6. 🧹 ตัวอย่าง Clean Code: Widget มีหน้าที่เดียว — แสดงกับสั่ง

// ❌ ตรรกะธุรกิจฝังในหน้าจอ — แก้กฎทีต้องคุ้ยหา Widget
onPressed: () async {
  final title = await Navigator.push<String>(...);
  if (title != null) {
    if (_tasks.any((t) => t.title == title)) {        // กฎธุรกิจข้อ 1
      /* SnackBar เตือนซ้ำ */
    } else if (_tasks.length >= 50) {                 // กฎธุรกิจข้อ 2
      /* SnackBar เตือนเต็ม */
    } else {
      setState(() => _tasks.add(Task(title: title))); // ปนกับเรื่องจอ
    }
  }
}
// หน้าอื่นอยากเพิ่มงานบ้าง? ก็อปกฎพวกนี้ไปอีกชุด → ความจริงสองชุดภาคตรรกะ!
// ✅ กฎธุรกิจอยู่ที่ provider — Widget เหลือแค่ "แสดงผล + ส่งคำสั่ง"

// ── ใน TaskProvider: กฎอยู่บ้านเดียว ทุกหน้าได้กฎเดียวกันอัตโนมัติ ──
/// เพิ่มงานใหม่ — คืนข้อความ error ถ้าเพิ่มไม่ได้, null ถ้าสำเร็จ (ท่า validator บทที่ 16!)
String? addTask(String title) {
  if (_tasks.any((t) => t.title == title)) return 'มีงานชื่อนี้อยู่แล้ว';
  if (_tasks.length >= 50) return 'งานเต็ม 50 รายการแล้ว';
  _tasks.add(Task(title: title));
  notifyListeners();
  return null;
}

// ── ในหน้าจอ: สั้น อ่านออก และหน้าไหนๆ ก็เขียนเหมือนกัน ──
final error = context.read<TaskProvider>().addTask(title);
if (error != null && context.mounted) {
  ScaffoldMessenger.of(context).showSnackBar(SnackBar(content: Text(error)));
}

ถามตัวเองว่า "กฎนี้เป็นเรื่องของ ข้อมูล หรือเรื่องของ จอ?" — ห้ามซ้ำ/จำกัดจำนวน = เรื่องข้อมูล → อยู่ใน provider · สีแดง/ขีดฆ่า/SnackBar = เรื่องจอ → อยู่ใน Widget เส้นแบ่งนี้คือรากของ "สถาปัตยกรรม" ที่จะเรียนต่อในบทที่ 21

7. ⚠️ ข้อผิดพลาดที่เจอบ่อย

① "Could not find the correct Provider<TaskProvider> above this Widget"

สาเหตุ: Widget ที่เรียก watch/read อยู่ "นอกร่ม" ของ ChangeNotifierProvider — มักเกิดจากครอบไว้ต่ำเกินไป (ครอบแค่หน้าแรก แล้วหน้าที่ push มาทีหลังหาไม่เจอ) หรือลืมครอบเลย
วิธีแก้: ครอบที่ "นอกสุด" ใน main() ตามขั้นที่ 3 — ทุกหน้ารวมถึงหน้าที่ push มาจะอยู่ใต้ร่มเดียวกันเสมอ (error message ของ provider เขียนดีมาก มีคำอธิบาย+วิธีแก้ยาวเหยียด — ฝึกอ่าน error บทที่ 1 ใช้ได้เต็มที่)

② เรียก method แล้วจอไม่เปลี่ยน

สาเหตุ: ลืม notifyListeners() ท้าย method — ญาติแท้ๆ ของ "ลืม setState" บทที่ 10: กระดานถูกแก้แล้วจริง แต่ไม่มีใครกดปุ่มประกาศ ลำโพงทุกตัวเลยเงียบ
วิธีแก้: ทุก method ใน provider ที่แก้ข้อมูล ต้องจบด้วย notifyListeners() เสมอ · เช็คลิสต์เร็ว: จอนิ่งผิดปกติ → ① method มี notifyListeners ไหม ② หน้าจอใช้ watch (ไม่ใช่ read) ใน build ไหม

③ ใช้ watch/read สลับที่กัน

อาการ 1: ใช้ read ใน build → จอแสดงค่าเก่าตลอด ไม่เคยอัปเดต (โทรศัพท์ถามครั้งเดียวแล้ววางสาย — ไม่ได้สมัครรับประกาศ)
อาการ 2: ใช้ watch ใน onPressed → Flutter โยน error ทันที ("Tried to listen to a value... outside of the widget tree")
วิธีแก้: ท่องสูตรจากหัวข้อ 2: build → watch 🔊 · callback → read ☎️ — สลับเมื่อไหร่ อาการโผล่ทันทีทั้งสองทาง

8. 🏋️ แบบฝึกหัดท้ายบท (ลงมือทำจริง)

🥉 ระดับ 1 — เปิดห้องประชาสัมพันธ์

ทำขั้นที่ 1–4 ให้ครบ: ติดตั้ง package, สร้าง TaskProvider, ครอบ main, HomeScreen อ่านผ่าน watch — ทดสอบว่าเพิ่ม/ติ๊ก/ลบยังทำงานครบ แล้วลองตั้งใจลบ notifyListeners() ออกหนึ่งจุด ดูอาการ "กระดานเปลี่ยนแต่ลำโพงเงียบ" ด้วยตาตัวเอง (แล้วใส่กลับ)

🥈 ระดับ 2 — รื้อ setState ทิ้งทั้งแอป

ย้ายทุก action เข้า provider จน HomeScreen เป็น StatelessWidget เต็มตัว (ห้ามเหลือ setState เกี่ยวกับ tasks แม้แต่ตัวเดียว — ค้นทั้งโปรเจกต์ด้วย Ctrl+Shift+F) ฟอร์มเพิ่มงานจากบทที่ 16 ต่อเข้า addTask ของ provider — commit ด้วย refactor: migrate task state to Provider

🥇 ระดับ 3 — พลังเสียงตามสายเต็มระบบ

(1) หน้ารายละเอียด (บทที่ 15) เปลี่ยนมาใช้ provider: ปุ่มติ๊ก/ลบเรียก read ตรงๆ — สังเกตว่ากลับมาหน้าหลักแล้วทุกอย่างตรงกันเอง โดยไม่ต้องใช้ pop(ค่า) อีกแล้ว (2) header card แสดง "เสร็จ x/y" ผ่าน Consumer เฉพาะจุด (3) ย้ายกฎธุรกิจ "ห้ามชื่อซ้ำ" เข้า addTask ตามหัวข้อ 6 พร้อมแสดง error ที่ฟอร์ม — commit ด้วย feat: provider everywhere with business rules

9. ✅ เช็กลิสต์และสรุปบทเรียน

  • อธิบายได้ว่า Provider แก้ปัญหาอะไร (prop drilling) ด้วยภาพเสียงตามสาย
  • เขียน ChangeNotifier + notifyListeners ได้เอง และรู้ว่ามันคือญาติของ setState
  • ท่องสูตรได้: build → watch · callback → read และอธิบายอาการเมื่อสลับกัน
  • อธิบายได้ว่าทำไม getter ควรคืน List.unmodifiable
  • แยกได้ว่ากฎไหนควรอยู่ใน provider กฎไหนควรอยู่ใน Widget
  • todo_app ไม่มี setState เกี่ยวกับ tasks เหลืออยู่ และหน้ารายละเอียดเลิกใช้ pop(ค่า) แล้ว

📖 สรุปศัพท์ใหม่บทนี้

ศัพท์ความหมายสั้นๆภาพจำ
Providerระบบแจกจ่าย state ให้ทั้งแอประบบเสียงตามสายทั้งอาคาร
ChangeNotifierคลาสที่มีความสามารถ "ประกาศ" ในตัวห้องประชาสัมพันธ์
notifyListenersประกาศว่าข้อมูลเปลี่ยน ทุกผู้ฟังวาดใหม่กดปุ่มประกาศ 📢
ChangeNotifierProviderติดตั้งระบบไว้เหนือสุดของแอปเดินสายลำโพงทั้งอาคาร
context.watchอ่านค่า + สมัครรับประกาศ (ใช้ใน build)ลำโพงประจำห้อง 🔊
context.readสั่งงานครั้งเดียว ไม่สมัครรับ (ใช้ใน callback)โทรศัพท์ภายใน ☎️
Consumerฟังประกาศเฉพาะจุด ประหยัดการวาดลำโพงเฉพาะมุมห้อง
🎓 สรุปบทนี้ใน 3 บรรทัด

Provider = กระดานกลาง (บท 12) ที่ย้ายไปอยู่ห้องประชาสัมพันธ์ + เสียงตามสาย · notifyListeners คือ setState ระดับทั้งแอป · build → watch, callback → read · กฎของข้อมูลอยู่ใน provider — Widget แค่แสดงกับสั่ง

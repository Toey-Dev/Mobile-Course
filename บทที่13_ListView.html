ListView: แสดงข้อมูลเป็นรายการแบบมืออาชีพ

รองรับงานหมื่นชิ้นได้ลื่นๆ ด้วย ListView.builder พร้อมท่าเด็ดของแอปมือถือ: ปัดเพื่อลบ

ตอนนี้เราวาดรายการด้วย Column ซึ่งมีจุดอ่อนร้ายแรง: งาน 100 ชิ้นก็วาด 100 ชิ้นทั้งที่จอแสดงได้ ~10 — บทนี้เจอ Widget ที่ฟีด Facebook, แชท Line และแอปดังทุกตัวใช้

1. 🖼️ ภาพเล่าเรื่อง: ร้านซูชิสายพาน

ทำไมต้องมี ListView ในเมื่อ Column ก็แสดงรายการได้? — มาดูครัวของร้านอาหารสองแบบ:

❌ ครัวแบบ Column ✅ ครัวแบบ ListView.builder (ทำทุกจานทิ้งไว้ล่วงหน้า) (ร้านซูชิสายพาน — ทำเฉพาะที่กำลังจะถูกเห็น) เมนู 10,000 จาน เมนู 10,000 จาน → ทำครบ 10,000 จานตั้งแต่เปิดร้าน → บนสายพานมีแค่ ~15 จาน → โต๊ะลูกค้าวางได้แค่ 10 จาน (เท่าที่ลูกค้าเห็นตรงหน้า) → อีก 9,990 จานเน่าอยู่หลังครัว → จานไหนเลื่อนพ้นสายตา เชฟเก็บกลับ 😱 ครัวระเบิด (แอปค้าง/กินแรมมหาศาล) แล้วปั้นจานใหม่แทนตำแหน่งที่กำลังจะโผล่ 😎 ลูกค้ากี่หมื่นเมนูก็ลื่น จอมือถือ = โต๊ะลูกค้า: แสดงของได้ทีละ ~10-15 ชิ้นเท่านั้น จะทำของที่มองไม่เห็นไว้ทำไม? ← นี่คือหัวใจของ builder
🤔 ทำไมเรื่องนี้สำคัญระดับ "ขาดไม่ได้"

ฟีด Facebook มีโพสต์เป็นล้าน แชท Line มีข้อความเป็นหมื่น — ทุกแอปดังใช้เทคนิค "ทำเฉพาะที่มองเห็น" นี้ (ศัพท์ทางการ: virtualization) ถ้าไม่มีมัน มือถือไม่มีทางแสดงรายการยาวๆ ได้เลย และ ListView.builder ให้เทคนิคนี้มาฟรีๆ

2. 💻 ลงมือจริง: อัปเกรด To-Do เป็น ListView

ขั้นที่ 1 — เปลี่ยน Column → ListView.builder

// เดิม (บทที่ 12): Column + .map() — วาดทุกชิ้นเสมอ
// ใหม่: ListView.builder — วาดเฉพาะที่เห็น + เลื่อนได้ในตัว

body: ListView.builder(
  itemCount: _tasks.length,              // บอกจำนวนจานทั้งหมดในเมนู
  itemBuilder: (context, index) {        // สูตรปั้น "จานที่ index" เมื่อถึงคิวโผล่
    final task = _tasks[index];          // ความรู้ index จากบทที่ 7 ตรงๆ
    return TaskTile(
      task: task,
      onToggle: () => _toggleTask(task),
    );
  },
),

อ่านว่า: "รายการนี้มีของ itemCount ชิ้น — พอชิ้นที่ index กำลังจะปรากฏบนจอ ให้เรียกสูตร itemBuilder ปั้นมันขึ้นมา" — Flutter จัดการที่เหลือให้หมด: เลื่อน, รีไซเคิล, ประสิทธิภาพ

ขั้นที่ 2 — รู้จัก ListTile: แถวรายการสำเร็จรูป

CheckboxListTile ที่ใช้บทที่แล้วคือญาติของ ListTile — แม่แบบแถวรายการมาตรฐานที่มี "ช่องเสียบ" ครบ:

ListTile(
  leading: const Icon(Icons.task_alt),   // ช่องซ้าย: ไอคอน/รูป
  title: Text(task.title),               // ข้อความหลัก
  subtitle: const Text('แตะเพื่อดูรายละเอียด'), // ข้อความรอง (บทที่ 15 จะกดได้จริง)
  trailing: IconButton(                  // ช่องขวา: ปุ่ม/เวลา/ลูกศร
    icon: const Icon(Icons.delete_outline),
    onPressed: onDelete,
  ),
  onTap: () {},                          // แตะทั้งแถวได้ด้วย
)

ตำแหน่ง leading-title-subtitle-trailing ตรงกับแถวใน Line, Spotify, Gmail — เพราะทั้งวงการใช้แม่แบบเดียวกันนี้

ขั้นที่ 3 — ปัดเพื่อลบ ด้วย Dismissible + แจ้งผลด้วย SnackBar

itemBuilder: (context, index) {
  final task = _tasks[index];
  return Dismissible(
    // Key = "ป้ายทะเบียน" ประจำแถว — Flutter ใช้แยกแยะว่าแถวไหนถูกปัดหาย
    // ต้องไม่ซ้ำกัน: ใช้เนื้อหางาน ไม่ใช่ index (index ขยับได้เวลาลบ!)
    key: ValueKey(task.title),
    direction: DismissDirection.endToStart,      // ปัดจากขวาไปซ้ายเท่านั้น
    background: Container(                       // พื้นหลังสีแดงตอนกำลังปัด
      color: Colors.red,
      alignment: Alignment.centerRight,
      padding: const EdgeInsets.only(right: 20),
      child: const Icon(Icons.delete, color: Colors.white),
    ),
    onDismissed: (_) {
      setState(() => _tasks.removeAt(index));    // ลบข้อมูลจริงจากกระดานกลาง!
      // SnackBar = แถบแจ้งผลชั่วคราวด้านล่างจอ — UX ที่ดีต้องยืนยันผลการกระทำ
      ScaffoldMessenger.of(context).showSnackBar(
        SnackBar(content: Text('ลบ "${task.title}" แล้ว')),
      );
    },
    child: TaskTile(task: task, onToggle: () => _toggleTask(task)),
  );
},

รันแล้วปัดแถวไปทางซ้าย: พื้นแดงโผล่ → แถวหายพร้อมแอนิเมชัน → แถบยืนยันเด้งขึ้น — ท่านี้แหละที่ทำให้แอปรู้สึก "เป็นแอปจริง"

3. 🔧 เทคนิคการประยุกต์ใช้

เทคนิคที่ 1 — เช็ค Empty State ก่อนเข้า ListView เสมอ

เทคนิค isEmpty จากบทที่ 7 ถึงเวลาเปล่งประกาย: body: _tasks.isEmpty ? const _EmptyState() : ListView.builder(...) — หน้าจอ "ยังไม่มีงาน เพิ่มเลย! 🎉" ดูเป็นมิตรกว่าจอขาวโล่งมาก แอปดีๆ ทุกตัวทำ

เทคนิคที่ 2 — คั่นแถวสวยๆ ด้วย ListView.separated

พี่น้องของ builder ที่เพิ่มพารามิเตอร์ separatorBuilder — ใส่ const Divider() แล้วได้เส้นคั่นทุกแถวอัตโนมัติ ไม่ต้องคำนวณเองว่าแถวสุดท้ายไม่ควรมีเส้น

เทคนิคที่ 3 — Key ใช้ค่าที่ "ติดตัวข้อมูล" ไม่ใช่ตำแหน่ง

ValueKey(task.title) ✓ เพราะติดตัวงานชิ้นนั้น · ValueKey(index) ✗ เพราะพอลบแถวแรก ทุกแถวขยับ index — Flutter จะสับสนว่าใครเป็นใคร (ในแอปจริง model มักมีฟิลด์ id ไว้เป็น Key โดยเฉพาะ)

เทคนิคที่ 4 — ListView ใน Column ต้องครอบ Expanded

บทที่ 11 สอนว่า Expanded = "ขอพื้นที่ที่เหลือ" — ListView ต้องรู้ขอบเขตความสูงของตัวเอง ถ้าวางใน Column เปล่าๆ มันไม่รู้ว่าสูงได้แค่ไหน (จะพังแบบ error ① ข้างล่าง) ครอบ Expanded(child: ListView...) = จบ

4. 🏭 ตัวอย่างการเขียนจริงในงานจริง

🏭 สถานการณ์จริง: "หน้าจอแชทที่มีข้อความสามหมื่นข้อความ"

ทีมแอปแชทแห่งหนึ่งเจอปัญหา: ห้องแชทกลุ่มใหญ่เปิดช้ามาก กินแรมจนแอปเด้ง — สาเหตุ: dev มือใหม่ในทีมใช้ Column + SingleChildScrollView แสดงข้อความทั้งหมด

ก่อนแก้: วาดข้อความทั้ง 30,000 bubble ตอนเปิดห้อง → เปิดห้อง 8 วินาที, แรม 1.2GB หลังแก้: ListView.builder → เปิดห้อง 0.3 วินาที, แรม 90MB (วาดเฉพาะ ~20 bubble ที่อยู่บนจอ)

การแก้ครั้งนั้นเปลี่ยนโค้ดไม่ถึง 20 บรรทัด แต่เปลี่ยนรีวิวแอปจาก 3.1 → 4.4 ดาว — ความรู้เรื่อง builder ไม่ใช่ optional สำหรับแอปจริง และ interviewer ชอบถามเรื่องนี้ตอนสัมภาษณ์งาน Flutter มาก: "Column กับ ListView.builder ต่างกันยังไง เมื่อไหร่ใช้อะไร" — ตอนนี้คุณตอบได้แล้วพร้อมภาพร้านซูชิ

5. 🧹 ตัวอย่าง Clean Code: itemBuilder ผอมๆ

// ❌ ยัดทุกอย่างใน itemBuilder — 50 บรรทัดต่อแถว อ่านไม่ไหว
itemBuilder: (context, index) {
  final task = _tasks[index];
  return Dismissible(
    key: ValueKey(task.title),
    background: Container( /* 8 บรรทัด */ ),
    onDismissed: (_) { /* 10 บรรทัด */ },
    child: ListTile(
      leading: /* 5 บรรทัด */,
      title: /* 6 บรรทัดพร้อมเงื่อนไขขีดฆ่า */,
      trailing: /* 8 บรรทัด */,
    ),
  );  // ← itemBuilder เดียว 50 บรรทัด — จะหาอะไรเจอ
},
// ✅ itemBuilder ทำหน้าที่เดียว: หยิบข้อมูล ส่งให้ Widget ที่มีชื่อ
itemBuilder: (context, index) {
  final task = _tasks[index];
  return DismissibleTaskTile(          // ชื่อเล่าเรื่องครบ: ปัดได้ + แถวงาน
    task: task,
    onToggle: () => _toggleTask(task),
    onDismissed: () => _removeTask(task),
  );
},
// รายละเอียดทั้งหมดอยู่ใน lib/widgets/dismissible_task_tile.dart
// — หนึ่งไฟล์ หนึ่งหน้าที่ (บทที่ 2) ในร่างของ Widget
🧠 หลักการเดียว โผล่ทุกระดับ

สังเกตว่ากฎเดิมโผล่ซ้ำทุกระดับ: ฟังก์ชันผอม (บทที่ 6) → build ผอม (บทที่ 11) → itemBuilder ผอม (บทนี้) — หลักการเดียว ใช้ได้ทุกที่ นี่แหละที่บอกว่า "เข้าใจ > จำ"

6. ⚠️ ข้อผิดพลาดที่เจอบ่อย

① "Vertical viewport was given unbounded height"

สาเหตุ: วาง ListView ใน Column โดยไม่บอกขอบเขตความสูง — ListView ถามว่า "ฉันสูงได้แค่ไหน?" Column ตอบว่า "เท่าไหร่ก็ได้" → ตกลงกันไม่ได้ = พัง (error พี่น้องกับ overflow บทที่ 11)
วิธีแก้: ครอบ ListView ด้วย Expanded เมื่ออยู่ใน Column: Column(children: [ HeaderCard(), Expanded(child: ListView.builder(...)) ])

② "A dismissed Dismissible widget is still part of the tree"

สาเหตุ: ปัดแถวหายไปจากจอ แต่ "ข้อมูลจริง" ยังอยู่ใน List — Flutter วาดรอบใหม่เจอของที่เพิ่งปัดทิ้งโผล่มาอีก เลยประท้วง (หรือ Key ซ้ำกันจนแยกแถวไม่ออก)
วิธีแก้: ใน onDismissed ต้อง "ลบข้อมูลจริง" ใน setState เสมอ (จอคือเงาของข้อมูล!) และเช็คว่า Key ไม่ซ้ำ — สองแถวชื่อเดียวกันควรใช้ id ไม่ใช่ title

③ "RangeError (index): Invalid value: ... : 5"

สาเหตุ: itemCount ไม่ตรงกับ List จริง — เช่น itemCount: _tasks.length + 1 (เผื่อแถวพิเศษ) แต่ itemBuilder ลืมจัดการ index ตัวเกิน
วิธีแก้: itemCount ต้องเท่ากับจำนวนแถวที่ itemBuilder รองรับจริงเป๊ะ — เพื่อนเก่าจากบทที่ 7 กลับมาทัก: ช่องสุดท้ายคือ length - 1 เสมอ

7. 🏋️ แบบฝึกหัดท้ายบท (ลงมือทำจริง)

🥉 ระดับ 1 — เข้าสายพาน

เปลี่ยนรายการงานจาก Column เป็น ListView.builder ตามขั้นที่ 1 แล้วพิสูจน์พลังของมัน: ใน initState เติมงานทดสอบ 1,000 ชิ้นด้วย for (บทที่ 5) — เลื่อนดูว่าลื่นไหม แล้วลองคิดว่าถ้าเป็น Column จะเกิดอะไรขึ้น (ลบงานทดสอบออกก่อน commit)

🥈 ระดับ 2 — ปัดเพื่อลบ

ใส่ Dismissible + พื้นหลังแดง + SnackBar ยืนยันตามขั้นที่ 3 และเพิ่ม Empty State ("ยังไม่มีงาน 🎉") เมื่อรายการว่าง — commit ด้วย feat: swipe to delete with snackbar

🥇 ระดับ 3 — แยกโซนเสร็จ/ค้าง

อัปเกรดหน้าหลัก: งานค้างอยู่บนสุด งานเสร็จแล้วอยู่ล่างใต้หัวข้อ "เสร็จแล้ว" — เทคนิค: สร้าง List ใหม่จาก .where() สองรอบ (ค้าง+เสร็จ) แล้วประกอบเป็นรายการเดียวส่งให้ builder, หัวข้อคั่นทำเป็นแถวพิเศษ (ระวัง itemCount!) — ติ๊กงานแล้วมันย้ายโซนเองทันที commit ด้วย feat: separate pending and completed sections

8. ✅ เช็กลิสต์และสรุปบทเรียน

  • อธิบายได้ว่าทำไม Column ไม่เหมาะกับรายการยาว (ภาพร้านซูชิสายพาน)
  • เขียน ListView.builder พร้อม itemCount/itemBuilder ได้เอง
  • ใช้ ListTile ครบช่อง leading/title/subtitle/trailing
  • อธิบายได้ว่าทำไม Key ต้องใช้ค่าติดตัวข้อมูล ไม่ใช่ index
  • รู้ว่า ListView ใน Column ต้องครอบ Expanded และอธิบายได้ว่าทำไม
  • To-Do ปัดลบได้ มี SnackBar และ Empty State แล้ว
